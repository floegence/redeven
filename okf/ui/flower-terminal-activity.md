# Summary

Terminal activity is a renderer-first view over canonical Floret Activity items, not a second process lifecycle. Stable activity identity preserves disclosure and viewport state across live replacements; terminal output may poll Redeven process handles, but canonical status and completion remain projection-owned. Measured disclosure animation, local scrolling, and transcript following preserve user reading intent and reduced-motion behavior.

# Contract

## Mechanism

Terminal process activity remains part of that canonical projection. A running `terminal.exec` row receives its command, status, process id, cumulative retained snapshot, and sequence facts from the Floret `ActivityTimeline` payload produced by Redeven tool presentation metadata. `terminal.exec` publishes the process id as soon as the PTY starts through Floret's running activity update path, so `yield_ms` affects only when the agent receives the tool result, not when the user can inspect command progress. Each model-issued `terminal.read` activity uses its required user-language description as the compact row title, while the returned `payload.command` remains the real command shown only in expanded terminal details. Result presentation omits a label so Floret preserves the call intent, and Flower does not synthesize generic terminal-output titles. Each read remains a separate activity row and its expanded panel shows only that call's `output` delta: a first read can show phase 1 and the next read can show phase 2 without making the command appear to have executed again. `terminal.terminate` follows the same presentation contract: its required user-language description remains the row title through the successful result, the tool name is not used as a visible fallback, and the expanded command detail shows the real command plus the public `process_id` needed for inspection and diagnostics. That public payload must not expose host-local path or input fields such as `cwd`, `workdir`, or `stdin`.

Expanded terminal rows are read-only product presentation. Completed status, output, exit code, and duration come directly from the canonical Floret Activity payload and never call a Redeven completion-output endpoint. The shared terminal store applies two explicit sequence operations while the Activity remains running: a newer exec snapshot replaces the cumulative view, while a contiguous PTY process-read delta appends to it. Equal or older snapshots are ignored by sequence, an empty delta does not clear existing exec output, and an explicit `truncated=true` gap replaces the unavailable prefix with the retained delta. The store never decides cumulative-versus-incremental behavior with string prefix or substring checks. Flower and Env App send only the required `after_seq` process-read query for a running Activity with a public process id; server-owned waits and page limits are not UI options. Polling may continue while details are collapsed so a stored delta is not lost merely because it was not visible when it arrived. A valid process read can return a terminal `success`, `error`, or `canceled` snapshot when the PTY finishes between polls, including an empty delta whose `last_seq` remains equal to `after_seq`; that response stops local polling but can never change canonical Activity status, create lifecycle facts, or trigger settlement. Raw AppServer errors are logged for diagnostics and render only localized live-output or terminal-action failure text. The surface exposes icon-only actions to reveal and copy the full raw command and caps output at five visual line boxes using explicit font size, line height, padding, and border-box sizing; shorter output uses its natural height, while later lines increase only the local scroll height. Every pending terminal process binds its complete Floret settlement target and creating top-level or subagent Host before launch. After the pending handle is exposed and PTY reap completes, one finalizer performs one settlement through that owner; process reads, writes, cleanup, and later UI access never trigger or retry settlement, and no maintenance Host fallback exists. A ready settlement projection advances the same `ThroughOrdinal` cursor as every other projection and publishes the canonical `timeline.replaced` snapshot. An unavailable settlement projection remains committed Floret state, produces diagnostic logging, and is not read back or synthesized locally. `terminal.terminate` waits for process-tree exit, output drain, and finalization before returning, while `Service.Close` completes finalizers before releasing Floret Hosts and stores. Consequently, canonical terminal ordering and completion facts come only from durable Floret projections rather than a terminal-specific database, retry path, floor, refresh, or UI repair mechanism.

# Boundaries

Flower activity details are renderer-first product views, not generic payload inspectors. `error`, `terminal`, `file`, `patch`, `web_search`, `question`, `completion`, `todos`, and `subagents` each have an explicit display model that prioritizes the information a normal user is trying to inspect: failure reason, terminal output, file content, diffs, search results, prompts and choices, outcomes, todo lists, or delegated task status. Failed tools render the public error message first and do not expose error codes, retryability flags, item ids, tool ids, or lifecycle diagnostics in the ordinary detail body. Successful tools do not show generic completion summaries or event-code details such as `tool execution completed`, `todos.updated`, `file.updated`, `success`, `completed`, or `ok`. A compact structured metadata block remains available only for allowlisted product details that do not belong to the primary renderer. Flower must not fall back to displaying every payload key, or raw `status`/`kind`/`tool`/`item` rows, as the main detail experience.

Flower activity disclosure is presentation state over the canonical activity lifecycle, not a second execution state machine. Every stateful activity view uses one semantic identity composed from thread, run, turn, and canonical `item_id`; status, payload, message block position, item position, and renderer object identity are not part of that identity. `timeline.replaced` may move an activity between message blocks, but it must reuse the same activity row, disclosure controller, detail component, and terminal viewport. A live projection may first expose the stable item without a renderer or detail payload and later enrich that same item; the row keeps its DOM identity, while the disclosure trigger remains bound and its native disabled state follows detail availability. Routine `pending` or `running` detail waits 400 ms before opening, so activity that settles sooner remains compact and never produces an expand-collapse flash. Opening uses a 360 ms measured-height animation. An automatically opened terminal command starts its 1200 ms completion hold only after the canonical `success` or `canceled` presentation and latest visible output have completed their terminal presentation frame; another settled output frame restarts the full hold. The detail then uses a 300 ms closing animation. Other routine activity remains intent-anchored. Error, waiting, approval, or blocking attention opens immediately and remains open until the user changes it. A manual expand or collapse overrides later lifecycle updates for that stable activity item, and pointer, focus, wheel, or touch interaction inside automatically opened detail converts it to an explicit open choice so the surface cannot close content while the user is reading, selecting, scrolling, or operating it. Current ordinary approval rows remain non-expandable when the composer owns the actionable approval.

Open activity detail owns its real pixel height. A shared disclosure motion controller mounts at `0px`, measures the nearest detail content from its bounding rectangle, and uses the Chromium `ResizeObserver` and Web Animations APIs as required platform capabilities. Opening, resizing, retargeting, reversal, and closing are all owned by one current animation; its `finished` promise is the only completion signal that may advance presence state or unmount content. CSS does not own a second height-transition clock, and animation completion is not inferred from elapsed timers. Consecutive content changes cancel the previous animation and redirect the current visual height to the latest measured target. Content growth and shrinkage use a 280 ms animation, including terminal polling, command expansion, responsive reflow, error replacement, and disappearing controls. The nearest detail disclosure owns this dynamic resize; the Env App timeline wrapper animates only its own open or close, preventing nested height animations from compounding movement. Detail content remains capped at `min(42rem, 72vh)`, and terminal output remains capped at five visual lines; growth beyond those limits becomes local scrolling instead of continuing to move the conversation.

Flower schedules its existing transcript-tail follower on every disclosure animation frame. It preserves the bottom anchor only while the user is already following latest content, and it never forces a conversation back to the end after the user scrolls upward. Env App continues to use `VirtualMessageList` row `ResizeObserver` handling: following mode applies each intermediate height delta, while paused mode restores the captured viewport anchor after every observed height. Terminal panels use a separate local 24 px near-bottom threshold. Binding or reopening a terminal viewport begins a new local viewing session at the latest line. Each output update receives one presentation frame: following mode scrolls to the bottom before reporting the output as presented, while paused mode preserves `scrollTop` and still reports presentation completion. Upward wheel or scroll pauses local following, returning to the bottom resumes it, and canonical activity replacement does not reset that choice. Output merging, polling cadence, canonical status ownership, and persistence are unchanged. Reduced-motion presentation suppresses routine automatic expansion, keeps attention and manual access immediate, commits measured pixel height without animation, and unmounts closed detail immediately. These presentation rules do not alter Floret activity status, event ordering, persistence, read state, audit state, or terminal settlement.
