Flower is a reusable chat surface over a Redeven-owned product timeline. Redeven owns thread persistence, turn anchoring, live drafts, and message ordering; Flower renders the ordered messages that Redeven projects for the selected thread.

# Mechanism

The Flower live bootstrap includes `timeline_messages` as the only ordered message source for a thread snapshot. Redeven builds product timeline messages from persisted Redeven transcript rows, conversation turn anchors, Floret turn projections, and live assistant drafts. For Floret-hosted assistant execution, the block order inside the assistant message is derived from Floret `ThreadTurnProjection`: assistant text entries and tool activity entries are interleaved by Floret ordinal, with tool display details coming from the Floret `ActivityTimeline` plus Redeven's `ToolPresentationSpec`. Turn anchors order each user message beside its assistant projection, while live drafts can replace the matching assistant message during an active run. Completed Floret-backed assistant projections are read through Floret public APIs such as `TurnResult.Projection` during the run and provider-free `ThreadMaintenanceHost.ReadTurnProjection` after reload; Redeven does not persist them as assistant transcript shadow rows.

Projection application is strict and monotonic. Every formal projection must carry the exact `ThreadID`, `TurnID`, and `RunID`, a supported turn status, a positive durable `ThroughOrdinal`, and only supported segment shapes. Activity timelines must pass Floret's public validator; control segments must carry durable control content, and typed signals must be one of Redeven's configured `ask_user` or `task_complete` contracts. Redeven keeps the last applied ordinal for each full execution identity, ignores equal or older projections, and advances the cursor only after the complete projection maps and replaces the assistant blocks. A valid zero-segment projection remains a canonical empty response and may clear placeholders. Runtime events are rejected before any stream, context, activity, or projection field is applied when their event type, stream type, lifecycle state, activity timeline, or projection contract is unknown. Invalid or unknown contracts record `floret.contract.rejected` and leave the current presentation unchanged.

Floret v0.5.0 reports projection availability separately from engine execution. Redeven accepts only a validated `ready` projection. An `unavailable` result never triggers an immediate `ReadTurnProjection` retry: an engine failure remains the primary failure, while a successful engine run records `floret_projection_unavailable`, preserves streamed presentation, and instructs the user to refresh rather than rerun. Explicit refresh and history reconstruction may call `ReadTurnProjection`, validate the full returned contract, and omit a missing or rejected projection instead of synthesizing assistant text from product run errors. Pending terminal settlement uses the same outcome and ordinal rules, so durable ordering protects terminal activity without a host-specific lifecycle floor or summary repair algorithm.

Thread snapshots, list items, and thread patches carry `model_id` and `reasoning_selection` as thread defaults. Flower renders a fused model/reasoning segmented control in the composer when model options are available; the reasoning segment is shown only when the selected model exposes controllable reasoning capability. A connected Desktop model source contributes a read-only model catalog alongside editable provider models, preserving each opaque model id, display label, token limits, modalities, and reasoning capability without presenting remote provider configuration as editable local settings. On a new thread, the selected draft model is written to `current_model_id` and used as the create-and-launch default, while draft reasoning selection is sent with the create request only when the selected model supports reasoning control. Switching to a model without reasoning removes any draft reasoning override before launch. On an existing thread, changing the model first sends a thread PATCH through `setThreadModel` and then writes the same model through `setCurrentModel` so the next newly created thread defaults to that choice; changing the composer reasoning default sends only the thread reasoning PATCH. Existing-thread launch does not add the global current model to the send request, so subsequent turns use the persisted thread default unless the request is intentionally local to a waiting-input response. The model/reasoning segmented control is disabled while the selected thread is active, waiting for user input, compacting, read-only, or otherwise unable to accept a thread preference change. The Flower UI quality gate explicitly runs a cross-layer Desktop model-source test that renders the shared surface over the real Env adapter and verifies catalog projection, labels, reasoning defaults, thread creation, and first-turn RPC payloads together.

Flower treats user-triggered operation failures and input blockers as transient feedback, not transcript content. The shared surface emits a host notification intent for send failures, invalid or blocked submits, model/default-model persistence failures, permission changes, approval submits, and sidebar actions; Env App and Desktop render that intent as a toast through their own notification systems. These toasts do not change the canonical thread timeline, and Flower no longer inserts composer or sidebar error cards into the page flow for those transient actions. Run failures, tool failures, subagent detail errors, and other structured timeline/detail content remain rendered by their owning timeline or detail renderer rather than being guessed from the notification channel.

Thread navigation separates visual, requested, and displayed ownership. A user click marks the list row immediately, then commits selection after the intent paint. A thread whose detail is already in the explicit warm set is displayed atomically without calling `loadThread` again. A cold target loads in staging while the previous transcript remains visible and read-only; only the newest request may publish the target transcript, focus the composer, restore tail position, or emit `content_presented`. Compose mode, permission changes, programmatic focus ownership, and newer thread requests cancel or supersede the pending user transaction. Codex uses the same product contract with separate selected, foreground, displayed, loading, and session-cache state, so warm sessions switch without provider or transcript remount and stale loads cannot replace the current thread.

Live streaming uses canonical server events to create only the rows those events define. `message.started` may create or merge one assistant streaming row and marks it as the single active cursor. User messages are not locally invented; a canonical `message.committed` payload upserts the committed user or product-owned message by id, so a newly accepted user turn can appear in the current window without a placeholder. Floret text deltas may render temporary live markdown, while Floret `runtime.Event.Projection` replaces the live assistant draft with the current ordered assistant text and activity blocks whenever Floret commits a thread-entry fact. Aggregate Floret `runtime.Event.ActivityTimeline` remains lifecycle observation data and does not create main-thread activity blocks by itself. A local tool row is pending when Floret has observed the model's tool call but has not dispatched the handler; it becomes running only from Floret's dispatch-start lifecycle fact after its own permission and approval gate passes. Batched ordinary calls may therefore start and finish in any actual order. A sibling that requires approval can remain waiting while already allowed siblings run, and approval display order never serializes handler execution. A committed assistant projection that contains an invalid activity segment is rejected as a unit and does not clear the existing live activity block; a valid projection with no content can still intentionally replace the assistant body with an empty markdown block. Stop and hard-cancel paths update run/thread state and may detach live presentation, but they do not persist a canceled assistant transcript row; any terminal assistant boundary shown after reload must come from Floret terminal projection or explicit Redeven-owned run lifecycle state, not from a shadow copy. Block, activity, or failure updates still require their target message to exist; activity timeline changes enter the UI only inside Floret-derived `ThreadTurnProjection` activity segments delivered through `message.block_set`, a live draft, or a projection-backed timeline snapshot. Ordinary `message.block_delta`, `message.block_set`, `message.committed`, `run.status_changed`, and `thread.patched` events do not automatically publish companion full timeline replacements. If local state is missing, the reducer marks the batch as needing resync. Explicit `timeline.replaced` snapshots are reserved for resync, snapshot recovery, and terminal settlement finalization after Redeven has applied a canonical Floret projection; the payload carries `stream_generation` and `snapshot_through_seq`, and the reducer replaces the full message array while advancing the cursor to that snapshot boundary. `stream_generation` is a Service-instance epoch rather than a protocol constant. When the selected thread observes a newer generation after holding an older cursor, Flower resets the live cursor for that thread before fetching low-sequence events from the new stream, so runtime restart or stream rebuild cannot cause a stale high cursor to hide new events.

Terminal process activity remains part of that canonical projection. A running `terminal.exec` row receives its command, status, process id, output tail, and sequence facts from the Floret `ActivityTimeline` payload produced by Redeven tool presentation metadata. `terminal.exec` publishes the process id as soon as the PTY starts through Floret's running activity update path, so `yield_ms` affects only when the agent receives the tool result, not when the user can inspect command progress. That public payload must not expose host-local path or input fields such as `cwd`, `workdir`, or `stdin`. Expanded terminal rows in the Flower timeline are read-only product presentation: the surface renders a compact shell-highlighted command preview, exposes icon-only actions to reveal and copy the full raw command, shows output in a compact terminal panel, and polls the Redeven terminal read API by sequence while the item is running and a process id is available. Flower and Env App cap that output viewport at five visual line boxes using explicit font size, line height, padding, and border-box sizing; shorter output uses its natural height, while later lines increase only the local scroll height. Output polling is a detail/read path, not a timeline lifecycle source: returned process status may stop local output polling, but it cannot change canonical activity status or create lifecycle facts. When a pending process exits, Redeven runs one idempotent Floret settlement mutation through the run-owned host or provider-free `ThreadMaintenanceHost`, then persists the terminal audit result. A ready settlement projection advances the same `ThroughOrdinal` cursor as every other projection and publishes the canonical `timeline.replaced` snapshot before acknowledging the process. An unavailable settlement projection is still a committed Floret mutation: Redeven records the diagnosis, does not immediately read or synthesize a replacement projection, and completes the product audit path without reporting a mutation failure. Concurrent reads wait for an in-flight settlement. True settlement mutation, audit persistence, canonical timeline build, or replacement publication failures leave settlement unacknowledged and retryable. Consequently, canonical terminal ordering comes from durable Floret ordinals rather than a terminal-specific floor, refresh, or UI repair path.

Subagent presentation uses two separate public contracts. Thread-level child membership comes from Floret `ListSubAgents` through the active parent host or provider-free `ThreadMaintenanceHost`; Redeven maps that product-neutral snapshot into `thread.subagents` during live bootstrap and publishes the same list through `thread.patched` after runtime child events. Thread list summaries do not own `thread.subagents`: if a refresh summary omits the field, Flower preserves the selected thread detail value, and only an explicit detail or live patch value replaces it. The parent header Sub Agents dropdown is derived only from `thread.subagents`, so a runtime restart can restore the count and child list without scanning transcript activity blocks. New child names are canonical 1-5 word English presentation labels normalized before Floret spawn, while the UI humanizes historical snake/kebab/camel names and applies an English role fallback when no English words are available, without migrating stored state. Redeven does not read Floret store tables, create child threadstore rows, or persist a duplicate mapped child list. Parent timeline rows still come from Floret `ThreadTurnProjection` activity segments for the `subagents` tool. Their public payload is sanitized to route and display facts only: `thread_id`, `subagent_id`, `parent_thread_id`, task name/title, `task_description`, agent type, status, timestamps, counts, timeout/truncation flags, and public error/summary fields. It filters raw child content, host-local paths, detail refs, context/debug fields, `last_message`, `waiting_prompt`, `can_*` control flags, queued input counts, and fields outside the current subagent payload allowlist. Inline `Open messages` actions read only the public `thread_id`/`subagent_id` facts in the activity payload; there is no activity sidecar fallback. The dropdown is a header-triggered floating surface, not a page or side panel: it uses the shared Floe surface layer and named Flower layer tokens, displays neutral compact chrome with localized active/ended metrics, and keeps active and ended sections visible. Each row exposes one status icon, the canonical display name, one-line responsibility text, localized status, elapsed duration when timestamps are available, and a detail chevron. Active rows sort waiting-input, running, queued, then unknown; ended rows sort failed, timed-out, completed, then canceled, with update time as the secondary order. Opening the dropdown focuses the first active row; arrow keys, Home, End, Enter, and Escape provide complete keyboard navigation, and reduced-motion mode removes popover and status animation. Selecting a child never selects or navigates to the child thread; it loads parent-scoped detail through the surface adapter and opens a read-only Floe `FloatingWindow` over the current parent conversation. The detail window reuses the same timeline/message/activity renderers as the main thread, adds a bottom live status lane, tails active child detail incrementally, auto-follows only while the user is at the bottom, and exposes a scroll-to-latest affordance when the user reviews earlier output. Detail tailing continues while the child is queued, running, or waiting for input even when the current page reports `has_more=false`; terminal child status stops the tail, and transient tail errors stay in the detail dock with a longer retry interval. The detail tail and full child execution timeline are human UI state only; they are never injected into the parent model context by default. The overlay renders Floret `ReadSubAgentDetail` through the parent thread scope: user/assistant messages, tool results, approvals, turn markers, compactions, errors, and generic custom rows come from the child detail events, while tools render from the top-level canonical child activity timeline. Bounded preview detail rows are inspection data and must not be presented as complete child user or assistant output; a human detail surface that needs complete visible child output must consume an explicit Floret display-detail contract rather than mapping preview text into the transcript body. Tailing updates replace that stable activity block instead of appending event-row activity snapshots, so older running tool updates disappear when a later result or settlement becomes canonical. Detached reads use provider-free `ThreadMaintenanceHost` and do not require provider configuration, a child threadstore row, or child Flower metadata in Redeven. Summary rows, when present, are explicit Floret detail data rendered in the same read-only message model. Subagent model results and activity payloads support canonical `items` only, and unsupported detail, transcript, raw-output, or non-allowlisted collection fields are filtered before presentation.

Active cursor ownership belongs to Redeven's live projection. The projection marks at most one live assistant draft with `active_cursor` while the selected thread is running, and canceled, completed, idle, background, or merely streaming-shaped messages are not treated as active unless the projection marks them active. The visible model status affordance is a localized bottom dock lane driven by `model_io.updated`, not by `active_cursor` or thread running status. `model_io` is derived from Floret provider lifecycle and stream observations: provider request maps to waiting for model response; assistant, reasoning, and model tool-call stream observations map to thinking; provider retry maps to retrying; provider finish maps to finalizing. The indicator is not a timeline message and does not depend on a visible assistant block. The indicator's readable text is the base rendered text; shimmer is a decorative overlay and must not be the only mechanism that makes the label visible.

Current approval decisions belong in the composer and follow the backend-owned `approval_queue`. The queue snapshot carries generation, revision, current action id, current position, total, and unresolved count; each action carries immutable queue order, queue generation, batch index, and batch size. When the selected thread has a pending `primary_action`, the composer body replaces the text or secret input with exactly that approval panel, and the footer primary controls become Reject and Approve. Other pending actions remain read-only locators in timeline and delegated surfaces. The panel shows product-safe presentation fields such as action label, purpose text, command preview, working directory, friendly effects, safe targets, risk flags, and stable `position / total` progress; it does not render raw tool arguments, argument hashes, internal run ids, or internal tool ids. A decision click synchronously enters a thread-local handoff before any request work: both buttons disable in the same JavaScript task, the selected button exposes loading and `aria-busy`, the original Approve/Reject label and button geometry stay stable, and the frozen approval card remains mounted. The server's existing approval response supplies `current_cursor`; Flower waits for live projection at that cursor to show a new actionable queue head or an explicit empty queue, and only after 1500 ms without reconciliation performs one canonical fallback reload. A live event that arrives before the HTTP receipt may settle the handoff immediately, while a later receipt for the old action is ignored. The next card replaces the old card directly, receives programmatic focus on its container, and triggers one `aria-live` progress announcement without auto-focusing Approve. The final card switches once to the ordinary composer only after authoritative empty approval state or a terminal thread; a short presentation-only handoff marker preserves the focused composer border and shadow across the DOM transition without delaying content. A stable approval conflict triggers a canonical reload without a generic send-failure toast; Flower retries the user's decision once only if the same action remains current and actionable, using the refreshed queue generation/revision and action fields. If the old action timed out, resolved, or lost the queue head, Flower does not retry and never applies that decision to the newly promoted action. A failed submission reload re-enables the buttons only when canonical state proves the same action is still current; if reload also fails, Flower keeps the card disabled and waits for live reconnect rather than risk a duplicate decision. Historical activity can still display waiting, queued, or processed lifecycle on the tool item for audit and navigation, but it exposes no inline decision controls and cannot create or fill current pending ordinary-tool approval state. `requires_approval` means the tool item passed through the approval lifecycle; the Flower surface treats an activity row as the current approval mirror only when it also has `approval_state=requested` and `status=waiting`. Approved, rejected, timed-out, canceled, running, pending, or successful tool rows keep their normal activity affordances, including expansion when detail exists. In live materialized state, a missing `approval_actions` or `approval_queue` field means the response did not sample that approval state and must not clear an existing composer approval; an explicit empty action object, null queue, or queue with zero unresolved items is the authoritative "no pending approvals" state. Thread list summaries do not own approval actions or queue detail, so an omitted field or a transient raw `running` summary cannot replace selected-thread live approval state. `GetThread` and `ListThreads` project the effective status as `waiting_approval` while the queue has unresolved actions, even when approved sibling handlers are still running. Refresh, reconnect, and resync preserve queue generation, order, current action, and the mounted composer card. A service restart cannot reconstruct in-memory waiters, so persisted delegated pending actions become unavailable audit records rather than actionable approvals. The surface derives selected-thread status in the order `waiting_user`, `waiting_approval`, active run `running`, then terminal/idle patch state, so a transient live bootstrap or list refresh cannot flicker the composer between an approval card and the textarea.

Context usage and compaction state share the canonical projection model while using distinct presentation affordances. Redeven stores the latest `context_usage` plus historical `context_compactions` and `timeline_decorations` in Flower live materialized state. `context.usage.updated` drives the compact circular context indicator in the composer footer beside the primary send action. When a new active run starts before its current `context.usage.updated` event arrives, the composer may keep showing the previous persisted usage as a "last known context" indicator; once the active run's usage arrives, the same indicator switches back to current usage. This is a Flower presentation fallback over Redeven's projected state, not a Floret context lifecycle rule or a local UI token estimate. `context.compaction.updated` upserts a `context_compaction` timeline decoration keyed by `operation_id`; the UI renders it as a non-interactive divider such as compression in progress, compressed, or failed. A decoration carries a typed timeline `anchor` targeting a message, block, or activity item plus a before/after edge. Missing or invalid anchors are contract errors or skipped render entries, not permission to guess a fallback position. The divider is not a transcript message and does not affect message ordering, copying, or model-visible history. Because context telemetry is visible thread activity, persisted context usage and compaction events advance the canonical Flower read activity revision together with `last_context_run_id`, and broadcasting those context stream events also publishes a thread summary patch so selected live polling receives the current read status.

Flower `/compact` is a composer command, not a chat message. The surface parses the exact slash command before the stop/send matrix, calls the shared `compactThreadContext` adapter action, scrolls to the tail, and never submits `/compact` through `launchTurn`. Running threads compact in place: Redeven accepts the command, binds the decoration anchor at the time the manual operation is queued, forwards the manual request into the active Floret hosted-turn lifecycle, and keeps the run running after start, complete, or failed compaction events. Idle threads use an operation-only background path: the actor captures the anchor, broadcasts the running divider immediately, releases the actor, and then calls Floret `Host.CompactThread`; after Floret completes, Redeven persists only the returned opaque provider continuation when the product thread boundary still matches. Idle compaction never creates user or assistant transcript rows and never stores Redeven-owned provider-visible context state. The selected thread continues live polling while the in-memory live state has a `compacting` operation, even if the thread status is otherwise idle, so the same open Flower window observes the divider transition to compacted or failed. Cold bootstrap restores only terminal persisted compaction events (`compacted`, `failed`, or `cancelled`); an unfinished start/running event without a live worker is not replayed as a permanent "compression in progress" divider after restart. User turns sent while an idle compact operation is running enter the queued follow-up lane instead of blocking the actor or writing a premature transcript row; after the operation clears, queued drain starts the turn through the same canonical turn-start path. The next provider-visible context remains Floret-owned through Floret thread state and opaque provider continuation.

Accepted user turns are durable only when their canonical start state is complete. Redeven starts an idle or dequeued user turn in one threadstore transaction that writes the user transcript message, run record, conversation turn anchor, thread run state, structured input records, secret answers, upload bindings, and optional source follow-up consumption. If that transaction fails, no user transcript row remains. After it commits, live projection can broadcast the user message and assistant draft from the same canonical product timeline boundary.

Selected-thread live polling treats `model_io.updated` as a presentation boundary. Even when one live-events response contains a short provider stream followed by finalizing or clearing events, Flower commits the model status lane and yields a render frame at each model I/O boundary. This keeps brief but real provider streaming phases visible to the user without inventing timeline rows, fallback timers, or thread-status heuristics.

Thread read state is a user-scoped appserver projection over a canonical Flower activity snapshot persisted on `ai_threads`. Threadstore owns `flower_activity_revision`, `flower_activity_signature`, and `flower_activity_waiting_prompt_id`, initializes them for new and migrated threads, and advances them in the same transaction as visible message changes, run-state changes, waiting prompts, persisted context usage or compaction events, projected thread or message changes, stale active-run resets, and fork initialization. Appserver builds `read_status.snapshot` from `ThreadView.FlowerActivity` rather than deriving a second clock from run status or timestamps; unread is only `current.activity_revision > record.last_seen_activity_revision`.

Mark-read requests echo a snapshot previously delivered by the server. Future snapshots are rejected, current-revision signature, waiting-prompt, or last-message mismatches remain protocol errors, and stale snapshots are accepted without pretending they covered newer activity; the response still returns the current authoritative `read_status`. The shared Flower adapter returns that `read_status` directly instead of reloading the bootstrap. A selected running thread that completes while the user keeps it selected is therefore marked read from the final snapshot by the Flower surface, while background completions remain unread until selected.

# Boundaries

Flower does not sort messages, infer canonical insertion points from timestamps, or pair recent user messages with assistant drafts. It may project an assistant streaming row only from a `message.started` event and may upsert a canonical user row only from a `message.committed` payload. Local pending user rows are presentation-only composer feedback while a launch or queued follow-up request is in flight; they are removed or reclassified when the canonical thread snapshot confirms a committed message or queued turn, and they are never copied into transcript storage or model-visible history. The selected thread may show a bottom model status lane while a matching active run has `model_io`, but every durable rendered chat message must come from the current thread timeline.

Flower does not rebuild Floret execution order from Redeven transcript rows, audit tables, provider deltas, or tool-call records. For Floret-hosted execution, `transcript_messages.message_json` is not an assistant projection cache or execution transcript authority; user/product transcript rows and `conversation_turns` only provide Redeven product timeline anchors and Floret `ThreadID`/`TurnID`/`RunID` correlation. Floret live `runtime.Event.Projection`, `TurnResult.Projection`, provider-free `ThreadMaintenanceHost.ReadTurnProjection`, and pending settlement projections are the assistant markdown and activity block source; default preview detail reads are inspection data and must not downgrade an already complete Flower assistant block. Flower does not scan cached block JSON or activity rows to replace pending, waiting, or running tool states after stop or approval; tool states must arrive through Floret `ThreadTurnProjection` activity segments or the terminal pending settlement projected by Floret. Redeven does not publish independent live activity blocks from aggregate `runtime.Event.ActivityTimeline`, synthesize final activity rows from audit records, hide queued siblings, or turn an invalid Floret projection into an empty assistant block.

Flower activity details are renderer-first product views, not generic payload inspectors. `error`, `terminal`, `file`, `patch`, `web_search`, `question`, `completion`, `todos`, and `subagents` each have an explicit display model that prioritizes the information a normal user is trying to inspect: failure reason, terminal output, file content, diffs, search results, prompts and choices, outcomes, todo lists, or delegated task status. Failed tools render the public error message first and do not expose error codes, retryability flags, item ids, tool ids, or lifecycle diagnostics in the ordinary detail body. Successful tools do not show generic completion summaries or event-code details such as `tool execution completed`, `todos.updated`, `file.updated`, `success`, `completed`, or `ok`. A compact structured metadata block remains available only for allowlisted product details that do not belong to the primary renderer. Flower must not fall back to displaying every payload key, or raw `status`/`kind`/`tool`/`item` rows, as the main detail experience.

Flower activity disclosure is presentation state over the canonical activity lifecycle, not a second execution state machine. Every stateful activity view uses one semantic identity composed from thread, run, turn, and canonical `item_id`; status, payload, message block position, item position, and renderer object identity are not part of that identity. `timeline.replaced` may move an activity between message blocks, but it must reuse the same activity row, disclosure controller, detail component, and terminal viewport. Routine `pending` or `running` detail waits 400 ms before opening, so activity that settles sooner remains compact and never produces an expand-collapse flash. Opening uses a 360 ms measured-height animation. An automatically opened terminal command starts its 1200 ms completion hold only after the canonical `success` or `canceled` presentation and latest visible output have completed their terminal presentation frame; another settled output frame restarts the full hold. The detail then uses a 300 ms closing animation. Other routine activity remains intent-anchored. Error, waiting, approval, or blocking attention opens immediately and remains open until the user changes it. A manual expand or collapse overrides later lifecycle updates for that stable activity item, and pointer, focus, wheel, or touch interaction inside automatically opened detail converts it to an explicit open choice so the surface cannot close content while the user is reading, selecting, scrolling, or operating it. Current ordinary approval rows remain non-expandable when the composer owns the actionable approval.

Open activity detail owns its real pixel height. A shared disclosure motion controller mounts at `0px`, measures the nearest detail content from its bounding rectangle, and uses the Chromium `ResizeObserver` and Web Animations APIs as required platform capabilities. Opening, resizing, retargeting, reversal, and closing are all owned by one current animation; its `finished` promise is the only completion signal that may advance presence state or unmount content. CSS does not own a second height-transition clock, and animation completion is not inferred from elapsed timers. Consecutive content changes cancel the previous animation and redirect the current visual height to the latest measured target. Content growth and shrinkage use a 280 ms animation, including terminal polling, command expansion, responsive reflow, error replacement, and disappearing controls. The nearest detail disclosure owns this dynamic resize; the Env App timeline wrapper animates only its own open or close, preventing nested height animations from compounding movement. Detail content remains capped at `min(42rem, 72vh)`, and terminal output remains capped at five visual lines; growth beyond those limits becomes local scrolling instead of continuing to move the conversation.

Flower schedules its existing transcript-tail follower on every disclosure animation frame. It preserves the bottom anchor only while the user is already following latest content, and it never forces a conversation back to the end after the user scrolls upward. Env App continues to use `VirtualMessageList` row `ResizeObserver` handling: following mode applies each intermediate height delta, while paused mode restores the captured viewport anchor after every observed height. Terminal panels use a separate local 24 px near-bottom threshold. Binding or reopening a terminal viewport begins a new local viewing session at the latest line. Each output update receives one presentation frame: following mode scrolls to the bottom before reporting the output as presented, while paused mode preserves `scrollTop` and still reports presentation completion. Upward wheel or scroll pauses local following, returning to the bottom resumes it, and canonical activity replacement does not reset that choice. Output merging, polling cadence, canonical status ownership, and persistence are unchanged. Reduced-motion presentation suppresses routine automatic expansion, keeps attention and manual access immediate, commits measured pixel height without animation, and unmounts closed detail immediately. These presentation rules do not alter Floret activity status, event ordering, persistence, read state, audit state, or terminal settlement.

Flower does not derive the current pending ordinary-tool approval from activity rows, transcript rows, audit records, or empty assistant block projections. Redeven projects ordinary pending approvals from Floret `ListPendingApprovals` only, and it maps that product-neutral snapshot to Flower `approval_actions` before bootstrap, `timeline.replaced`, and `approval.requested` delivery. Activity rows render the Floret `ActivityTimeline` item presentation that originated from Redeven `ToolPresentationSpec`; the UI does not look up a pending approval by run id/tool id to replace row labels or payload, and it does not hide duplicate tool/approval rows as a fallback. A Floret tool activity item with `approval_state=requested` must remain a single waiting row until explicit approval resolution, and Redeven validates that contract at the Floret projection boundary instead of repainting invalid lifecycle states in the UI. Resolved activity can remove or mark old approval actions, but an activity timeline entry with `approval_state=requested` is audit history unless the Floret pending snapshot still contains the matching approval. `requires_approval=true` without `approval_state=requested` and `status=waiting` is not a UI lock: it records historical approval participation while allowing terminal and structured detail expansion.

Flower does not infer read snapshots from run status, timestamps, message previews, context telemetry, or local activity tables. It applies `read_status` delivered by bootstrap/list/patch payloads and only persists read state for the currently selected thread. Selected-thread read sync updates local `read_status` from the `/read` response; if the response is still unread, Flower only queues another persistence attempt when the returned snapshot key differs and the thread remains selected.

Flower does not treat the global current model as an implicit existing-thread launch override. The current model is the default for the next newly created thread only. After a thread exists, its `model_id` and `reasoning_selection` are the displayed defaults; composer model changes write through to `current_model_id` only as a future new-thread default and never as a reverse sync into existing threads.

Flower does not parse provider request metadata, database rows, or transcript text to estimate context pressure. It renders context usage and compaction only from typed live/bootstrap fields projected by Redeven from structured Floret observations. The composer last-known context indicator can reuse a previous `context_usage` value, but it must label that freshness explicitly and must not synthesize a replacement ratio while waiting for the current run's Floret observation. Flower slash commands are local composer actions; `/compact` does not enter the transcript and does not cancel or replace a running agent turn.

Flower does not synthesize Sub Agents dropdown state from activity timeline sidecars, audit tables, provider-adapter side effects, child assistant text, Redeven child thread rows, or a separate transcript block type. `thread.subagents`, sourced from Floret `ListSubAgents`, is the only parent header child-list contract, and thread list summaries that omit `subagents` are not allowed to clear selected-thread detail state. Activity rows may expose `Open messages` only from sanitized public `thread_id`/`subagent_id` payload fields, not from hidden sidecars or fallback transcript scans. Full child execution detail belongs to the parent-scoped subagent detail API and is never injected into the parent model context by default. Subagent detail live tailing is a UI-only read path with request identity guards; it must not become a model-facing wait result, a second chat composer, or an alternate thread navigation target. Preview-only Floret detail rows cannot be used as a fallback for complete subagent output, and increasing a preview limit is not a substitute for a complete human-display contract. Redeven does not create read-only child thread rows for Floret subagents; child Flower rows are not a supported display source or compatibility path.

The Sub Agents detail window is a read-only execution ledger owned by the parent Flower surface, not a compact child chat. Its overview presents the canonical child status together with task description, agent type, and elapsed time. The ledger classifies projected user and system messages as collapsed delegated instructions or runtime constraints and keeps assistant narrative open for reading. Its presentation model groups adjacent pure canonical activity messages into one operation phase without rebuilding, copying, or persisting their activity items or statuses. Visible narrative, instruction, constraint, input, compaction, error, outcome, or mixed content/activity entries end the current phase, so tool work before and after an assistant explanation remains separate. A multi-operation phase renders one summary and preserves each canonical tool row underneath it; a phase containing one operation renders that tool row directly without a redundant one-operation heading. Successful phases larger than six operations start collapsed, while running, waiting, canceled, or failed phases remain open so active and exceptional work is not hidden. Each phase key derives from its first activity entry, preserving the user's disclosure state when live tailing appends or settles later tools. Terminal details omit an empty status dock; live model state, tail synchronization, tail errors, and pagination render the dock only while they have actionable content. Initial and incremental detail failures provide local retry actions without creating a child composer, changing parent-thread selection, or claiming that preview-mode rows are a complete transcript.

Redeven may publish `timeline.replaced` only as an explicit resync, snapshot recovery, or terminal settlement finalization event. It is not a high-frequency companion for streaming text, block updates, committed messages, run status changes, or thread patches. Desktop and Env App adapters both refresh the live bootstrap after thread-level stop operations, so the surface receives the same canonical timeline after a stop-only or stop+send interaction.
