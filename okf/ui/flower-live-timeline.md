Flower is a reusable chat surface over a Floret-owned Agent timeline. Floret owns admitted user and assistant content, turn/run lifecycle, canonical ordinal order, failure and waiting state, projection, control signals, approvals, and Agent todos. Redeven maps that public timeline into Flower, adds product resource decorations, and keeps only short-lived live drafts with exact canonical identity.

# Mechanism

The shared Flower settings contract separates `defaults.permission_type`, nullable `model_profile`, and optional `model_source`. Its adapter exposes independent `saveDefaultPermission`, `saveModelProfile`, and `persistDefaultModel` operations. The permission control is always present and uses a 700 ms serialized autosave controller that coalesces rapid changes, never allows response order to overwrite a newer choice, and restores the last confirmed value after failure. An absent environment profile is a neutral setup state rather than a global Flower error. A `remote_desktop` Env App session may expose the Desktop model catalog alongside the environment profile in the same snapshot; `local_host` and ordinary browser sessions ignore the unbound Desktop diagnostic object. The settings surface always edits the environment provider profile. Desktop source status never replaces the provider editor with a managed-source panel, and Desktop model selection belongs in the composer menu.

The Flower live bootstrap includes `timeline_messages` as the only ordered message source for a thread snapshot. Redeven obtains before/after/tail pages from Floret `ListThreadTurns` and preserves their ascending canonical ordinal order. Each `ThreadTurnSnapshot` maps to the Floret `UserEntryID` user row followed by that same turn's validated assistant projection or one anchored `turn_projection_unavailable` decoration. Tool display details come from the projection `ActivityTimeline` plus Redeven's `ToolPresentationSpec`. A live draft may replace only the assistant slot whose `ThreadID`, `TurnID`, and `RunID` exactly match the canonical turn; terminal projection removes it. Redeven stores no transcript, correlation row, run record, projection cache, preview, or ordering copy.

Projection application is strict and monotonic. Floret `ThreadTurnProjection.Validate()` owns identity completeness, finite status and segment kinds, positive durable `ThroughOrdinal`, activity timeline validity, and control-signal structure. Redeven checks only that the validated projection belongs to the current Flower run/thread/turn, then maps assistant text, activity timelines with Redeven file actions, and non-body control segments. Live projections accept `running`; history assembly separately requires `Status.IsTerminal()`. Redeven keeps the last applied ordinal for each full execution identity, ignores equal or older projections, and advances the cursor only after the complete projection maps and replaces the assistant blocks. A valid zero-segment projection clears the in-memory assistant block list without synthesizing an empty markdown block. Runtime events are accepted only after Floret `runtime.Event.Validate()` validates their public event type and nested stream, activity, and projection data. Invalid contracts record `floret.contract.rejected` and leave the current presentation unchanged; private harness events never enter the public runtime sink.

Published Floret v0.11.4 includes the complete projection in each admitted public turn snapshot, provides a canonical latest-turn read that excludes marker-only turns until their user entry is committed, and reports projection availability separately for live run and pending-settlement results. Floret validators own availability combinations, identity, status, through ordinal, control signals, approvals, and settlement agreement. An invalid snapshot or projection is a canonical consistency error and requests resynchronization; Redeven does not turn it into display content. Only a valid projection with no renderable assistant blocks maps to one `turn_projection_unavailable` decoration with reason `not_renderable` immediately after that turn's user row. Flower renders “Response unavailable” with no retry control and no invented assistant body. Storage, I/O, pagination, unknown row cursors, and same-revision thread consistency failures remain request errors rather than being downgraded. Bootstrap, paged history, and `timeline.replaced` never recover content from metadata, product run errors, audit records, previews, or stale live drafts.

Timeline decorations are a strict discriminated union. `context_compaction` owns only a compaction payload, while `turn_projection_unavailable` owns only its projection-unavailable payload and finite reason. Unknown kinds, missing payloads, mixed payloads, invalid reason codes, or missing message anchors fail mapping explicitly. `ListThreadMessagesResponse`, live bootstrap, and full replacement snapshots carry `timeline_decorations` beside the messages produced by the same timeline projection, so a valid assistant projection and an unavailable decoration are mutually exclusive for one turn.

Thread snapshots, list items, and thread patches carry `model_id` and `reasoning_selection` as thread defaults. Flower renders a fused model/reasoning segmented control in the composer when model options are available; the reasoning segment is shown only when the selected model exposes controllable reasoning capability. A ready Desktop model source contributes a read-only model catalog alongside editable environment provider models, preserving each opaque model id, display label, token limits, modalities, and reasoning capability. Every composer option has an explicit source: `model_profile`, `desktop_model_source`, or `thread_snapshot`. `thread_snapshot` is a disabled, ungrouped option used only when an existing thread references a model no longer present in either catalog; Flower does not regroup or silently replace that model. When both active catalogs exist, the compact menu groups the environment models first under the environment display name and Desktop models second under the fixed `Desktop` name; a single active source remains a flat list. A new Flower mount selects the environment profile's persisted `current_model_id`, falling back to the Desktop current model only when the environment profile is absent. Selecting another environment model calls `persistDefaultModel`; selecting a Desktop model writes only the mounted surface's pending-new-thread draft. Repeated New Chat actions in that mount retain the draft, while a remount restores the environment default. Creating a new thread always sends the chosen draft model explicitly, and draft reasoning is included only when that model supports reasoning control. Switching to a model without reasoning removes any draft reasoning override before launch. An existing thread always resolves its own `model_id` first. Changing it sends `setThreadModel`; an environment-model change then persists the environment default, while a Desktop-model change stops after the thread PATCH. Returning to New Chat restores the mount's pending-new-thread choice without copying the thread model into the environment default. Readiness is evaluated for the selected option's source: environment models require their provider and web-search keys, while Desktop models require a `ready` source containing that exact model id. If the selected model is unavailable but another source is usable, the composer and model menu remain visible, send stays disabled, the model trigger shows a compact warning, and the menu shows one compact Desktop status row without a global banner. If no model from either source is usable, Flower keeps the ordinary empty state and renders one single-line composer status row with warning, refresh, and one owner-specific action; it does not render a transcript-sized setup guide. Refresh calls only `adapter.loadSettings()` and atomically replaces the settings snapshot, preserving thread selection, input, attachments, mounted Desktop draft, and environment default. `missing_keys` and `empty` recover through local Flower settings, `unsupported` through remote Runtime settings, and `unbound`, `connecting`, `expired`, or `error` through Desktop Connection Center. Existing-thread launch does not add a global model override, so subsequent turns use the persisted thread default unless the request is intentionally local to a waiting-input response. The model/reasoning segmented control is disabled while the selected thread is active, waiting for user input, compacting, read-only, or otherwise unable to accept a thread preference change. The Flower UI quality gate explicitly runs a cross-layer Desktop model-source test that renders the shared surface over the real Env adapter and verifies catalog projection, labels, reasoning defaults, thread creation, and first-turn RPC payloads together.

Flower treats user-triggered operation failures and input blockers as transient feedback, not transcript content. The shared surface emits a host notification intent for send failures, invalid or blocked submits, model/default-model persistence failures, permission changes, approval submits, and sidebar actions; Env App and Desktop render that intent as a toast through their own notification systems. These toasts do not change the canonical thread timeline, and Flower no longer inserts composer or sidebar error cards into the page flow for those transient actions. Run failures, tool failures, subagent detail errors, and other structured timeline/detail content remain rendered by their owning timeline or detail renderer rather than being guessed from the notification channel.

Thread navigation separates visual, requested, and displayed ownership. A user click marks the list row immediately, then commits selection after the intent paint. A thread whose detail is already in the explicit warm set is displayed atomically without calling `loadThread` again. A cold target loads in staging while the previous transcript remains visible and read-only; only the newest request may publish the target transcript, focus the composer, restore tail position, or emit `content_presented`. Compose mode, permission changes, programmatic focus ownership, and newer thread requests cancel or supersede the pending user transaction. Codex uses the same product contract with separate selected, foreground, displayed, loading, and session-cache state, so warm sessions switch without provider or transcript remount and stale loads cannot replace the current thread.

Live streaming uses server events only for current-process presentation. `message.started` may create one assistant streaming row and marks it as the single active cursor. There is no realtime transcript-message or transcript-reset event, and user messages are never inserted through the live stream. Floret text deltas may render temporary live markdown, while a validated `runtime.Event.Projection` with status `running` replaces only the matching live assistant draft with current assistant text and activity blocks. Aggregate Floret `runtime.Event.ActivityTimeline` remains lifecycle observation data and does not create main-thread activity blocks by itself. A local tool row is pending when Floret has observed the model's tool call but has not dispatched the handler; it becomes running only from Floret's dispatch-start lifecycle fact after permission and approval gates pass. Batched ordinary calls may start and finish in actual execution order, and approval display order never serializes handler execution. A projection that fails Floret validation or cannot be mapped is rejected as a unit and does not clear existing live content; a valid zero-segment projection clears the assistant blocks without inventing a body block. Stop and hard-cancel paths update current-process presentation but do not persist a canceled assistant row. Activity timeline changes enter the UI only inside Floret-derived projection activity segments delivered through `message.block_set`, a live draft, or a projection-backed timeline snapshot. Ordinary incremental events do not publish companion full timeline replacements. Every ordinary terminal turn and pending terminal settlement deletes the exact matching draft and publishes one `timeline.replaced` snapshot rebuilt from `ListThreadTurns`; an identity or ordinal mismatch requests resynchronization. `stream_generation` remains a Service-instance epoch so restart cannot let a stale high cursor hide new events.

Terminal process activity remains part of that canonical projection. A running `terminal.exec` row receives its command, status, process id, cumulative retained snapshot, and sequence facts from the Floret `ActivityTimeline` payload produced by Redeven tool presentation metadata. `terminal.exec` publishes the process id as soon as the PTY starts through Floret's running activity update path, so `yield_ms` affects only when the agent receives the tool result, not when the user can inspect command progress. Each model-issued `terminal.read` activity uses its required user-language description as the compact row title, while the returned `payload.command` remains the real command shown only in expanded terminal details. Result presentation omits a label so Floret preserves the call intent, and Flower does not synthesize generic terminal-output titles. Each read remains a separate activity row and its expanded panel shows only that call's `output` delta: a first read can show phase 1 and the next read can show phase 2 without making the command appear to have executed again. `terminal.terminate` follows the same presentation contract: its required user-language description remains the row title through the successful result, the tool name is not used as a visible fallback, and the expanded command detail shows the real command plus the public `process_id` needed for inspection and diagnostics. That public payload must not expose host-local path or input fields such as `cwd`, `workdir`, or `stdin`.

Expanded terminal rows are read-only product presentation. Completed status, output, exit code, and duration come directly from the canonical Floret Activity payload and never call a Redeven completion-output endpoint. The shared terminal store applies two explicit sequence operations while the Activity remains running: a newer exec snapshot replaces the cumulative view, while a contiguous PTY process-read delta appends to it. Equal or older snapshots are ignored by sequence, an empty delta does not clear existing exec output, and an explicit `truncated=true` gap replaces the unavailable prefix with the retained delta. The store never decides cumulative-versus-incremental behavior with string prefix or substring checks. Flower and Env App send only the required `after_seq` process-read query for a running Activity with a public process id; server-owned waits and page limits are not UI options. Polling may continue while details are collapsed so a stored delta is not lost merely because it was not visible when it arrived. A valid process read can return a terminal `success`, `error`, or `canceled` snapshot when the PTY finishes between polls, including an empty delta whose `last_seq` remains equal to `after_seq`; that response stops local polling but can never change canonical Activity status, create lifecycle facts, or trigger settlement. Raw AppServer errors are logged for diagnostics and render only localized live-output or terminal-action failure text. The surface exposes icon-only actions to reveal and copy the full raw command and caps output at five visual line boxes using explicit font size, line height, padding, and border-box sizing; shorter output uses its natural height, while later lines increase only the local scroll height. Every pending terminal process binds its complete Floret settlement target and creating top-level or subagent Host before launch. After the pending handle is exposed and PTY reap completes, one finalizer performs one settlement through that owner; process reads, writes, cleanup, and later UI access never trigger or retry settlement, and no maintenance Host fallback exists. A ready settlement projection advances the same `ThroughOrdinal` cursor as every other projection and publishes the canonical `timeline.replaced` snapshot. An unavailable settlement projection remains committed Floret state, produces diagnostic logging, and is not read back or synthesized locally. `terminal.terminate` waits for process-tree exit, output drain, and finalization before returning, while `Service.Close` completes finalizers before releasing Floret Hosts and stores. Consequently, canonical terminal ordering and completion facts come only from durable Floret projections rather than a terminal-specific database, retry path, floor, refresh, or UI repair mechanism.

Subagent presentation uses two separate public contracts. Thread-level child membership comes from Floret `ListSubAgents` through the active parent host or provider-free `ThreadMaintenanceHost`; Redeven maps that product-neutral snapshot into `thread.subagents` during live bootstrap and publishes the same list through `thread.patched` after runtime child events. Thread list summaries do not own `thread.subagents`: if a refresh summary omits the field, Flower preserves the selected thread detail value, and only an explicit detail or live patch value replaces it. The parent header Sub Agents dropdown is derived only from `thread.subagents`, so a runtime restart can restore the count and child list without scanning transcript activity blocks. New child names are canonical 1-5 word English presentation labels normalized before Floret spawn, while the UI humanizes historical snake/kebab/camel names and applies an English role fallback when no English words are available, without migrating stored state. Redeven does not read Floret store tables, create child threadstore rows, or persist a duplicate mapped child list. Parent timeline rows still come from Floret `ThreadTurnProjection` activity segments for the `subagents` tool. Their public payload is sanitized to route and display facts only: `thread_id`, `subagent_id`, `parent_thread_id`, task name/title, `task_description`, agent type, status, timestamps, counts, timeout/truncation flags, and public error/summary fields. It filters raw child content, host-local paths, detail refs, context/debug fields, `last_message`, `waiting_prompt`, `can_*` control flags, queued input counts, and fields outside the current subagent payload allowlist. Inline `Open messages` actions read only the public `thread_id`/`subagent_id` facts in the activity payload; there is no activity sidecar fallback. The dropdown is a header-triggered floating surface, not a page or side panel: it uses the shared Floe surface layer and named Flower layer tokens, displays neutral compact chrome with localized active/ended metrics, and keeps active and ended sections visible. Each row exposes one status icon, the canonical display name, one-line responsibility text, localized status, elapsed duration when timestamps are available, and a detail chevron. Active rows sort waiting-input, running, queued, then unknown; ended rows sort failed, timed-out, completed, then canceled, with update time as the secondary order. Opening the dropdown focuses the first active row; arrow keys, Home, End, Enter, and Escape provide complete keyboard navigation, and reduced-motion mode removes popover and status animation. Selecting a child never selects or navigates to the child thread; it loads parent-scoped detail through the surface adapter and opens a read-only Floe `FloatingWindow` over the current parent conversation. The detail window reuses the same timeline/message/activity renderers as the main thread, adds a bottom live status lane, tails active child detail incrementally, auto-follows only while the user is at the bottom, and exposes a scroll-to-latest affordance when the user reviews earlier output. Detail tailing continues while the child is queued, running, or waiting for input even when the current page reports `has_more=false`; terminal child status stops the tail, and transient tail errors stay in the detail dock with a longer retry interval. The detail tail and full child execution timeline are human UI state only; they are never injected into the parent model context by default. The overlay renders Floret `ReadSubAgentDetail` through the parent thread scope: user/assistant messages, tool results, approvals, turn markers, compactions, errors, and generic custom rows come from the child detail events, while tools render from the top-level canonical child activity timeline. Bounded preview detail rows are inspection data and must not be presented as complete child user or assistant output; a human detail surface that needs complete visible child output must consume an explicit Floret display-detail contract rather than mapping preview text into the transcript body. Tailing updates replace that stable activity block instead of appending event-row activity snapshots, so older running tool updates disappear when a later result or settlement becomes canonical. Detached reads use provider-free `ThreadMaintenanceHost` and do not require provider configuration, a child threadstore row, or child Flower metadata in Redeven. Summary rows, when present, are explicit Floret detail data rendered in the same read-only message model. Subagent model results and activity payloads support canonical `items` only, and unsupported detail, transcript, raw-output, or non-allowlisted collection fields are filtered before presentation.

Active cursor ownership belongs to Redeven's live projection. The projection marks at most one live assistant draft with `active_cursor` while the selected thread is running, and canceled, completed, idle, background, or merely streaming-shaped messages are not treated as active unless the projection marks them active. The visible model status affordance is a localized bottom dock lane driven by `model_io.updated`, not by `active_cursor` or thread running status. `model_io` is derived in memory from validated Floret provider lifecycle and stream observations: provider request maps to waiting for model response; assistant, reasoning, and model tool-call stream observations map to thinking; provider retry maps to retrying; provider finish maps to finalizing. Redeven does not persist Floret step, run, context-continuation, or compaction lifecycle mirrors. Provider diagnostics use typed finish fields; metadata cannot replace lifecycle facts.

Current approval decisions belong in the composer and follow the in-memory product projection of Floret's pending approval snapshot. The queue snapshot carries generation, revision, current action id, current position, total, and unresolved count; each action carries immutable queue order, queue generation, batch index, and batch size. When the selected thread has a pending `primary_action`, the composer body replaces the text or secret input with exactly that approval panel, and the footer primary controls become Reject and Approve. Other pending actions remain read-only locators. A decision validates the current Floret pending identity before release, and Redeven records only the authorization audit. Refresh, reconnect, and restart rebuild approval presentation from Floret; no Redeven approval request, lifecycle event, idempotency, or outbox row is queryable as a second authority. Historical activity may show approval participation but cannot create current actionable state.

Context usage and compaction state share one Floret-owned canonical model while using distinct Flower presentation affordances. During an active process, `context.usage.updated` and `context.compaction.updated` update only the in-memory live materialized state. Bootstrap and reconnect call `ThreadMaintenanceHost.ReadThreadContext`, validate the returned snapshot, and map its current usage plus compaction history into the response; Redeven does not persist those mapped values or a context cursor. A compaction divider is keyed only by typed `operation_id`, with typed `request_id` and `source`; metadata aliases and generated identities are rejected. The divider is not a transcript message and does not affect message ordering, read revision, copying, or model-visible history.

Flower `/compact` is a composer command, not a chat message. Running threads forward the request into the active Floret hosted-turn lifecycle. Idle threads use an operation-only background path that owns only product command admission, queueing, mutual exclusion, cancellation, and the in-memory running divider; the worker calls Floret `Host.CompactThread` and validates the canonical terminal result. Redeven does not synthesize noop/completion lifecycle, commit provider continuation, or run a context-boundary compensation transaction. A cancellation before Floret starts ends only the product operation. Cold bootstrap reads terminal compaction history from `ReadThreadContext`; it never replays an uncommitted in-memory running divider. User turns sent during idle compaction enter the queued follow-up lane and start through the ordinary turn path after the operation clears.

Before Floret admission, Redeven may persist a pending command with a pre-generated opaque turn/run identity, prompt, product resource references, and launch options. No assistant draft is published at this stage. Floret admission creates the canonical user entry and turn; after `ListThreadTurns` confirms the exact `TurnID`, Redeven removes the pending prompt and retains only allowed product references. A crash before admission leaves the command pending, while a crash after admission is reconciled by exact Floret identity. Secret answers never enter plaintext persistence.

Selected-thread live polling treats `model_io.updated` as a presentation boundary. Even when one live-events response contains a short provider stream followed by finalizing or clearing events, Flower commits the model status lane and yields a render frame at each model I/O boundary. This keeps brief but real provider streaming phases visible to the user without inventing timeline rows, fallback timers, or thread-status heuristics.

Thread read state is a user-scoped product acknowledgement over a canonical Floret activity snapshot. `ThreadView.FlowerActivity` derives its revision and signature from Floret `ThroughOrdinal`, latest canonical message time, and waiting signal identity at read time; those values are not stored on `ai_threads`. The read-state store persists only the user's acknowledgement. Floret context telemetry does not advance this product acknowledgement.

Mark-read requests echo a snapshot previously delivered by the server. Future snapshots are rejected, current-revision signature, waiting-prompt, or last-message mismatches remain protocol errors, and stale snapshots are accepted without pretending they covered newer activity; the response still returns the current authoritative `read_status`. The shared Flower adapter returns that `read_status` directly instead of reloading the bootstrap. A selected running thread that completes while the user keeps it selected is therefore marked read from the final snapshot by the Flower surface, while background completions remain unread until selected.

# Boundaries

Flower does not sort messages, infer canonical insertion points from timestamps or IDs, or pair recent user messages with assistant drafts. `timeline.replaced` replaces the reducer message array in server order. A streaming assistant row requires exact canonical thread/turn/run identity; an unmatched draft triggers resynchronization and is never appended. Local pending user rows are composer feedback for commands not yet admitted by Floret and never become durable history by themselves.

Flower does not rebuild Floret execution order from Redeven database rows, audit data, provider deltas, previews, or tool-call records. Floret `ListThreadTurns` is the durable message and ordering source; live runtime projections and pending settlement projections may update only the matching canonical turn. Flower does not scan cached block JSON or activity rows to repair pending, waiting, or running tool states after stop or approval. Redeven does not publish independent activity blocks from aggregate observations, synthesize final rows, hide queued siblings, or turn an invalid projection into an empty assistant block.

Flower activity details are renderer-first product views, not generic payload inspectors. `error`, `terminal`, `file`, `patch`, `web_search`, `question`, `completion`, `todos`, and `subagents` each have an explicit display model that prioritizes the information a normal user is trying to inspect: failure reason, terminal output, file content, diffs, search results, prompts and choices, outcomes, todo lists, or delegated task status. Failed tools render the public error message first and do not expose error codes, retryability flags, item ids, tool ids, or lifecycle diagnostics in the ordinary detail body. Successful tools do not show generic completion summaries or event-code details such as `tool execution completed`, `todos.updated`, `file.updated`, `success`, `completed`, or `ok`. A compact structured metadata block remains available only for allowlisted product details that do not belong to the primary renderer. Flower must not fall back to displaying every payload key, or raw `status`/`kind`/`tool`/`item` rows, as the main detail experience.

Flower activity disclosure is presentation state over the canonical activity lifecycle, not a second execution state machine. Every stateful activity view uses one semantic identity composed from thread, run, turn, and canonical `item_id`; status, payload, message block position, item position, and renderer object identity are not part of that identity. `timeline.replaced` may move an activity between message blocks, but it must reuse the same activity row, disclosure controller, detail component, and terminal viewport. A live projection may first expose the stable item without a renderer or detail payload and later enrich that same item; the row keeps its DOM identity, while the disclosure trigger remains bound and its native disabled state follows detail availability. Routine `pending` or `running` detail waits 400 ms before opening, so activity that settles sooner remains compact and never produces an expand-collapse flash. Opening uses a 360 ms measured-height animation. An automatically opened terminal command starts its 1200 ms completion hold only after the canonical `success` or `canceled` presentation and latest visible output have completed their terminal presentation frame; another settled output frame restarts the full hold. The detail then uses a 300 ms closing animation. Other routine activity remains intent-anchored. Error, waiting, approval, or blocking attention opens immediately and remains open until the user changes it. A manual expand or collapse overrides later lifecycle updates for that stable activity item, and pointer, focus, wheel, or touch interaction inside automatically opened detail converts it to an explicit open choice so the surface cannot close content while the user is reading, selecting, scrolling, or operating it. Current ordinary approval rows remain non-expandable when the composer owns the actionable approval.

Open activity detail owns its real pixel height. A shared disclosure motion controller mounts at `0px`, measures the nearest detail content from its bounding rectangle, and uses the Chromium `ResizeObserver` and Web Animations APIs as required platform capabilities. Opening, resizing, retargeting, reversal, and closing are all owned by one current animation; its `finished` promise is the only completion signal that may advance presence state or unmount content. CSS does not own a second height-transition clock, and animation completion is not inferred from elapsed timers. Consecutive content changes cancel the previous animation and redirect the current visual height to the latest measured target. Content growth and shrinkage use a 280 ms animation, including terminal polling, command expansion, responsive reflow, error replacement, and disappearing controls. The nearest detail disclosure owns this dynamic resize; the Env App timeline wrapper animates only its own open or close, preventing nested height animations from compounding movement. Detail content remains capped at `min(42rem, 72vh)`, and terminal output remains capped at five visual lines; growth beyond those limits becomes local scrolling instead of continuing to move the conversation.

Flower schedules its existing transcript-tail follower on every disclosure animation frame. It preserves the bottom anchor only while the user is already following latest content, and it never forces a conversation back to the end after the user scrolls upward. Env App continues to use `VirtualMessageList` row `ResizeObserver` handling: following mode applies each intermediate height delta, while paused mode restores the captured viewport anchor after every observed height. Terminal panels use a separate local 24 px near-bottom threshold. Binding or reopening a terminal viewport begins a new local viewing session at the latest line. Each output update receives one presentation frame: following mode scrolls to the bottom before reporting the output as presented, while paused mode preserves `scrollTop` and still reports presentation completion. Upward wheel or scroll pauses local following, returning to the bottom resumes it, and canonical activity replacement does not reset that choice. Output merging, polling cadence, canonical status ownership, and persistence are unchanged. Reduced-motion presentation suppresses routine automatic expansion, keeps attention and manual access immediate, commits measured pixel height without animation, and unmounts closed detail immediately. These presentation rules do not alter Floret activity status, event ordering, persistence, read state, audit state, or terminal settlement.

Flower does not derive the current pending ordinary-tool approval from activity rows, transcript rows, audit records, or empty assistant block projections. Redeven projects ordinary pending approvals from Floret `ListPendingApprovals` only, and it maps that product-neutral snapshot to Flower `approval_actions` before bootstrap, `timeline.replaced`, and `approval.requested` delivery. Activity rows render the Floret `ActivityTimeline` item presentation that originated from Redeven `ToolPresentationSpec`; the UI does not look up a pending approval by run id/tool id to replace row labels or payload, and it does not hide duplicate tool/approval rows as a fallback. A Floret tool activity item with `approval_state=requested` must remain a single waiting row until explicit approval resolution, and Redeven validates that contract at the Floret projection boundary instead of repainting invalid lifecycle states in the UI. Resolved activity can remove or mark old approval actions, but an activity timeline entry with `approval_state=requested` is audit history unless the Floret pending snapshot still contains the matching approval. `requires_approval=true` without `approval_state=requested` and `status=waiting` is not a UI lock: it records historical approval participation while allowing terminal and structured detail expansion.

Flower does not infer read snapshots from run status, timestamps, message previews, context telemetry, or local activity tables. It applies `read_status` delivered by bootstrap/list/patch payloads and only persists read state for the currently selected thread. Selected-thread read sync updates local `read_status` from the `/read` response; if the response is still unread, Flower only queues another persistence attempt when the returned snapshot key differs and the thread remains selected.

Flower does not treat the global current model as an implicit existing-thread launch override. The current model is the default for the next newly created thread only. After a thread exists, its `model_id` and `reasoning_selection` are the displayed defaults; composer model changes write through to `current_model_id` only as a future new-thread default and never as a reverse sync into existing threads.

Flower does not parse provider request metadata, database rows, or transcript text to estimate context pressure. It renders context usage and compaction only from typed live/bootstrap fields projected by Redeven from structured Floret observations. The composer last-known context indicator can reuse a previous `context_usage` value, but it must label that freshness explicitly and must not synthesize a replacement ratio while waiting for the current run's Floret observation. Flower slash commands are local composer actions; `/compact` does not enter the transcript and does not cancel or replace a running agent turn.

Flower does not synthesize Sub Agents dropdown state from activity timeline sidecars, audit tables, provider-adapter side effects, child assistant text, Redeven child thread rows, or a separate transcript block type. `thread.subagents`, sourced from Floret `ListSubAgents`, is the only parent header child-list contract, and thread list summaries that omit `subagents` are not allowed to clear selected-thread detail state. Activity rows may expose `Open messages` only from sanitized public `thread_id`/`subagent_id` payload fields, not from hidden sidecars or fallback transcript scans. Full child execution detail belongs to the parent-scoped subagent detail API and is never injected into the parent model context by default. Subagent detail live tailing is a UI-only read path with request identity guards; it must not become a model-facing wait result, a second chat composer, or an alternate thread navigation target. Preview-only Floret detail rows cannot be used as a fallback for complete subagent output, and increasing a preview limit is not a substitute for a complete human-display contract. Redeven does not create read-only child thread rows for Floret subagents; child Flower rows are not a supported display source or compatibility path.

The Sub Agents detail window is a read-only execution ledger owned by the parent Flower surface, not a compact child chat. Its overview presents the canonical child status together with task description, agent type, and elapsed time. The ledger classifies projected user and system messages as collapsed delegated instructions or runtime constraints and keeps assistant narrative open for reading. Its presentation model groups adjacent pure canonical activity messages into one operation phase without rebuilding, copying, or persisting their activity items or statuses. Visible narrative, instruction, constraint, input, compaction, error, outcome, or mixed content/activity entries end the current phase, so tool work before and after an assistant explanation remains separate. A multi-operation phase renders one summary and preserves each canonical tool row underneath it; a phase containing one operation renders that tool row directly without a redundant one-operation heading. Successful phases larger than six operations start collapsed, while running, waiting, canceled, or failed phases remain open so active and exceptional work is not hidden. Each phase key derives from its first activity entry, preserving the user's disclosure state when live tailing appends or settles later tools. Terminal details omit an empty status dock; live model state, tail synchronization, tail errors, and pagination render the dock only while they have actionable content. Initial and incremental detail failures provide local retry actions without creating a child composer, changing parent-thread selection, or claiming that preview-mode rows are a complete transcript.

Redeven may publish `timeline.replaced` only as an explicit resync, snapshot recovery, or terminal settlement finalization event. It is not a high-frequency companion for streaming text, block updates, committed messages, run status changes, or thread patches. Desktop and Env App adapters both refresh the live bootstrap after thread-level stop operations, so the surface receives the same canonical timeline after a stop-only or stop+send interaction.
