Flower is a reusable chat surface over a Redeven-owned product timeline. Redeven owns thread persistence, turn anchoring, live drafts, and message ordering; Flower renders the ordered messages that Redeven projects for the selected thread.

# Mechanism

The Flower live bootstrap includes `timeline_messages` as the only ordered message source for a thread snapshot. Redeven builds product timeline messages from persisted transcript rows, conversation turn anchors, and live assistant drafts. For Floret-hosted assistant execution, the block order inside the assistant message is derived from Floret ordered thread detail events: assistant text entries and tool activity entries are interleaved by Floret ordinal, with tool display details coming from the Floret `ActivityTimeline` plus Redeven's `ToolPresentationSpec`. Turn anchors order each user message beside its assistant message, while live drafts can replace the matching assistant message before the final assistant transcript row is committed.

Thread snapshots, list items, and thread patches carry `model_id` and `reasoning_selection` as thread defaults. Flower renders a model selector in the composer when model options are available. On a new thread, the draft model and reasoning selection are sent with the create-and-launch request; on an existing thread, changing the model sends a thread PATCH through `setThreadModel`, and changing the composer reasoning default sends the thread reasoning PATCH. Existing-thread launch does not add the global current model to the send request, so subsequent turns use the persisted thread default unless the request is intentionally local to a waiting-input response. The model and default reasoning controls are disabled while the selected thread is active, waiting for user input, compacting, read-only, or otherwise unable to accept a thread preference change.

Live streaming uses canonical server events to create only the rows those events define. `message.started` may create or merge one assistant streaming row and marks it as the single active cursor. User messages are not locally invented; a canonical `message.committed` payload upserts the committed message by id, so a newly accepted user turn can appear in the current window without a placeholder. Floret text deltas may render temporary live markdown, but Redeven reconciles durable assistant block order from Floret committed thread events during the run and from `ListThreadDetailEvents` when the turn completes. Stop materialization immediately persists the canceled assistant boundary so the next user message can be ordered correctly; if Floret later returns the same run and turn's terminal projection, Redeven updates that same assistant message and broadcasts it through `message.committed`. Block, activity, or failure updates still require their target message to exist; activity timeline changes enter the UI only as Floret-derived `activity-timeline` message blocks delivered through `message.block_set` or through a committed assistant message. Ordinary `message.block_delta`, `message.block_set`, `message.committed`, `run.status_changed`, and `thread.patched` events do not automatically publish companion full timeline replacements. If local state is missing, the reducer marks the batch as needing resync. Explicit `timeline.replaced` snapshots are reserved for resync and snapshot recovery; the payload carries `stream_generation` and `snapshot_through_seq`, and the reducer replaces the full message array while advancing the cursor to that snapshot boundary. `stream_generation` is a Service-instance epoch rather than a protocol constant. When the selected thread observes a newer generation after holding an older cursor, Flower resets the live cursor for that thread before fetching low-sequence events from the new stream, so runtime restart or stream rebuild cannot cause a stale high cursor to hide new events.

Subagent presentation is also derived from the canonical timeline. The parent thread records delegation tool activity through Floret `ActivityTimeline` payloads for `subagents`; Redeven refreshes that parent activity timeline through Floret `ListSubAgentActivityTimeline`, so asynchronous child completion updates the parent Sub Agents dropdown without an extra UI state source. Floret owns child lifecycle status reduction, ordering, counts, attention reasons, and engine payloads. Redeven adds Flower-only presentation metadata through the `ActivityTimelineBlock.subagent_actions` sidecar keyed by activity item id, so fields such as `operation=subagents`, `action=inspect`, `delegation_runtime=floret`, normalized agent type, and context mode do not get written back into the Floret activity payload. The Flower surface projects the dropdown from the canonical item payload plus that sidecar, merges child items by child thread id, and shows active and settled counts in the selected parent thread header. The dropdown is a header-triggered floating surface, not a page or side panel: it uses the shared Floe surface floating layer, named Flower layer tokens, an opaque popover surface, and the same running loader language as inline tool activity rows. Selecting a child never selects or navigates to the child thread; it loads parent-scoped detail through the surface adapter and opens a read-only Floe `FloatingWindow` over the current parent conversation. The detail window reuses the same timeline/message/activity renderers as the main thread, adds a bottom live status lane, tails active child detail incrementally, auto-follows only while the user is at the bottom, and exposes a scroll-to-latest affordance when the user reviews earlier output. Detail tailing continues while the child is queued, running, or waiting for input even when the current page reports `has_more=false`; terminal child status stops the tail, and transient tail errors stay in the detail dock with a longer retry interval. The detail tail and full child execution timeline are human UI state only; they are never injected into the parent model context by default. The overlay can render child user/assistant messages, tool calls, tool results, approvals, turn markers, compactions, errors, and generic custom rows from Floret `ReadSubAgentDetail` using the parent thread scope, without requiring a child threadstore row or child Flower metadata in Redeven. If the detail API currently has only summary data, Flower projects that summary into the same read-only message model rather than showing an empty transcript. Legacy transcript-local `subagent` blocks and legacy `snapshots`, `snapshots_by_id`, or free-form `subagents` collection payloads are not part of the live timeline contract.

Active cursor ownership belongs to Redeven's live projection. The projection marks at most one live assistant draft with `active_cursor` while the selected thread is running, and canceled, completed, idle, background, or merely streaming-shaped messages are not treated as active unless the projection marks them active. The visible model status affordance is a localized bottom dock lane driven by `model_io.updated`, not by `active_cursor` or thread running status. `model_io` is derived from Floret provider lifecycle and stream observations: provider request maps to waiting for model response; assistant, reasoning, and model tool-call stream observations map to thinking; provider retry maps to retrying; provider finish maps to finalizing. The indicator is not a timeline message and does not depend on a visible assistant block. The indicator's readable text is the base rendered text; shimmer is a decorative overlay and must not be the only mechanism that makes the label visible.

Current approval decisions belong in the composer. When Flower live/bootstrap state contains a pending, user-decidable primary approval action for the selected thread, the composer body replaces the text or secret input with an approval panel, and the footer primary controls become Reject and Approve. The panel shows product-safe presentation fields such as action label, purpose text, command preview, working directory, friendly effects, safe targets, and risk flags; it does not render raw tool arguments, argument hashes, internal run ids, or internal tool ids. If several approval actions are pending, the composer chooses the oldest primary action and shows a compact count for the remaining queue. Historical activity rows can still display approval activity for audit and navigation, but they are not the only actionable approval surface and are not allowed to create or fill the current pending ordinary-tool approval state. In live materialized state, a missing `approval_actions` field means the response did not sample approvals and must not clear an existing composer approval; an explicit empty object is the authoritative "no pending approvals" state. The surface derives selected-thread status in the order `waiting_user`, `waiting_approval`, active run `running`, then terminal/idle patch state, so a transient live bootstrap cannot flicker the composer between an approval card and the textarea.

Context usage and compaction state share the canonical projection model while using distinct presentation affordances. Redeven stores the latest `context_usage` plus historical `context_compactions` and `timeline_decorations` in Flower live materialized state. `context.usage.updated` drives the compact circular context indicator in the composer footer beside the primary send action. When a new active run starts before its current `context.usage.updated` event arrives, the composer may keep showing the previous persisted usage as a "last known context" indicator; once the active run's usage arrives, the same indicator switches back to current usage. This is a Flower presentation fallback over Redeven's projected state, not a Floret context lifecycle rule or a local UI token estimate. `context.compaction.updated` upserts a `context_compaction` timeline decoration keyed by `operation_id`; the UI renders it as a non-interactive divider such as compression in progress, compressed, or failed. A decoration carries a typed timeline `anchor` targeting a message, block, or activity item plus a before/after edge. Missing or invalid anchors are contract errors or skipped render entries, not permission to guess a fallback position. The divider is not a transcript message and does not affect message ordering, copying, or model-visible history. Because context telemetry is visible thread activity, persisted context usage and compaction events advance the canonical Flower read activity revision together with `last_context_run_id`, and broadcasting those context stream events also publishes a thread summary patch so selected live polling receives the current read status.

Flower `/compact` is a composer command, not a chat message. The surface parses the exact slash command before the stop/send matrix, calls the shared `compactThreadContext` adapter action, scrolls to the tail, and never submits `/compact` through `launchTurn`. Running threads compact in place: Redeven accepts the command, binds the decoration anchor at the time the manual operation is queued, forwards the manual request into the active Floret hosted-turn lifecycle, and keeps the run running after start, complete, or failed compaction events. Idle threads use an operation-only background path: the actor captures the anchor, broadcasts the running divider immediately, releases the actor, and then calls Floret `Host.CompactThread`; after Floret completes, Redeven persists only the returned opaque provider continuation when the product thread boundary still matches. Idle compaction never creates user or assistant transcript rows and never stores Redeven-owned provider-visible context state. The selected thread continues live polling while the in-memory live state has a `compacting` operation, even if the thread status is otherwise idle, so the same open Flower window observes the divider transition to compacted or failed. Cold bootstrap restores only terminal persisted compaction events (`compacted`, `failed`, or `cancelled`); an unfinished start/running event without a live worker is not replayed as a permanent "compression in progress" divider after restart. User turns sent while an idle compact operation is running enter the queued follow-up lane instead of blocking the actor or writing a premature transcript row; after the operation clears, queued drain starts the turn through the same canonical turn-start path. The next provider-visible context remains Floret-owned through Floret thread state and opaque provider continuation.

Accepted user turns are durable only when their canonical start state is complete. Redeven starts an idle or dequeued user turn in one threadstore transaction that writes the user transcript message, run record, conversation turn anchor, thread run state, structured input records, secret answers, upload bindings, and optional source follow-up consumption. If that transaction fails, no user transcript row remains. After it commits, live projection can broadcast the user message and assistant draft from the same canonical product timeline boundary.

Selected-thread live polling treats `model_io.updated` as a presentation boundary. Even when one live-events response contains a short provider stream followed by finalizing or clearing events, Flower commits the model status lane and yields a render frame at each model I/O boundary. This keeps brief but real provider streaming phases visible to the user without inventing timeline rows, fallback timers, or thread-status heuristics.

Thread read state is a user-scoped appserver projection over a canonical Flower activity snapshot persisted on `ai_threads`. Threadstore owns `flower_activity_revision`, `flower_activity_signature`, and `flower_activity_waiting_prompt_id`, initializes them for new and migrated threads, and advances them in the same transaction as visible message changes, run-state changes, waiting prompts, persisted context usage or compaction events, projected thread or message changes, stale active-run resets, and fork initialization. Appserver builds `read_status.snapshot` from `ThreadView.FlowerActivity` rather than deriving a second clock from run status or timestamps; unread is only `current.activity_revision > record.last_seen_activity_revision`.

Mark-read requests echo a snapshot previously delivered by the server. Future snapshots are rejected, current-revision signature, waiting-prompt, or last-message mismatches remain protocol errors, and stale snapshots are accepted without pretending they covered newer activity; the response still returns the current authoritative `read_status`. The shared Flower adapter returns that `read_status` directly instead of reloading the bootstrap. A selected running thread that completes while the user keeps it selected is therefore marked read from the final snapshot by the Flower surface, while background completions remain unread until selected.

# Boundaries

Flower does not sort messages, infer canonical insertion points from timestamps, or pair recent user messages with assistant drafts. It may project an assistant streaming row only from a `message.started` event and may upsert a canonical user row only from a `message.committed` payload. Local pending user rows are presentation-only composer feedback while a launch or queued follow-up request is in flight; they are removed or reclassified when the canonical thread snapshot confirms a committed message or queued turn, and they are never copied into transcript storage or model-visible history. The selected thread may show a bottom model status lane while a matching active run has `model_io`, but every durable rendered chat message must come from the current thread timeline.

Flower does not rebuild Floret execution order from Redeven transcript rows, audit tables, provider deltas, or tool-call records. `transcript_messages.message_json` can cache Flower product blocks for bootstrap and search, but for Floret-hosted execution those blocks must be derivable from Floret public ordered events and Redeven presentation projections. If the cache and Floret ordered events disagree, Floret ordered events are the execution transcript authority and the cache should be regenerated rather than patched with local ordering heuristics. Flower does not scan cached block JSON or activity rows to replace running tool states after stop; terminal tool states must arrive from Floret `ActivityTimeline` or `ThreadTurnProjection` projection. Redeven does not publish an independent activity live event or synthesize final activity rows from audit records.

Flower does not derive the current pending ordinary-tool approval from activity rows, transcript rows, audit records, or empty assistant block projections. Redeven projects ordinary pending approvals from Floret `ListPendingApprovals` only, and it maps that product-neutral snapshot to Flower `approval_actions` before bootstrap, `timeline.replaced`, and `approval.requested` delivery. Activity rows render the Floret `ActivityTimeline` item presentation that originated from Redeven `ToolPresentationSpec`; the UI does not look up a pending approval by run id/tool id to replace row labels or payload. Resolved activity can remove or mark old approval actions, but an activity timeline entry with `ApprovalState=requested` is audit history unless the Floret pending snapshot still contains the matching approval.

Flower does not infer read snapshots from run status, timestamps, message previews, context telemetry, or local activity tables. It applies `read_status` delivered by bootstrap/list/patch payloads and only persists read state for the currently selected thread. Selected-thread read sync updates local `read_status` from the `/read` response; if the response is still unread, Flower only queues another persistence attempt when the returned snapshot key differs and the thread remains selected.

Flower does not treat the global current model as an implicit existing-thread launch override. The current model is only a fallback when creating a new thread without a draft model. After a thread exists, its `model_id` and `reasoning_selection` are the displayed defaults, and model or default-reasoning changes must flow through thread patch actions that return canonical bootstrap state.

Flower does not parse provider request metadata, database rows, or transcript text to estimate context pressure. It renders context usage and compaction only from typed live/bootstrap fields projected by Redeven from structured Floret observations. The composer last-known context indicator can reuse a previous `context_usage` value, but it must label that freshness explicitly and must not synthesize a replacement ratio while waiting for the current run's Floret observation. Flower slash commands are local composer actions; `/compact` does not enter the transcript and does not cancel or replace a running agent turn.

Flower does not synthesize Sub Agents dropdown state from audit tables, provider-adapter side effects, child assistant text, Redeven child thread rows, or a separate transcript block type. The parent `ActivityTimeline` item payload, child thread id, and Flower `subagent_actions` display sidecar are the summary display contract, while full child execution detail belongs to the parent-scoped subagent detail API and is never injected into the parent model context by default. Subagent detail live tailing is a UI-only read path with request identity guards; it must not become a model-facing wait result, a second chat composer, or an alternate thread navigation target. Redeven does not create read-only child thread rows for Floret subagents, and old child rows are not a supported display source or compatibility path.

Redeven may publish `timeline.replaced` only as an explicit resync or snapshot recovery event. It is not a high-frequency companion for streaming text, block updates, committed messages, run status changes, or thread patches. Desktop and Env App adapters both refresh the live bootstrap after thread-level stop operations, so the surface receives the same canonical timeline after a stop-only or stop+send interaction.
