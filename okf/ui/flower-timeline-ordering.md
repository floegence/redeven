# Summary

Flower renders server-ordered Floret turn pages and validated live projections without client-side message sorting or reconstruction. Durable ordinal order, projection availability, draft identity, decorations, and replacement events remain explicit contracts. Live drafts may update only the matching canonical thread, turn, and run; resynchronization replaces the timeline rather than inventing assistant content or recovering from product audit rows.

# Contract

## Mechanism

The Flower live bootstrap includes `timeline_messages` as the only ordered message source for a thread snapshot. Redeven obtains before/after/tail pages from Floret `ListThreadTurns` and preserves their ascending canonical ordinal order. Each `ThreadTurnSnapshot` maps to the Floret `UserEntryID` user row followed by that same turn's validated assistant projection or one anchored `turn_projection_unavailable` decoration. Tool display details come from the projection `ActivityTimeline` plus Redeven's `ToolPresentationSpec`. A live draft may replace only the assistant slot whose `ThreadID`, `TurnID`, and `RunID` exactly match the canonical turn; terminal projection removes it. Redeven stores no transcript, correlation row, run record, projection cache, preview, or ordering copy.

Projection application is strict and monotonic. Floret `ThreadTurnProjection.Validate()` owns identity completeness, finite status and segment kinds, positive durable `ThroughOrdinal`, activity timeline validity, and control-signal structure. Redeven checks only that the validated projection belongs to the current Flower run/thread/turn, then maps assistant text, activity timelines with Redeven file actions, and non-body control segments. Live projections accept `running`; history assembly separately requires `Status.IsTerminal()`. Redeven keeps the last applied ordinal for each full execution identity, ignores equal or older projections, and advances the cursor only after the complete projection maps and replaces the assistant blocks. A valid zero-segment projection clears the in-memory assistant block list without synthesizing an empty markdown block. Runtime events are accepted only after Floret `runtime.Event.Validate()` validates their public event type and nested stream, activity, and projection data. Invalid contracts record `floret.contract.rejected` and leave the current presentation unchanged; private harness events never enter the public runtime sink.

Published Floret v0.19.0 includes the complete projection and structured attachments in each admitted public turn snapshot. `ReadThreadOverview` returns canonical thread state and the latest admitted turn from the same active path, excluding marker-only turns until their user entry is committed. Floret validators own availability combinations, identity, status, through ordinal, attachments, control signals, approvals, and settlement agreement. An invalid snapshot or projection is a canonical consistency error and requests resynchronization; Redeven does not turn it into display content. Only a valid projection with no renderable assistant blocks maps to one `turn_projection_unavailable` decoration with reason `not_renderable` immediately after that turn's user row. Flower renders “Response unavailable” with no retry control and no invented assistant body. Storage, I/O, pagination, unknown row cursors, and same-revision thread consistency failures remain request errors rather than being downgraded. Bootstrap, paged history, and `timeline.replaced` never recover content from metadata, product run errors, audit records, previews, or stale live drafts.

Timeline decorations are a strict discriminated union. `context_compaction` owns only a compaction payload, while `turn_projection_unavailable` owns only its projection-unavailable payload and finite reason. Unknown kinds, missing payloads, mixed payloads, invalid reason codes, or missing message anchors fail mapping explicitly. `ListThreadMessagesResponse`, live bootstrap, and full replacement snapshots carry `timeline_decorations` beside the messages produced by the same timeline projection, so a valid assistant projection and an unavailable decoration are mutually exclusive for one turn.

Live streaming uses server events only for current-process presentation. `message.started` may create one assistant streaming row and marks it as the single active cursor. There is no realtime transcript-message or transcript-reset event, and user messages are never inserted through the live stream. Floret text deltas may render temporary live markdown, while a validated `runtime.Event.Projection` with status `running` replaces only the matching live assistant draft with current assistant text and activity blocks. Aggregate Floret `runtime.Event.ActivityTimeline` remains lifecycle observation data and does not create main-thread activity blocks by itself. A local tool row is pending when Floret has observed the model's tool call but has not dispatched the handler; it becomes running only from Floret's dispatch-start lifecycle fact after permission and approval gates pass. Batched ordinary calls may start and finish in actual execution order, and approval display order never serializes handler execution. A projection that fails Floret validation or cannot be mapped is rejected as a unit and does not clear existing live content; a valid zero-segment projection clears the assistant blocks without inventing a body block. Stop and hard-cancel paths update current-process presentation but do not persist a canceled assistant row. Activity timeline changes enter the UI only inside Floret-derived projection activity segments delivered through `message.block_set`, a live draft, or a projection-backed timeline snapshot. Ordinary incremental events do not publish companion full timeline replacements. Every ordinary terminal turn and pending terminal settlement deletes the exact matching draft and publishes one `timeline.replaced` snapshot rebuilt from `ListThreadTurns`; an identity or ordinal mismatch requests resynchronization. `stream_generation` remains a Service-instance epoch so restart cannot let a stale high cursor hide new events.

Active cursor ownership belongs to Redeven's live projection. The projection marks at most one live assistant draft with `active_cursor` while the selected thread is running, and canceled, completed, idle, background, or merely streaming-shaped messages are not treated as active unless the projection marks them active. The visible model status affordance is a localized bottom dock lane driven by `model_io.updated`, not by `active_cursor` or thread running status. `model_io` is derived in memory from validated Floret provider lifecycle and stream observations: provider request maps to waiting for model response; assistant, reasoning, and model tool-call stream observations map to thinking; provider retry maps to retrying; provider finish maps to finalizing. Redeven does not persist Floret step, run, context-continuation, or compaction lifecycle mirrors. Provider diagnostics use typed finish fields; metadata cannot replace lifecycle facts.

Before Floret admission, Redeven may persist a pending command with a pre-generated opaque turn/run identity, prompt, product resource references, and launch options. No assistant draft is published at this stage. Floret admission creates the canonical user entry and turn; after `ListThreadTurns` confirms the exact `TurnID`, Redeven removes the pending prompt and retains only allowed product references. A crash before admission leaves the command pending, while a crash after admission is reconciled by exact Floret identity. Secret answers never enter plaintext persistence.

Selected-thread live polling treats `model_io.updated` as a presentation boundary. Even when one live-events response contains a short provider stream followed by finalizing or clearing events, Flower commits the model status lane and yields a render frame at each model I/O boundary. This keeps brief but real provider streaming phases visible to the user without inventing timeline rows, fallback timers, or thread-status heuristics.

# Boundaries

Flower does not sort messages, infer canonical insertion points from timestamps or IDs, or pair recent user messages with assistant drafts. `timeline.replaced` replaces the reducer message array in server order. A streaming assistant row requires exact canonical thread/turn/run identity; an unmatched draft triggers resynchronization and is never appended. Local pending user rows are composer feedback for commands not yet admitted by Floret and never become durable history by themselves.

Flower does not rebuild Floret execution order from Redeven database rows, audit data, provider deltas, previews, or tool-call records. Floret `ListThreadTurns` is the durable message and ordering source; live runtime projections and pending settlement projections may update only the matching canonical turn. Flower does not scan cached block JSON or activity rows to repair pending, waiting, or running tool states after stop or approval. Redeven does not publish independent activity blocks from aggregate observations, synthesize final rows, hide queued siblings, or turn an invalid projection into an empty assistant block.

Redeven may publish `timeline.replaced` only as an explicit resync, snapshot recovery, or terminal settlement finalization event. It is not a high-frequency companion for streaming text, block updates, committed messages, run status changes, or thread patches. Desktop and Env App adapters both refresh the live bootstrap after thread-level stop operations, so the surface receives the same canonical timeline after a stop-only or stop+send interaction.
