Workbench input ownership is explicit. Canvas zoom, widget activation, local scrolling, text selection, and action controls must be distinguishable by marker contracts rather than inferred from hover state or incidental DOM layout. The projected Workbench layer is anchored as a non-scrolling canvas invariant, while mounted terminal focus is a warm input handoff to the published floeterm core. Terminal click focus must not rely on terminal-surface scrollTop restoration or viewport navigation to hide canvas flicker.

# Mechanism

Wheel ownership uses exported local scroll viewport props that set `data-floe-canvas-wheel-interactive="true"` and a local-scroll role. Layout-only regions can declare a non-interactive wheel role. Text selection surfaces use separate marker props and do not receive wheel ownership unless combined with local scroll viewport props. Action surfaces declare their own marker so clickable controls can remain interactive inside widgets without becoming text selection surfaces. Terminal keyboard shortcuts such as panel-local search and session switching are scoped to the owning TerminalPanel event tree; they do not create a window-level shortcut owner and do not widen wheel, selection, copy, or widget activation ownership. Terminal session selection uses panel-local optimistic display state and a Floe Sidebar session list, so the selected-session visual reaches the screen through the active sidebar item state and primary rail without waiting for slider measurement, parent Workbench state propagation, terminal core mount, history replay, focus restoration, RPCs, disposal work, or Workbench viewport navigation. Mounted terminal sessions remain warm across ordinary sidebar switches: inactive live output is bounded, sequence-aware catchup work, and returning to an already mounted session must not trigger blocking reload, core disposal, attach, full history replay, or widget centering. Terminal sidebar items are directory-first: their visible title and circular initial derive from the session working directory, while their running badge reflects the session work state and takes priority over unread decoration. Terminal sidebar path actions are separate panel-local controls; activity mode opens the floating file browser and workbench mode requests a new Files canvas surface through the existing file-browser handoff. Terminal session creation, first activation of an unmounted session, and close interactions are UI-first inside the owning panel: sidebar selection or removal reaches the screen before terminal core mount, initial history replay, delete RPCs, or disposal work begins.

Projected Workbench surfaces must delegate pointer-anchored overlays to the shared surface floating layer. Menus opened from right-click, menu buttons, or keyboard anchors should pass client or anchor coordinates to `SurfaceFloatingLayer`, which owns surface-local projection, clamping, and z-index. The menu panel itself remains ordinary overlay content with role, focus, and item styling, and must not own `position: fixed` viewport coordinates inside a transformed projected surface.

# Boundaries

Selected widget boundaries guard canvas zoom. A surface that supports text selection does not automatically own wheel input, and a surface that owns local scrolling must declare the exported wheel contract props. Terminal shortcuts may switch only the current panel's visible sidebar sessions and must preserve platform primary-mod behavior so terminal-native Ctrl sequences are not stolen on macOS. Terminal sidebar path actions are file-browser handoffs only; they must not select the terminal session, bypass read permission checks, create a global file-browser command, or change wheel, text-selection, copy, shortcut, terminal transport, or widget activation ownership. Terminal UI-first scheduling is a responsiveness contract only; it must not pre-mount hidden terminal sessions, move terminal shortcuts to a global listener, restore the top tab strip or measured slider animation for terminal session selection, or change terminal transport semantics. Delayed terminal mounts are cancelled when the user switches away before paint, so rapidly skipped sessions do not attach or replay history merely because their sidebar item was briefly selected. Warm mounted-session catchup may use history pages only for sequence gaps or bounded-buffer overflow; it must keep shell-integration side effects idempotent and reset local sequence state after a terminal clear. Env App Workbench surface activation keeps `ensureVisible` and `centerViewport` separate: `ensureVisible` asks the Workbench page to reveal a target only when it is not already projected into the current viewport, while explicit `centerViewport: false` and anchored creation preserve the current viewport. Terminal input focus restoration is separate from Workbench viewport navigation, so focusing an already visible terminal widget must not start a viewport-centering animation or mutate the terminal surface scroll offset as a visual fallback. Native components embedded in projected Workbench surfaces must not hand-roll viewport-to-surface coordinate conversion for context menus; using the shared floating layer is the contract that keeps standalone and projected placements equivalent.
