# Summary

Redeven owns the product tool registry, presentation metadata, concrete handlers, current permission setting, and approval UX, while Floret owns the generic invocation lifecycle. Visible tools are rebuilt from the exact current thread setting before provider steps and local dispatch. Missing, unreadable, empty, invalid, or unauditable permission state fails closed; no previous snapshot or default permission can authorize execution. Ordinary calls in one model response may execute concurrently, while dependent work must wait for a later response.

# Contract

## Mechanism

The builtin registry includes readonly file/search/fetch helpers, standard mutation, patch, terminal, web search, OKF, todo, interaction, skill, and subagent tools. Flower's product permission is `permission_type`, with `readonly`, `approval_required`, and `full_access` as the only current values. Readonly-exclusive file/search/fetch helpers appear only on the `readonly` surface; the other modes expose the standard host tools. Redeven projects builtins into Floret definitions with effects, read-only/destructive/open-world flags, resource extractors, dynamic permission, and Activity projection. `ToolPresentationSpec` remains the single Redeven display policy, while Floret remains the lifecycle authority; Flower renders only allowlisted semantic detail and never falls back to raw tool-state field tables.

Ordinary tool scheduling follows the model's response boundaries. When one model response contains multiple ordinary tool calls, published Floret v0.8.0 starts the whole batch concurrently; each call independently validates arguments, extracts resources, checks permission, waits for approval when required, dispatches, projects output, and records audit facts. Result arrays and provider transcript entries retain the model's original call order, while dispatch and result observations retain actual start and completion order. A failed, rejected, or timed-out call does not stop an independent sibling. A dependent call must be emitted by the model in a later response after the prerequisite result is visible. Neither Floret nor Redeven infers dependencies or conflicts from tool names, arguments, effects, resources, filesystem paths, shell commands, permissions, or approval requirements, and the tool registry contains no scheduling-safety metadata. Main and subagent prompts state this response-boundary contract. `MaxToolCalls=200`, context cancellation, per-call timeouts, permission and approval gates, output limits, terminal active-process capacity, and other resource limits remain protections; they are not ordering policies. Ordinary calls in a mixed response close before Floret reevaluates control signals such as `ask_user` or `task_complete`.

Floret owns the generic permission, approval, invocation, and effect lifecycle. Redeven decides product policy from `ai_thread_settings.permission_type` and tool metadata, then exposes that decision through Floret `PermissionSpec` and an `EffectAuthorizationGate`. `currentThreadPermissionType` has no fallback: store absence, query failure, missing settings, empty value, or unknown value returns an error. Each hosted turn installs a dynamic surface provider that performs that read before every provider step and Floret dispatch, rebuilds tools/signals/prompt/host context, validates a non-empty snapshot, and persists the exact version-2 snapshot before using it. Initial snapshot persistence occurs only after Floret has validated the existing canonical thread authority. No stale in-memory surface, previous audit row, schema hash, or `approval_required` default is an authorization source.

Immediately before an irreversible local effect, Redeven's gate validates the complete Floret request and lease identity, loads the strict persisted v2 snapshot, binds root or parent-child product policy authority, and acquires a shared effect lease on the root thread lifecycle gate. Under that lease it rejects durable delete intent, rereads the current permission, rejects a stale admitted mode, waits for the exact approval when required, rereads permission after the wait, records audit evidence, and registers one temporary proof containing the exact effect-attempt, thread, turn, run, tool call, lease owner/generation, effect fingerprint, policy revision, approval, audit hash, and authorization time. The concrete invocation consumes that proof exactly once, constructs its exact run context, and completes the handler while the shared lease remains held. Permission changes, fork/delete intent, and other lifecycle mutations take the exclusive side of the same gate, so none can cross the final authorization-to-effect window. Parent and child effects can hold shared leases concurrently. `subagents close` registers an exact target-child join scope, while `close_all` and `wait` register a canonical descendant scope for child work required by the active handler. If a lifecycle writer queues, only those required child effects may join; unrelated root/child effects and direct PTY writes are fenced. Once the cohort reaches zero, the queued writer takes exclusive authority before new effects, avoiding both handler/child deadlock and writer starvation. Read-only `subagents list` and `inspect` release their lease after the final check and before their handler. `wait` remains gated because Floret may admit pending child input and start provider work while waiting; spawn, input, and close actions also remain gated. Failure before proof consumption calls neither provider nor handler and creates no authorization substitute.

Concurrent ordinary handlers never authorize from a shared mutable snapshot after refresh. Each effect request receives its own lifecycle-gated, invocation-bound proof and may run concurrently under the shared effect side of the gate. One concurrent tool therefore cannot overwrite or consume another tool's authorization evidence, and an exclusive permission downgrade or lifecycle intent cannot interleave between final authorization and the concrete effect.

Permission snapshot v2 is append-only security audit and pending-approval evidence. Its visible tools, approval decisions, registry/schema/presentation hashes, identity, and epoch explain what was authorized for one step, but current permission changes are always read from settings. Decoding rejects v1, unknown fields, multiple JSON values, missing snapshot id/hash, inconsistent content/hash, empty JSON, empty state, and invalid provisional/finalized timestamps; persistence never fills omitted values. Approval creation also requires the active snapshot and never rebuilds authorization from the current permission value, host context, or tool metadata. Malformed, conflicting, or unpersistable snapshots fail closed. In `approval_required`, shell and mutating host actions ask before running; in `full_access`, per-tool approval is skipped while schema validation, resource extraction, target routing, output limits, product audit, and Floret activity projection still apply. Terminal process operations additionally require effective write and execute permission.

Already-sent provider requests cannot be rewritten retroactively, but a saved permission change affects the next provider step and the next local dispatch. `subagents` control actions do not require approval, while each child tool invocation receives a surface derived from the latest parent setting and still flows through Floret permission/resource/approval lifecycle. Tool handlers execute only already-approved domain actions. Floret remains the sole persistent source for tool identity, arguments, lifecycle, result, error, completion output, and Activity; Redeven audit never becomes a tool-state or Flower display source.

Readonly-exclusive helpers are not general convenience aliases. `read_file`, `read_files`, `rgrep`, `find`, and `web_fetch` are visible only in `readonly`; `approval_required` and `full_access` use the standard host surface instead. `web_fetch` is a host-provided Redeven tool with SSRF protections, manual redirect validation, resolver and dial-time public IP checks, proxy suppression, body and MIME limits, and public-network-only semantics. DNS-derived VPN fake-IP answers in `198.18.0.0/15` are allowed for named hosts so common transparent-proxy/VPN setups can fetch public pages, but literal `198.18.0.0/15` URL hosts and other blocked local/private/reserved targets remain denied. It is not a Floret builtin and must not appear as a Floret core tool or a capability available to non-readonly permission types.

When a thread is configured for explicit target routing, the runtime forwards target-scoped builtin calls through `TargetToolExecutor`. The target executor receives a `TargetToolCall` containing `target_id`, `tool_name`, sanitized arguments, and required capabilities. The run layer returns a result payload that preserves or injects `target_id` and `execution_location`, so target-routed tool results cannot lose provenance before they reach the model or activity timeline.

# Boundaries

Tool names are not aliases for deleted knowledge-era tools. Current repository knowledge access uses `okf.index`, `okf.search`, and `okf.open`: index for broad directory discovery, search for short candidate lists, and open for detailed concept content. OKF is an embedded project corpus and does not access the internet; external, current, recent, news, third-party, market, pricing, and general web facts must use direct authoritative URLs or web search discovery instead.

Target provenance is part of the tool contract, not a UI hint. Flower must not infer remote execution from thread context alone; it can only claim remote or target execution when a tool result or Redeven product command returns explicit execution provenance.
