# Summary

Redeven owns PTY-backed terminal processes and bounded output retention, while Floret remains the canonical tool-lifecycle authority. The terminal tools expose one process handle across exec, read, write, and terminate; reads are incremental and termination waits for process-tree exit, output drain, and one owner-bound settlement. UI polling may observe process completion but cannot synthesize or retry canonical Activity settlement.

# Contract

## Mechanism

Terminal shell execution is a Redeven-owned process resource rather than a Floret-owned shell lifecycle. `terminal.exec` starts a PTY-backed process in the local AI runtime, immediately publishes a Floret `Invocation.UpdateActivity` presentation update with the public `process_id` read handle, waits only for the `yield_ms` initial wait, and returns either a completed result or a Floret pending tool result whose provider-visible handle is the same bare `process_id` used by `terminal.read`, `terminal.write`, and `terminal.terminate`. `yield_ms` is the only initial-wait argument and is not a hard timeout or automatic process termination request. Long-running commands continue outside the provider step. A reaped command succeeds only when its exit code is zero; every nonzero exit, including signal-style code 137, produces `status=error`, preserves the exact exit code, and settles the canonical Floret tool call as failed with a non-retryable error. Redeven stores output as monotonic chunks of at most 4,000 bytes and retains at most 1,000,000 bytes by evicting complete oldest chunks. `Snapshot` returns the retained cumulative view for `terminal.exec`, final settlement, write/terminate responses, and cumulative UI replacement. `ReadAfter` returns only chunks after the caller's last consumed sequence; `terminal.read` uses a fixed five-second wait and 8,000-byte page, while the Flower process-read API uses a fixed one-second wait and 256,000-byte page. A valid Flower process read returns the current snapshot even when the process has already reached `success`, `error`, or `canceled`, so the terminal status can stop UI polling without becoming a canonical Activity lifecycle fact. A future cursor is invalid, an expired cursor returns the currently retained delta with `truncated=true`, and an empty delta preserves `last_seq=after_seq` with `first_seq=0`. Incremental results contain the single text field `output` plus `first_seq`, `last_seq`, `latest_seq`, `has_more`, process status, command, byte count, timing, and completion facts. They do not expose `stdout`, `stderr`, `latest_output`, caller-selected wait/size arguments, or field aliases.

Every `terminal.read` call requires exactly `process_id`, `description`, and `after_seq`. The first read uses zero; every later read copies the preceding result's `last_seq` unchanged, and `has_more=true` means another page is already available. The tool contract explicitly tells the model that `output` is new output since `after_seq`, never a replayed full snapshot, and that an empty value means no new output arrived during the wait. The bounded model-authored `description` uses the current user language, names the command or task being checked, and naturally distinguishes later checks. Redeven projects that description as the call activity label, emits no result label, and lets Floret preserve the call label while the result contributes the real command and that call's delta payload. The description is not duplicated into activity description or payload. `terminal.read` declares Floret's generic polling repeat policy, excludes only `description` from repeat identity through `AnnotationRepeatIdentityIgnoredArguments`, and returns a progress metadata token derived from `process_id`, `last_seq`, `status`, and `ended_at_ms`; changing presentation text cannot bypass no-progress detection. Redeven passes each incremental result once through `tools.Result.Text` and its independent display activity through `tools.Result.Activity`; it does not query Floret storage or persist a second mapped terminal state.

Every `terminal.terminate` call requires exactly `process_id` and a 1-120 rune `description` in the current user language that clearly names the command or task being stopped. Redeven uses `description` as the call activity label, keeps `process_id` in the compact call payload and terminal detail identity, and emits no result label, so Floret preserves the descriptive title instead of replacing it with the tool name. Termination stops the Redeven PTY process, waits for process-tree reap and output drain, settles the original pending `terminal.exec` once, consumes the returned canonical Floret projection, removes the matching in-memory live draft when that projection is terminal, and returns so AppServer can write its product operation audit. Published Floret v0.23.0 requires the complete `PendingToolSettlementTarget`, including the exact `EffectAttemptID` that produced the pending result, returns the same target with the settlement result, and validates the canonical result through `Validate()`. Redeven receives that attempt identity only through the invocation's one-shot authorization proof and binds it to the process before starting the PTY. A `terminal.terminate` tool call made during the owning turn settles through the active turn or SubAgent Host. Cancellation or cleanup that finalizes after `RunTurn` returns must first observe the exact terminal authority barrier, then invokes the post-terminal recovery coordinator once with the execution-thread and root-or-parent authority identities. Redeven does not give active execution a recovery factory, accept a missing barrier, retry `ErrThreadBusy`, poll or guess authority, issue `pkill` or `kill` compensation, swallow settlement errors, query Floret storage, or synthesize Flower state from audit rows.

Terminal snapshots and incremental reads carry `execution_location=local_runtime`, while the public Flower activity payload excludes host-local path/input fields such as `cwd`, `workdir`, or `stdin` as well as Floret pending lifecycle fields. Redeven caps active processes, retained output, stdin/input size, initial waits, and maximum runtime. Before a pending terminal process starts, Redeven requires the exact Floret `PendingToolSettlementTarget` with a non-empty effect-attempt identity, an active settlement owner, a post-terminal recovery coordinator with exact authority identity, a non-nil exact `RunTurn` authority barrier, and a one-shot finalizer. Missing ownership or proof rejects process creation before manager state or PTY side effects; Redeven never guesses the Host from a run registry and never opens a maintenance Host for terminal settlement. Once PTY creation succeeds, failure to write initial stdin is post-dispatch: Redeven keeps and returns the registered process, terminates it, and exposes a non-retryable terminal error so an already executed command cannot be hidden and retried as if no effect occurred. The finalizer becomes eligible only after the pending handle has been exposed to Floret and the PTY has completed reap. The active tool path invokes the active owner once; the run-end path additionally requires successful barrier release before the coordinator creates and uses one recovery owner. `terminal.read` and `terminal.write` are pure PTY operations and cannot trigger, wait for, or retry settlement. Both tool-driven and direct UI writes nevertheless acquire the exact root-thread shared effect authority and recheck writable settings before writing bytes, so a persisted delete intent rejects input and an exclusive lifecycle mutation cannot cross a PTY write. `terminal.terminate`, run cancellation, hosted-turn cleanup, and service shutdown all use the same process termination primitive. Run-scoped cleanup enumerates only PTYs with the exact Redeven run identity, terminates every one, and waits for reap, output drain, and each process-bound one-shot settlement; another run's PTYs remain untouched. `Service.Close` completes finalizers before releasing Floret Hosts and stores. A ready settlement projection advances the shared durable `ThroughOrdinal` cursor and publishes canonical Flower state. An unavailable projection remains committed Floret state, emits diagnostic logging, and is not read back, retried, or locally synthesized. A settlement mutation failure is reported once by the finalizer and is not retried from read, write, cleanup, or later process access. Completion does not synthesize a duplicate provider-visible tool result, query Floret SQLite, persist a tool lifecycle copy, or automatically launch a follow-up model turn.

`terminal.exec` is the local AI runtime shell. Its schema describes local PTY process execution, with `yield_ms` as the only initial-wait argument. Terminal results include `execution_location=local_runtime` plus the resolved local working directory and process identity. Thread target context, Welcome environment context, or a target-shaped id does not change where this builtin runs. Remote or target diagnostics must use Redeven product commands whose own result payload carries target provenance.
