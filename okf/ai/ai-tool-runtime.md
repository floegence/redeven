Redeven's AI runtime treats tools as typed runtime capabilities. Builtins declare mutation and approval requirements, presentation metadata, grouping behavior, and renderer hints before execution dispatch reaches the runtime implementation. Floret owns the provider loop, turn execution, loop progress guards, provider-visible context lifecycle, tool dispatch lifecycle, permission/resource/approval lifecycle, streaming observation, core control-signal projection, opaque provider state lifecycle, activity observation, and engine metrics. Redeven owns product threads, persisted messages, concrete tool implementations, product policy decisions, approval UI, provider credentials, provider/profile selection, opaque provider-state persistence, and Flower timeline projection.

# Mechanism

The builtin registry includes readonly file/search/fetch helpers, standard file mutation, patch, terminal, web search, OKF index/search/open, todo, interaction, skill, and subagent tools. Flower's product permission is `permission_type`, with `readonly`, `approval_required`, and `full_access` as the only current permission values. `read_file`, `read_files`, `rgrep`, `find`, and `web_fetch` are readonly-exclusive tools for the `readonly` surface; standard shell and mutation tools are not visible there. `approval_required` and `full_access` expose the standard host tool surface instead of those readonly-exclusive helpers. `okf.index`, `okf.search`, and `okf.open` are read-only Redeven repository knowledge tools for directory discovery, short concept search, and detailed concept opening; web search is non-mutating product behavior but an open-world network effect in Floret's engine contract. `ToolPresentationSpec` is the single product display source for renderer, grouping, operation, label fields, fallback labels, compact call payload fields, result payload fields, and activity chip fields. Redeven projects builtins into Floret tool definitions with effects, read-only/destructive/open-world flags, resource extractors, static permission defaults, dynamic `PermissionFor`, and activity projection. Tool execution normalizes success summaries, errors, truncation, provenance, and builtin dispatch results so the chat activity timeline can render compact tool activity instead of raw low-level payloads; generic success summaries and event-code details remain model/audit material and are not copied into Flower expanded detail bodies by default. Flower activity detail rendering is Redeven product presentation over the Floret timeline: failures expand to a semantic `error` block that shows only the public failure reason, terminal rows expand to a read-only terminal output panel, file and patch rows render contents or diffs, web/search rows render result cards, question rows render prompts and choices, completion rows render outcome sections, todo rows render todo lists, subagent rows render delegated task summaries with message-entry actions, and generic structured rows show only allowlisted product detail fields. Flower does not fall back to raw `status`/`kind`/`tool`/`item` field tables when a renderer has no useful detail payload, and it filters non-actionable text such as `tool execution completed`, event codes, `success`, `completed`, or `ok`.

Floret owns the permission lifecycle. Redeven decides product policy from the current thread `permission_type` and tool metadata, then returns allow/ask/deny through the Floret `Approver` before tool dispatch. Each hosted turn installs a dynamic Floret tool surface provider: before each provider step it rebuilds the visible tools, hosted control signals, system prompt, host context, capability contract, and `PermissionSnapshot`; before local dispatch it refreshes the snapshot again so a stale model tool call cannot bypass the current thread policy. The snapshot supplies the prompt capability list, visible tool names, Floret registry names, per-tool approval decision, registry/schema/presentation hashes, and an epoch used in `tool_surface.updated` run events. In `approval_required`, shell and mutating host actions ask before running; in `full_access`, per-tool approval is skipped while schema validation, resource extraction, target routing for eligible tools, output limits, audit records, and Floret activity projection still apply. Command classification remains useful for audit and presentation, but it is not the source of permission decisions. Already-sent provider requests cannot be rewritten retroactively, but saved thread permission changes affect the next Floret provider request and the next local tool dispatch. `subagents` is a delegation-control tool visible in all three permission types, and its own `spawn`, `send_input`, `wait`, `list`, `inspect`, `close`, and `close_all` invocations do not require approval. That allow decision covers only the delegation control action itself; child runs receive a derived dynamic tool surface from the latest parent thread permission, and child tool invocations still flow through Floret permission/resource/approval lifecycle. A child approval-required action can delegate the ask back through the parent approver path; direct child user input remains disabled. Tool handlers execute already-approved domain actions and write audit/query records such as `ai_tool_calls`, run events, and execution spans; those records support diagnostics and tool-detail lookup, not Flower UI activity generation.

Current ordinary tool approvals come from Floret's product-neutral pending approval read model. Redeven calls `ListPendingApprovals` on the active Floret host, maps each generic pending approval to a Flower `FlowerApprovalAction` using `ToolPresentationSpec`, generic effects/resources/labels/host context, permission policy, and delegated approval metadata, and then publishes that product projection through Flower live state. Pending approval summaries derive their user-visible labels from the same Redeven tool presentation projection used for Floret invocation activity; Floret carries that sanitized activity presentation through approval lifecycle observations, so a requested approval is rendered as the same tool activity item with `requires_approval`, `approval_state=requested`, and `status=waiting`, not as a second approval row or a UI-layer pending approval backfill. `requires_approval` records that the invocation participated in the approval lifecycle; once Floret resolves the approval, approved pending/running/success tool rows keep that history without becoming a locked current approval row in Flower. Activity timeline rows remain audit and locator material; they can project resolved or unavailable approval history on the tool item, but they must not be used to discover or fill the current pending ordinary-tool approval. Submitting an ordinary approval rechecks the Floret pending snapshot before calling Redeven's approver channel, so UI decisions still flow through Floret's approver and dispatch gate. The task-complete confirmation surface is a Redeven `control_confirm` product confirmation for a Floret control signal, not a Floret tool approval, and it remains separate from ordinary tool pending approvals.

Readonly-exclusive helpers are not general convenience aliases. `read_file`, `read_files`, `rgrep`, `find`, and `web_fetch` are visible only in `readonly`; `approval_required` and `full_access` use the standard host surface instead. `web_fetch` is a host-provided Redeven tool with SSRF protections, manual redirect validation, resolver and dial-time public IP checks, proxy suppression, body and MIME limits, and public-network-only semantics. DNS-derived VPN fake-IP answers in `198.18.0.0/15` are allowed for named hosts so common transparent-proxy/VPN setups can fetch public pages, but literal `198.18.0.0/15` URL hosts and other blocked local/private/reserved targets remain denied. It is not a Floret builtin and must not appear as a Floret core tool or a capability available to non-readonly permission types.

Terminal shell execution is a Redeven-owned process resource rather than a Floret-owned shell lifecycle. `terminal.exec` starts a PTY-backed process in the local AI runtime, immediately publishes a Floret `Invocation.UpdateActivity` presentation update with the public `process_id` read handle, waits only for `yield_ms`, and returns either a completed result or a Floret pending tool result whose provider-visible handle is the same bare `process_id` used by `terminal.read`, `terminal.write`, and `terminal.terminate`. Long-running commands continue outside the provider step; `terminal.read` polls by sequence, `terminal.write` sends input to the PTY, and `terminal.terminate` cancels the process. `terminal.read` declares Floret's generic polling repeat policy and returns a progress metadata token derived from `process_id`, `last_seq`, `status`, and `ended_at_ms`, while its description tells the model to use the returned `last_seq` as the next `after_seq`. Process snapshots and terminal detail/read APIs carry status, command, cwd, output tail, sequence numbers, total bytes, duration, exit code when available, and `execution_location=local_runtime`, but the public Flower activity payload excludes host-local path/input fields such as `cwd`, `workdir`, and `stdin` as well as Floret pending lifecycle fields. Redeven caps active processes, output tails, read waits, stdin/input size, and maximum runtime, and `Service.Close` terminates active processes instead of attempting PID recovery. A terminal process distinguishes exposed pending state, process completion, and Floret settlement acknowledgement. When a pending process finishes, Redeven asks the active run's Floret host, or a provider-free Floret `ThreadMaintenanceHost` opened over the same published store contract, to `SettlePendingTool` and applies the returned Floret projection before writing the Redeven diagnostic `ai_tool_calls` audit record or run event. Parent run cleanup and subagent lifecycle reads also enumerate only Redeven's in-memory live terminal process manager for the matching run scope; terminal child subagents have their remaining pending processes canceled and settled through Floret public settlement APIs, while running or waiting children are left alone. Local process state is marked settlement-acknowledged only after the Floret settlement projection path succeeds; if settlement fails, later authorized lifecycle reads can retry the same public Floret settlement instead of treating audit tables as UI state. Completion does not synthesize a duplicate provider-visible tool result, does not query Floret SQLite schema directly, does not derive Flower activity from `ai_tool_calls` or run events, and does not automatically launch a follow-up model turn.

Provider adapters are model gateways, not Flower renderers. `floretProviderAdapter` maps provider stream bytes into Floret `ModelEvent` values. Assistant text, reasoning deltas, and model-generated tool-call stream observations are emitted as provider-neutral Floret model events before Redeven projects them into Flower live state. `ToolCallStream` identifies a tool call while the model is still generating it; final executable tool calls remain the separate `ToolCalls` batch. Redeven applies assistant text, reasoning deltas, tool-call stream observations, retry/done/abort markers, committed thread-entry observations, and source observations only from Floret runtime events in the event sink. Provider continuation persistence reads the complete opaque provider state from the Floret turn result, stores it as a state JSON envelope including `Attributes`, and only matches provider id, model, base URL, and state kind before passing it back to Floret.

Thread model and reasoning defaults are Redeven product state. `ai_threads` persists `model_id` and `reasoning_selection_json`; there is no separate model-lock state. Existing thread model changes validate the requested model against the allowed runtime or Desktop model source list, require an idle mutable thread, normalize the thread reasoning selection for the new model, and clear provider continuation because the opaque continuation is keyed by provider/model/base URL/state kind. Turn launch resolves the effective model in this order: explicit request model, thread `model_id`, Desktop model-source override/default, then configured `current_model_id`. An explicit request model affects only that turn. User turns and waiting-input responses resolve and store their effective model and reasoning before they start or enter the queued follow-up lane, so queued work is not affected by later thread default changes. Floret receives the resolved model and reasoning as hosted runtime input and owns execution, provider loop, and opaque state transport only; it does not own thread preferences, provider profiles, UI policy, or product model switching.

Context pressure and compaction presentation come from structured Floret runtime observations. Redeven maps Floret context status into `context.usage.updated` and maps Floret compaction lifecycle events into `context.compaction.updated`; those product events are persisted as run events and projected into Flower live materialized state. Floret compaction debug observations are persisted separately as `floret.context.compact.debug` context run events with safe correlation, phase, token-pressure, validation, duration, provider-state-kind, error fields, and `next_action`. These debug rows are diagnostic records and never create timeline decorations or chat rows. Redeven owns only product control entrypoints, thread serialization, sanitized live projection, read-state activity, and opaque provider-state persistence for context lifecycle. Manual `/compact` requests enter Redeven only as Flower UI/RPC control signals and are forwarded into Floret `Host` compaction lifecycle APIs. Active manual compaction keeps the current Floret turn running, and `CompactionEvent(complete)` never implies a Redeven run or thread terminal state. Idle manual compaction is a Flower operation rather than an assistant run: the actor publishes the running divider and returns, a background worker calls Floret `Host.CompactThread`, and user turns submitted before the operation clears are held as queued follow-ups without transcript rows. Redeven persists only the opaque provider-state envelope returned by Floret after validating that the product thread boundary has not changed. Gate events for registered, cancel requested, commit attempted, commit rejected, and commit persisted phases make late commits, thread cancellation, and queued-turn drain diagnosable without treating diagnostics as Flower timeline rows.

Redeven accepts a user turn only at one durable boundary. The service prepares the run object and user message payload before storage, then threadstore atomically writes the user transcript message, `ai_runs` row, `conversation_turns` anchor, thread run state, structured user inputs, secret answers, upload references, and optional source follow-up consumption. If storage fails, the reserved active run is released and no orphan transcript message is left behind. Conversation turns and structured user inputs remain product timeline and UI/audit records; they are not a Redeven-owned provider history source. Floret owns provider-visible history and context lifecycle through its thread store and opaque provider-state envelope.

Flower assistant execution text and tool order come from Floret `ThreadTurnProjection`. During a live run, Floret streaming observations can update temporary markdown, while `runtime.Event.Projection` replaces the in-memory Flower assistant draft with the current Floret display projection whenever a thread entry is committed. At turn completion, Redeven maps `TurnResult.Projection` segments into Flower markdown and activity blocks. Aggregate `runtime.Event.ActivityTimeline` remains lifecycle observation data and is not a main-thread block placement source. After reload or when a terminal projection must be reconstructed without an active provider host, Redeven opens the published provider-free Floret `ThreadMaintenanceHost` and calls `ReadTurnProjection` with explicit `ThreadID`, `TurnID`, and `RunID`. Floret live event projections, `TurnResult.Projection`, `ThreadMaintenanceHost.ReadTurnProjection`, and pending settlement projections are the authoritative raw-capable current-turn display projections; default preview detail reads are not a source for final assistant markdown. A committed projection is applied atomically: if any Floret activity segment fails the public timeline validator, Redeven records the diagnostic and rejects that projection instead of replacing an existing live activity row with an empty markdown block. A completed, failed, or canceled hosted turn must not use `TurnResult.Output`, task-complete signal text, provider deltas, preview detail rows, aggregate activity timelines, or audit rows as a fallback execution transcript; those values may drive lifecycle, control, telemetry, or thread-list preview state, but assistant markdown and activity block order in Flower responses must remain derived from Floret's public projection. Redeven no longer stores Floret-backed assistant projection JSON as `transcript_messages` shadow rows; `transcript_messages` remains Redeven-owned user/product message storage, while `conversation_turns` records only product message and Floret turn correlation. Thread `last_message_preview` is Redeven-owned thread-list summary state and may be updated from the final assistant projection text without becoming a transcript row. Flower UI tool activity comes from Floret `ThreadTurnProjection` activity segments whose `ActivityTimeline` presentation was produced from Redeven `ToolPresentationSpec` before crossing into Floret. The expanded detail body is a renderer-first product projection in `flowerActivityPresentation`: it consumes only the timeline item renderer, label, chips, public payload, and file/subagent sidecars, and it does not query `ai_tool_calls`, audit tables, threadstore rows, or Floret SQLite to fill fields. A running terminal detail may use the timeline payload's public `process_id` to call Redeven's terminal read API for live output; that read path is detail presentation only and never creates activity rows, transcript messages, or lifecycle facts. Floret's local `tool_call` activity means the model requested a queued tool invocation; only Floret's `tool_dispatch_started` lifecycle fact marks that same item running after permission and approval gates pass. Batched sibling calls therefore remain visible as pending instead of running while an earlier sibling waits for approval. Redeven can key detail lookups from timeline item ids, but it must not synthesize chat activity rows from `ai_tool_calls`, `tool.call`, `tool.result`, execution spans, handler outcomes, provider-adapter side effects, pending approval snapshots, aggregate live activity timelines, or stale live activity leftovers.

Flower subagents use the published Floret `Host` lifecycle rather than a Redeven-owned goroutine task manager. Main Flower runs, subagents, and compaction share the Floret-managed `stateDir/ai/floret_threads.sqlite` store, and Redeven accesses that store only through Floret public runtime APIs. Redeven caches one Floret subagent runtime per parent Flower thread so child controllers and the Floret host can outlive an individual parent turn, while restart recovery uses provider-free `ThreadMaintenanceHost` over the same published store contract without querying SQLite directly. Parent host initialization uses Floret `EnsureThread` instead of `ReadThread`. Parent Stop, CancelThread, and CancelRun are execution-lifecycle operations: after canceling the parent run they call Floret `CloseSubAgents` for the parent thread, which closes unfinished descendants, clears queued child input, and preserves completed history. DeleteThread is a data-lifecycle operation: Redeven first stops active execution, then calls Floret `DeleteThread` for the parent engine thread so Floret cascades deletion through descendants, prompt scopes, and artifacts; only after that does Redeven delete Flower product projections, uploads, and audit/detail records. `Service.Close` releases local host resources during process shutdown. The `subagents` builtin validates `spawn`, `send_input`, `wait`, `list`, `inspect`, `close`, and `close_all`, then forwards lifecycle operations to Floret `SpawnSubAgent`, `SendSubAgentInput`, `WaitSubAgents`, `ListSubAgents`, `CloseSubAgent`, and `CloseSubAgents`; parent activity refresh calls Floret `ListSubAgentActivityTimeline` only to update Flower-owned sidecars on existing Floret projection activity blocks, not to create independent main-thread activity blocks. `wait` defaults to five minutes and caps requested waits at twenty minutes, while child runs pass a twenty-minute Floret subagent run timeout. The child `ThreadID` is the canonical subagent identity and is also returned as `subagent_id` for model-facing compatibility. Redeven stores a distinct product `child_run_id` for child permission snapshots and delegated approval refs; Floret execution `RunID` remains a per-engine-execution identity and is not interpreted as the product child run id. Delegated approval refs reject missing child run ids, child-thread aliases, and parent-run aliases. `spawn` persists a provisional child permission snapshot keyed by the real Floret `subagents` tool call id, calls `SpawnSubAgent`, and finalizes the snapshot only after host spawn succeeds; those stored snapshots remain product audit and permission records rather than child transcript ownership. Active child hosts install their own dynamic tool surface provider, derive the child surface from the latest parent thread permission on the next provider step and next tool dispatch, preserve child thread id and product `child_run_id` in host context, and emit `subagent.tool_surface.updated` when the child epoch changes. Task names and paths are labels, not lifecycle targets, and the model cannot supply a child `thread_id` or raw `fork_mode`. The parent model chooses `context_mode` on `spawn`: `mission_only` maps to `SubAgentForkNone` and `full_history` maps to `SubAgentForkFullPath`; after spawn, wait/list/detail payloads and parent activity presentation derive `context_mode` only from Floret `ForkMode` instead of a Redeven memory map or duplicate persistence. The thread-scoped child host registers the parent thread's delegated tool surface for the current permission snapshot, removes parent-only controls, and disables hidden Floret control surfaces. If the parent run's model/tool/session configuration changes, the runtime rebuilds the child host only when no child is active; active children continue on the existing host generation while still refreshing permission-derived tool surfaces. Child tools use the Floret invocation `ThreadID` and `TurnID` to verify execution scope, while Redeven audit records and execution spans use the explicit product `child_run_id` from host context rather than deriving it from Floret execution `RunID`. Child threads run with a Flower child prompt that forbids direct user input, states the visible tool list and context mode, and requires a complete but focused final handoff report.

Delegated approval is thread-owned product state, not subagent detail state. Redeven stores delegated approval requests with parent thread, parent user, parent run/turn, child thread/run/turn, child tool call, approval id, ref hash, request fingerprint, delivery state, version, and surface epoch. User decisions submit through the parent thread with a delegated ref; the store compares owner, ref hash, action id, version, surface epoch, and idempotency key before resolving. Idempotent replay is accepted only when the key matches the same action, same ref hash, and same decision. Resolved delivery uses compare-and-swap; if delivery acknowledgement cannot be confirmed, Redeven marks the record as acknowledgement-unknown instead of emitting a false delivered state. The parent Flower waiting projection is the only approval surface; subagent detail is a read-only mirror or locator.

Subagent data is intentionally split into model-facing summaries, parent timeline presentation, routing sidecars, and human detail. `subagents(action="spawn")` requires a short `task_name`, a user-facing one-sentence `task_description`, an `agent_type`, and the full delegated `message`; the task description is the stable responsibility text shown in parent Flower UI, while live progress belongs in the child detail overlay. Model-facing `wait` results return bounded status `items` with child identity, task name, task description, agent type, status, timing, timeout facts, and `detail_ref`; they do not include child transcripts, tool calls, raw output, commands, stdout, stderr, `last_message`, `waiting_prompt`, or `result_digest`. Completed waits can include `final_handoff_report`, which carries each child’s final handoff text under a token budget without process detail. Timed-out waits can include `progress_summary`, which carries current status, waiting blockers when relevant, and suggested parent actions instead of execution logs. List and inspect remain bounded monitoring views with detail refs. Parent Flower timeline state comes from Floret `ListSubAgentActivityTimeline`; Redeven projects returned `ActivityTimeline` item payloads to a public subagent display payload before publishing them to Flower. That display payload keeps only user-facing fields: `action`, `status`, `task_name`, `title`, `task_description`, `agent_type`, `items`, item status/timing, `timed_out`, `truncated`, and public `error`; it filters child/thread ids, detail refs, context/debug diagnostics, handoff/progress internals, `last_message`, `waiting_prompt`, queued input counts, `can_*` control flags, and legacy `snapshot`/`subagent`/`item`/`subagents` collection fields. Redeven publishes the child id required to open detail only in the block-level `subagent_actions` sidecar keyed by activity item id; the sidecar may keep routing and timing fields such as child id, task name/title, task description, agent type, status, created/started/updated timestamps, and nested child items, but context mode and ids are never rendered as Flower tool-detail text. Flower activity rows render action-specific subagent titles, task/name previews, task descriptions, running elapsed time when available, exceptional status only when the child failed, canceled, timed out, or needs input, and an `Open messages` action that opens the existing parent-scoped subagent detail overlay without switching the selected thread. The parent thread header also exposes a Sub Agents dropdown derived from the same visible payload and `subagent_actions` sidecar, with active and settled counts; selecting a child opens the same parent-scoped detail overlay and keeps the selected Flower thread unchanged. The overlay reads `GET /api/ai/threads/{parent}/subagents/{child}/detail`, which validates the parent thread in Redeven, uses the active Floret host when present or a provider-free `ThreadMaintenanceHost` otherwise, calls Floret `ReadSubAgentDetail` with `IncludeRaw=false`, maps Floret child not-found or ownership mismatch through `errors.Is` against Floret public runtime not-found sentinels to 404, and renders Floret's top-level canonical child activity timeline together with user/assistant messages, tool results, approvals, turn markers, compactions, errors, and custom rows from the child execution timeline. Event-row activity snapshots are not a display source, so later tool results or pending-tool settlements replace earlier running updates in the same canonical activity block. Child detail does not require a child row or child Flower metadata in Redeven threadstore, and subagents do not appear as child threads in the sidebar. Subagent model results use canonical `items` only; unsupported detail, transcript, raw-output, single-object, or free-form collection fields are filtered before they reach model-facing results or Flower activity payloads.

Control signal projection is separate from Redeven waiting UI persistence. Floret signal projection emits engine signal facts for `ask_user` and `task_complete`; Flower `RequestUserInputPrompt` objects are created only by Redeven's waiting projection and persistence layer.

`terminal.exec` is the local AI runtime shell. Its schema describes local PTY process execution, and terminal results include `execution_location=local_runtime` plus the resolved local working directory and process identity. Thread target context, Welcome environment context, or a target-shaped id does not change where this builtin runs. Remote or target diagnostics must use Redeven product commands whose own result payload carries target provenance.

When a thread is configured for explicit target routing, the runtime forwards target-scoped builtin calls through `TargetToolExecutor`. The target executor receives a `TargetToolCall` containing `target_id`, `tool_name`, sanitized arguments, and required capabilities. The run layer returns a result payload that preserves or injects `target_id` and `execution_location`, so target-routed tool results cannot lose provenance before they reach the model or activity timeline.

Flower execution uses the published Floret runtime package through `runFloretHostedTurn` and `flruntime.Host`. Redeven passes the current user input, initial tool registry, dynamic tool surface provider, control signals, provider adapter, opaque previous provider state, and loop limits to Floret; it does not pass a preassembled provider-visible history. Floret `Metrics.Steps` is consumed as telemetry for result projection and activity accounting, not as a product configuration or request field. `ask_user` and `task_complete` use Floret core control definitions and provider-safe projection. Ordered assistant/tool display is derived from Floret `ThreadTurnProjection` segments, while the shape of Flower markdown blocks, activity timeline blocks, file actions, and tool presentation remains Redeven product projection.

# Boundaries

Tool names are not aliases for deleted knowledge-era tools. Current repository knowledge access uses `okf.index`, `okf.search`, and `okf.open`: index for broad directory discovery, search for short candidate lists, and open for detailed concept content. OKF is an embedded project corpus and does not access the internet; external, current, recent, news, third-party, market, pricing, and general web facts must use direct authoritative URLs or web search discovery instead.

Target provenance is part of the tool contract, not a UI hint. Flower must not infer remote execution from thread context alone; it can only claim remote or target execution when a tool result or Redeven product command returns explicit execution provenance.

Redeven must consume published Floret releases. Repository builds, tests, Desktop runs, and release validation must not depend on a sibling checkout, Go workspace, local replace directive, or package-manager local link. The runtime boundary depends on the published `github.com/floegence/floret` release that exposes Floret's `Host` turn and compaction facade, `ThreadTurnProjection`, live `runtime.Event.Projection`, transcript-free `EnsureThread`, dynamic tool surface provider hook, explicit execution `RunID` turn contracts, aggregate `runtime.Event.ActivityTimeline` lifecycle observations, `Invocation.UpdateActivity` / `tool_activity_updated` running presentation updates, committed thread-entry observations whose nonterminal turn markers do not settle tool activity, queued local tool calls that become running only at `tool_dispatch_started`, structured context observations, cancellation-aware manual compaction lifecycle events, cancellation-preserving turn-result projections after a host stops a run, manual compaction admission no-op observations for short or low-savings context, opaque provider state, provider request budget validation, multi-tool batch boundary preservation during compaction, continuation transcript delta preservation, Floret-owned activity timeline detail and subagent activity timeline projections, tool approval activity as a lifecycle state on the tool item until explicit resolution, structured tool-result status, parent-scoped subagent detail API, product-neutral pending approval snapshots, detail-only pending tool settlement, `CloseSubAgents`, durable `ForkMode`, and cascading engine thread deletion. Redeven must not import Floret `internal/` packages or depend on provider-visible transcript, summary, range-selection, checkpoint payload, engine policy data, or raw tool arguments for Flower approval presentation.
