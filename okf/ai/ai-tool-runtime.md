Redeven's AI runtime treats tools as typed runtime capabilities. Builtins declare mutation and approval requirements, presentation metadata, grouping behavior, and renderer hints before execution dispatch reaches the runtime implementation. Floret owns the provider loop, turn execution, loop progress guards, provider-visible context lifecycle, tool dispatch lifecycle, permission/resource/approval lifecycle, streaming observation, core control-signal projection, opaque provider state lifecycle, activity observation, and engine metrics. Redeven owns product threads, persisted messages, concrete tool implementations, product policy decisions, approval UI, provider credentials, provider/profile selection, opaque provider-state persistence, and Flower timeline projection.

# Mechanism

The builtin registry includes readonly file/search/fetch helpers, standard file mutation, patch, terminal, web search, OKF index/search/open, todo, interaction, skill, and subagent tools. Flower's product permission is `permission_type`, with `readonly`, `approval_required`, and `full_access` as the only current permission values. `read_file`, `read_files`, `rgrep`, `find`, and `web_fetch` are readonly-exclusive tools for the `readonly` surface; standard shell and mutation tools are not visible there. `approval_required` and `full_access` expose the standard host tool surface instead of those readonly-exclusive helpers. `okf.index`, `okf.search`, and `okf.open` are read-only Redeven repository knowledge tools for directory discovery, short concept search, and detailed concept opening; web search is non-mutating product behavior but an open-world network effect in Floret's engine contract. `ToolPresentationSpec` is the single product display source for renderer, grouping, operation, label fields, fallback labels, compact call payload fields, result payload fields, and activity chip fields. Redeven projects builtins into Floret tool definitions with effects, read-only/destructive/open-world flags, resource extractors, static permission defaults, dynamic `PermissionFor`, and activity projection. Tool execution normalizes success summaries, errors, truncation, provenance, and builtin dispatch results so the chat activity timeline can render compact tool activity instead of raw low-level payloads.

Floret owns the permission lifecycle. Redeven decides product policy from the frozen run permission type and tool metadata, then returns allow/ask/deny through the Floret `Approver` before tool dispatch. Each run freezes a `PermissionSnapshot` that supplies the prompt capability list, visible tool names, Floret registry names, per-tool approval decision, and registry/schema/presentation hashes. Runtime handler entrypoints also consult that snapshot, so a hidden or approval-required tool cannot be executed through direct local dispatch without the Floret authorization context. In `approval_required`, shell and mutating host actions ask before running; in `full_access`, per-tool approval is skipped while schema validation, resource extraction, target routing, timeouts, output limits, audit records, and Floret activity projection still apply. Command classification remains useful for audit and presentation, but it is not the source of permission decisions. `subagents` is a delegation-control tool visible in all three permission types, and its own `spawn`, `send_input`, `wait`, `list`, `inspect`, `close`, and `close_all` invocations do not require approval. That allow decision covers only the delegation control action itself; child runs receive a derived tool surface from the parent permission snapshot, and child tool invocations still flow through Floret permission/resource/approval lifecycle. A child approval-required action can delegate the ask back through the parent approver path; direct child user input remains disabled. Tool handlers execute already-approved domain actions and write audit/query records such as `ai_tool_calls`, run events, and execution spans; those records support diagnostics and tool-detail lookup, not Flower UI activity generation.

Readonly-exclusive helpers are not general convenience aliases. `read_file`, `read_files`, `rgrep`, `find`, and `web_fetch` are visible only in `readonly`; `approval_required` and `full_access` use the standard host surface instead. `web_fetch` is a host-provided Redeven tool with SSRF protections, manual redirect validation, resolver and dial-time public IP checks, proxy suppression, body and MIME limits, and public-network-only semantics. It is not a Floret builtin and must not appear as a Floret core tool or a capability available to non-readonly permission types.

Provider adapters are model gateways, not Flower renderers. `floretProviderAdapter` maps provider stream bytes into Floret `ModelEvent` values. Assistant text, reasoning deltas, and model-generated tool-call stream observations are emitted as provider-neutral Floret model events before Redeven projects them into Flower live state. `ToolCallStream` identifies a tool call while the model is still generating it; final executable tool calls remain the separate `ToolCalls` batch. Redeven applies assistant text, reasoning deltas, tool-call stream observations, retry/done/abort markers, and source observations only from Floret runtime events in the event sink. Provider continuation persistence reads the complete opaque provider state from the Floret turn result, stores it as a state JSON envelope including `Attributes`, and only matches provider id, model, base URL, and state kind before passing it back to Floret.

Context pressure and compaction presentation come from structured Floret runtime observations. Redeven maps Floret context status into `context.usage.updated` and maps Floret compaction lifecycle events into `context.compaction.updated`; those product events are persisted as run events and projected into Flower live materialized state. Floret compaction debug observations are persisted separately as `floret.context.compact.debug` context run events with safe correlation, phase, token-pressure, validation, duration, provider-state-kind, error fields, and `next_action`. These debug rows are diagnostic records and never create timeline decorations or chat rows. Redeven owns only product control entrypoints, thread serialization, sanitized live projection, read-state activity, and opaque provider-state persistence for context lifecycle. Manual `/compact` requests enter Redeven only as Flower UI/RPC control signals and are forwarded into Floret `Host` compaction lifecycle APIs. Active manual compaction keeps the current Floret turn running, and `CompactionEvent(complete)` never implies a Redeven run or thread terminal state. Idle manual compaction is a Flower operation rather than an assistant run: the actor publishes the running divider and returns, a background worker calls Floret `Host.CompactThread`, and user turns submitted before the operation clears are held as queued follow-ups without transcript rows. Redeven persists only the opaque provider-state envelope returned by Floret after validating that the product thread boundary has not changed. Gate events for registered, cancel requested, commit attempted, commit rejected, and commit persisted phases make late commits, thread cancellation, and queued-turn drain diagnosable without treating diagnostics as Flower timeline rows.

Redeven accepts a user turn only at one durable boundary. The service prepares the run object and user message payload before storage, then threadstore atomically writes the user transcript message, `ai_runs` row, `conversation_turns` anchor, thread run state, structured user inputs, secret answers, upload references, and optional source follow-up consumption. If storage fails, the reserved active run is released and no orphan transcript message is left behind. Conversation turns and structured user inputs remain product timeline and UI/audit records; they are not a Redeven-owned provider history source. Floret owns provider-visible history and context lifecycle through its thread store and opaque provider-state envelope.

Flower UI tool activity comes from Floret `ActivityTimeline` projection. Redeven can key detail lookups from timeline item ids, but it must not synthesize chat activity rows from `ai_tool_calls`, `tool.call`, `tool.result`, execution spans, handler outcomes, or provider-adapter side effects.

Flower subagents use the published Floret `Host` lifecycle rather than a Redeven-owned goroutine task manager. Redeven caches one Floret subagent runtime per parent Flower thread, so child controllers, the Floret host, and the private SQLite store outlive an individual parent turn. Parent run cancellation or natural turn completion cancels only the parent sampling/tool loop; explicit `subagents(close)` or `subagents(close_all)` is the normal lifecycle path that stops children without deleting their thread records, and parent thread deletion closes known children, releases the cached host, and deletes its read-only child projection threads. `Service.Close` releases local host resources during process shutdown. The `subagents` builtin validates `spawn`, `send_input`, `wait`, `list`, `inspect`, `close`, and `close_all`, then forwards to Floret `SpawnSubAgent`, `SendSubAgentInput`, `WaitSubAgents`, `ListSubAgents`, and `CloseSubAgent`. `wait` defaults to five minutes and caps requested waits at twenty minutes, while child runs pass a twenty-minute Floret subagent run timeout. The child `ThreadID` is the canonical subagent identity and is also returned as `subagent_id` for model-facing compatibility, but child execution uses a distinct child `RunID`; delegated approval refs reject missing child run ids, child-thread aliases, and parent-run aliases. `spawn` persists a provisional child permission snapshot keyed by the real Floret `subagents` tool call id, calls `SpawnSubAgent`, and finalizes the snapshot only after host spawn succeeds. Later `send_input`, child tool execution, and child approval restore read the finalized snapshot instead of sampling the current parent permission. Task names and paths are labels, not lifecycle targets, and the model cannot supply a child `thread_id` or raw `fork_mode`. The parent model chooses `context_mode` on `spawn`: `mission_only` maps to `SubAgentForkNone` and `full_history` maps to `SubAgentForkFullPath`. Redeven stores that mode in child projection metadata so later wait/detail reads use the child’s real context contract instead of a wait-call argument. The thread-scoped child host registers the parent thread's delegated tool surface for the current permission snapshot, removes parent-only controls, and disables hidden Floret control surfaces. If the parent run's model/tool/session configuration changes, the runtime rebuilds the child host only when no child is active; active children continue on the existing host generation. Child tools execute with the Floret invocation `RunID`, `ThreadID`, and `TurnID`, so tool-call audit records and execution spans belong to the child thread instead of the parent. Child threads run with a Flower child prompt that forbids direct user input, states the visible tool list and context mode, and requires a complete but focused final handoff report.

Delegated approval is thread-owned product state, not subagent detail state. Redeven stores delegated approval requests with parent thread, parent user, parent run/turn, child thread/run/turn, child tool call, approval id, ref hash, request fingerprint, delivery state, version, and surface epoch. User decisions submit through the parent thread with a delegated ref; the store compares owner, ref hash, action id, version, surface epoch, and idempotency key before resolving. Idempotent replay is accepted only when the key matches the same action, same ref hash, and same decision. Resolved delivery uses compare-and-swap; if delivery acknowledgement cannot be confirmed, Redeven marks the record as acknowledgement-unknown instead of emitting a false delivered state. The parent Flower waiting projection is the only approval surface; subagent detail is a read-only mirror or locator.

Subagent data is intentionally split into model-facing summaries, parent timeline presentation, and human detail. Model-facing `wait` results return bounded status `items` with child identity, status, context mode, timeout facts, and `detail_ref`; they do not include child transcripts, tool calls, raw output, commands, stdout, stderr, `last_message`, `waiting_prompt`, or `result_digest`. Completed waits include `final_handoff_report`, which carries each child’s final handoff text under a token budget without process detail. Timed-out waits include `progress_summary`, which carries current status, waiting blockers when relevant, and suggested parent actions instead of execution logs. List and inspect remain bounded monitoring views with detail refs. Parent Flower timeline state still comes from Floret `ActivityTimeline` payloads for `subagents`, and the shared UI projection accepts only the current `snapshot`, `subagent`, `item`, and `items` envelope shapes. The parent thread header exposes a Sub Agents dropdown with active and settled counts; selecting a child opens a parent-scoped detail overlay and keeps the selected Flower thread unchanged. The overlay reads `GET /api/ai/threads/{parent}/subagents/{child}/detail`, which calls Floret `ReadSubAgentDetail` with `IncludeRaw=false` and renders user/assistant messages, tool calls, tool results, approvals, turn markers, compactions, errors, and custom rows from the child execution timeline. If a child detail response has only summary data, the UI projects that summary into the same read-only message transcript so the overlay is not blank. Child projection threads still carry structured ownership metadata with `owner_kind=subagent_projection` and `parent_thread_id` for read-only internal projection and cleanup, but `threadstore.ListThreads` excludes them from normal thread-list/sidebar results. The old transcript `type:"subagent"` block chain and legacy `snapshots`, `snapshots_by_id`, or free-form `subagents` collection payloads are not supported display sources.

Control signal projection is separate from Redeven waiting UI persistence. Floret signal projection emits engine signal facts for `ask_user` and `task_complete`; Flower `RequestUserInputPrompt` objects are created only by Redeven's waiting projection and persistence layer.

`terminal.exec` is the local AI runtime shell. Its schema describes local execution, and successful terminal results include `execution_location=local_runtime` plus the resolved local working directory. Thread target context, Welcome environment context, or a target-shaped id does not change where this builtin runs.

When a thread is configured for explicit target routing, the runtime forwards target-scoped builtin calls through `TargetToolExecutor`. The target executor receives a `TargetToolCall` containing `target_id`, `tool_name`, sanitized arguments, and required capabilities. The run layer returns a result payload that preserves or injects `target_id` and `execution_location`, so target-routed tool results cannot lose provenance before they reach the model or activity timeline.

Flower execution uses the published Floret runtime package through `runFloretHostedTurn` and `flruntime.Host`. Redeven passes the current user input, permission-filtered tool registry, control signals, provider adapter, opaque previous provider state, and loop limits to Floret; it does not pass a preassembled provider-visible history. Floret `Metrics.Steps` is consumed as telemetry for result projection and activity accounting, not as a product configuration or request field. `ask_user` and `task_complete` use Floret core control definitions and provider-safe projection.

# Boundaries

Tool names are not aliases for deleted knowledge-era tools. Current repository knowledge access uses `okf.index`, `okf.search`, and `okf.open`: index for broad directory discovery, search for short candidate lists, and open for detailed concept content. OKF is an embedded project corpus and does not access the internet; external, current, recent, news, third-party, market, pricing, and general web facts must use direct authoritative URLs or web search discovery instead.

Target provenance is part of the tool contract, not a UI hint. Flower must not infer remote execution from thread context alone; it can only claim remote or target execution when a tool result or Redeven product command returns explicit execution provenance.

Redeven must consume published Floret releases. Repository builds, tests, Desktop runs, and release validation must not depend on a sibling checkout, Go workspace, local replace directive, or package-manager local link. The current runtime boundary depends on `github.com/floegence/floret v0.3.39`, including Floret's `Host` turn and compaction facade, structured context observations, cancellation-aware manual compaction lifecycle events, manual compaction admission no-op observations for short or low-savings context, opaque provider state, provider request budget validation, multi-tool batch boundary preservation during compaction, and parent-scoped subagent detail API. Redeven must not import Floret `internal/` packages or depend on provider-visible transcript, summary, range-selection, checkpoint payload, or engine policy data.
