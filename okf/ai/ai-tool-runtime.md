Redeven's AI runtime treats tools as typed runtime capabilities. Builtins declare mutation and approval requirements, presentation metadata, grouping behavior, and renderer hints before execution dispatch reaches the runtime implementation. Floret owns the provider loop, turn execution, loop progress guards, provider-visible context lifecycle, tool dispatch lifecycle, permission/resource/approval lifecycle, streaming observation, core control-signal projection, opaque provider state lifecycle, activity observation, and engine metrics. Redeven owns product threads, persisted messages, concrete tool implementations, product policy decisions, approval UI, provider credentials, provider/profile selection, opaque provider-state persistence, and Flower timeline projection.

# Mechanism

The builtin registry includes readonly file/search/fetch helpers, standard file mutation, patch, terminal, web search, OKF index/search/open, todo, interaction, skill, and subagent tools. Flower's product permission is `permission_type`, with `readonly`, `approval_required`, and `full_access` as the only current permission values. `read_file`, `read_files`, `rgrep`, `find`, and `web_fetch` are readonly-exclusive tools for the `readonly` surface; standard shell and mutation tools are not visible there. `approval_required` and `full_access` expose the standard host tool surface instead of those readonly-exclusive helpers. `okf.index`, `okf.search`, and `okf.open` are read-only Redeven repository knowledge tools for directory discovery, short concept search, and detailed concept opening; web search is non-mutating product behavior but an open-world network effect in Floret's engine contract. `ToolPresentationSpec` is the single product display source for renderer, grouping, operation, label fields, fallback labels, compact call payload fields, result payload fields, and activity chip fields. Redeven projects builtins into Floret tool definitions with effects, read-only/destructive/open-world flags, resource extractors, static permission defaults, dynamic `PermissionFor`, and activity projection. Tool execution normalizes success summaries, errors, truncation, provenance, and builtin dispatch results so the chat activity timeline can render compact tool activity instead of raw low-level payloads.

Floret owns the permission lifecycle. Redeven decides product policy from the current thread `permission_type` and tool metadata, then returns allow/ask/deny through the Floret `Approver` before tool dispatch. Each hosted turn installs a dynamic Floret tool surface provider: before each provider step it rebuilds the visible tools, hosted control signals, system prompt, host context, capability contract, and `PermissionSnapshot`; before local dispatch it refreshes the snapshot again so a stale model tool call cannot bypass the current thread policy. The snapshot supplies the prompt capability list, visible tool names, Floret registry names, per-tool approval decision, registry/schema/presentation hashes, and an epoch used in `tool_surface.updated` run events. In `approval_required`, shell and mutating host actions ask before running; in `full_access`, per-tool approval is skipped while schema validation, resource extraction, target routing, timeouts, output limits, audit records, and Floret activity projection still apply. Command classification remains useful for audit and presentation, but it is not the source of permission decisions. Already-sent provider requests cannot be rewritten retroactively, but saved thread permission changes affect the next Floret provider request and the next local tool dispatch. `subagents` is a delegation-control tool visible in all three permission types, and its own `spawn`, `send_input`, `wait`, `list`, `inspect`, `close`, and `close_all` invocations do not require approval. That allow decision covers only the delegation control action itself; child runs receive a derived dynamic tool surface from the latest parent thread permission, and child tool invocations still flow through Floret permission/resource/approval lifecycle. A child approval-required action can delegate the ask back through the parent approver path; direct child user input remains disabled. Tool handlers execute already-approved domain actions and write audit/query records such as `ai_tool_calls`, run events, and execution spans; those records support diagnostics and tool-detail lookup, not Flower UI activity generation.

Current ordinary tool approvals come from Floret's product-neutral pending approval read model. Redeven calls `ListPendingApprovals` on the active Floret host, maps each generic pending approval to a Flower `FlowerApprovalAction` using `ToolPresentationSpec`, generic effects/resources/labels/host context, permission policy, and delegated approval metadata, and then publishes that product projection through Flower live state. Pending approval summaries derive their user-visible labels from the same tool presentation projection as running activity rows; locator rows may use a matching pending approval projection for display fields when a timeline item only identifies the tool call. Activity timeline rows remain audit and locator material; they can project resolved or unavailable approval history, but they must not be used to discover the current pending ordinary-tool approval. Submitting an ordinary approval rechecks the Floret pending snapshot before calling Redeven's approver channel, so UI decisions still flow through Floret's approver and dispatch gate. The task-complete confirmation surface is a Redeven `control_confirm` product confirmation for a Floret control signal, not a Floret tool approval, and it remains separate from ordinary tool pending approvals.

Readonly-exclusive helpers are not general convenience aliases. `read_file`, `read_files`, `rgrep`, `find`, and `web_fetch` are visible only in `readonly`; `approval_required` and `full_access` use the standard host surface instead. `web_fetch` is a host-provided Redeven tool with SSRF protections, manual redirect validation, resolver and dial-time public IP checks, proxy suppression, body and MIME limits, and public-network-only semantics. DNS-derived VPN fake-IP answers in `198.18.0.0/15` are allowed for named hosts so common transparent-proxy/VPN setups can fetch public pages, but literal `198.18.0.0/15` URL hosts and other blocked local/private/reserved targets remain denied. It is not a Floret builtin and must not appear as a Floret core tool or a capability available to non-readonly permission types.

Provider adapters are model gateways, not Flower renderers. `floretProviderAdapter` maps provider stream bytes into Floret `ModelEvent` values. Assistant text, reasoning deltas, and model-generated tool-call stream observations are emitted as provider-neutral Floret model events before Redeven projects them into Flower live state. `ToolCallStream` identifies a tool call while the model is still generating it; final executable tool calls remain the separate `ToolCalls` batch. Redeven applies assistant text, reasoning deltas, tool-call stream observations, retry/done/abort markers, committed thread-entry observations, and source observations only from Floret runtime events in the event sink. Provider continuation persistence reads the complete opaque provider state from the Floret turn result, stores it as a state JSON envelope including `Attributes`, and only matches provider id, model, base URL, and state kind before passing it back to Floret.

Thread model and reasoning defaults are Redeven product state. `ai_threads` persists `model_id` and `reasoning_selection_json`; there is no separate model-lock state. Existing thread model changes validate the requested model against the allowed runtime or Desktop model source list, require an idle mutable thread, normalize the thread reasoning selection for the new model, and clear provider continuation because the opaque continuation is keyed by provider/model/base URL/state kind. Turn launch resolves the effective model in this order: explicit request model, thread `model_id`, Desktop model-source override/default, then configured `current_model_id`. An explicit request model affects only that turn. User turns and waiting-input responses resolve and store their effective model and reasoning before they start or enter the queued follow-up lane, so queued work is not affected by later thread default changes. Floret receives the resolved model and reasoning as hosted runtime input and owns execution, provider loop, and opaque state transport only; it does not own thread preferences, provider profiles, UI policy, or product model switching.

Context pressure and compaction presentation come from structured Floret runtime observations. Redeven maps Floret context status into `context.usage.updated` and maps Floret compaction lifecycle events into `context.compaction.updated`; those product events are persisted as run events and projected into Flower live materialized state. Floret compaction debug observations are persisted separately as `floret.context.compact.debug` context run events with safe correlation, phase, token-pressure, validation, duration, provider-state-kind, error fields, and `next_action`. These debug rows are diagnostic records and never create timeline decorations or chat rows. Redeven owns only product control entrypoints, thread serialization, sanitized live projection, read-state activity, and opaque provider-state persistence for context lifecycle. Manual `/compact` requests enter Redeven only as Flower UI/RPC control signals and are forwarded into Floret `Host` compaction lifecycle APIs. Active manual compaction keeps the current Floret turn running, and `CompactionEvent(complete)` never implies a Redeven run or thread terminal state. Idle manual compaction is a Flower operation rather than an assistant run: the actor publishes the running divider and returns, a background worker calls Floret `Host.CompactThread`, and user turns submitted before the operation clears are held as queued follow-ups without transcript rows. Redeven persists only the opaque provider-state envelope returned by Floret after validating that the product thread boundary has not changed. Gate events for registered, cancel requested, commit attempted, commit rejected, and commit persisted phases make late commits, thread cancellation, and queued-turn drain diagnosable without treating diagnostics as Flower timeline rows.

Redeven accepts a user turn only at one durable boundary. The service prepares the run object and user message payload before storage, then threadstore atomically writes the user transcript message, `ai_runs` row, `conversation_turns` anchor, thread run state, structured user inputs, secret answers, upload references, and optional source follow-up consumption. If storage fails, the reserved active run is released and no orphan transcript message is left behind. Conversation turns and structured user inputs remain product timeline and UI/audit records; they are not a Redeven-owned provider history source. Floret owns provider-visible history and context lifecycle through its thread store and opaque provider-state envelope.

Flower assistant execution text and tool order come from Floret `ThreadTurnProjection`. During a live run, `runtime.Event.Committed` lets Redeven reconcile the assistant draft by passing committed Floret thread detail facts back through Floret's projection API; at turn completion, Redeven maps `TurnResult.Projection` segments into Flower markdown and activity blocks. A completed hosted turn must not use `TurnResult.Output`, task-complete signal text, provider deltas, or audit rows as a fallback execution transcript; those values may drive lifecycle, control, or telemetry projection, but assistant markdown in the durable Flower cache must remain derived from Floret's public projection. `transcript_messages.message_json` remains a Redeven product cache and index that can be rebuilt from Floret projection plus Redeven display metadata; it is not the execution transcript source of truth. Flower UI tool activity comes from Floret `ActivityTimeline` projection segments. Redeven can key detail lookups from timeline item ids, but it must not synthesize chat activity rows from `ai_tool_calls`, `tool.call`, `tool.result`, execution spans, handler outcomes, or provider-adapter side effects.

Flower subagents use the published Floret `Host` lifecycle rather than a Redeven-owned goroutine task manager. Main Flower runs, subagents, and compaction share the Floret-managed `stateDir/ai/floret_threads.sqlite` store; Redeven no longer opens or creates a separate `floret_subagents.sqlite` store, and the legacy file is not migrated, read, or automatically deleted. Redeven caches one Floret subagent runtime per parent Flower thread so child controllers and the Floret host can outlive an individual parent turn, while restart recovery can reopen a minimal Floret lifecycle host over the same shared store without querying SQLite directly. Parent host initialization uses Floret `EnsureThread` instead of `ReadThread`. Parent Stop, CancelThread, and CancelRun are execution-lifecycle operations: after canceling the parent run they call Floret `CloseSubAgents` for the parent thread, which closes unfinished descendants, clears queued child input, and preserves completed history. DeleteThread is a data-lifecycle operation: Redeven first stops active execution, then calls Floret `DeleteThread` for the parent engine thread so Floret cascades deletion through descendants, prompt scopes, and artifacts; only after that does Redeven delete Flower product projections, uploads, and audit/detail records. `Service.Close` releases local host resources during process shutdown. The `subagents` builtin validates `spawn`, `send_input`, `wait`, `list`, `inspect`, `close`, and `close_all`, then forwards lifecycle operations to Floret `SpawnSubAgent`, `SendSubAgentInput`, `WaitSubAgents`, `ListSubAgents`, `CloseSubAgent`, and `CloseSubAgents`; parent activity refresh calls Floret `ListSubAgentActivityTimeline`. `wait` defaults to five minutes and caps requested waits at twenty minutes, while child runs pass a twenty-minute Floret subagent run timeout. The child `ThreadID` is the canonical subagent identity and is also returned as `subagent_id` for model-facing compatibility. Redeven stores a distinct product `child_run_id` for child permission snapshots and delegated approval refs; Floret execution `RunID` remains a per-engine-execution identity and is not interpreted as the product child run id. Delegated approval refs reject missing child run ids, child-thread aliases, and parent-run aliases. `spawn` persists a provisional child permission snapshot keyed by the real Floret `subagents` tool call id, calls `SpawnSubAgent`, and finalizes the snapshot only after host spawn succeeds; those stored snapshots remain product audit and permission records rather than child transcript ownership. Active child hosts install their own dynamic tool surface provider, derive the child surface from the latest parent thread permission on the next provider step and next tool dispatch, preserve child thread id and product `child_run_id` in host context, and emit `subagent.tool_surface.updated` when the child epoch changes. Task names and paths are labels, not lifecycle targets, and the model cannot supply a child `thread_id` or raw `fork_mode`. The parent model chooses `context_mode` on `spawn`: `mission_only` maps to `SubAgentForkNone` and `full_history` maps to `SubAgentForkFullPath`; after spawn, wait/list/detail payloads and parent activity presentation derive `context_mode` only from Floret `ForkMode` instead of a Redeven memory map or duplicate persistence. The thread-scoped child host registers the parent thread's delegated tool surface for the current permission snapshot, removes parent-only controls, and disables hidden Floret control surfaces. If the parent run's model/tool/session configuration changes, the runtime rebuilds the child host only when no child is active; active children continue on the existing host generation while still refreshing permission-derived tool surfaces. Child tools use the Floret invocation `ThreadID` and `TurnID` to verify execution scope, while Redeven audit records and execution spans use the explicit product `child_run_id` from host context rather than deriving it from Floret execution `RunID`. Child threads run with a Flower child prompt that forbids direct user input, states the visible tool list and context mode, and requires a complete but focused final handoff report.

Delegated approval is thread-owned product state, not subagent detail state. Redeven stores delegated approval requests with parent thread, parent user, parent run/turn, child thread/run/turn, child tool call, approval id, ref hash, request fingerprint, delivery state, version, and surface epoch. User decisions submit through the parent thread with a delegated ref; the store compares owner, ref hash, action id, version, surface epoch, and idempotency key before resolving. Idempotent replay is accepted only when the key matches the same action, same ref hash, and same decision. Resolved delivery uses compare-and-swap; if delivery acknowledgement cannot be confirmed, Redeven marks the record as acknowledgement-unknown instead of emitting a false delivered state. The parent Flower waiting projection is the only approval surface; subagent detail is a read-only mirror or locator.

Subagent data is intentionally split into model-facing summaries, parent timeline presentation, and human detail. Model-facing `wait` results return bounded status `items` with child identity, status, context mode, timeout facts, and `detail_ref`; they do not include child transcripts, tool calls, raw output, commands, stdout, stderr, `last_message`, `waiting_prompt`, or `result_digest`. Completed waits include `final_handoff_report`, which carries each child’s final handoff text under a token budget without process detail. Timed-out waits include `progress_summary`, which carries current status, waiting blockers when relevant, and suggested parent actions instead of execution logs. List and inspect remain bounded monitoring views with detail refs. Parent Flower timeline state comes from Floret `ListSubAgentActivityTimeline`; Redeven keeps returned `ActivityTimeline` item payloads as Floret engine payloads and publishes Flower display fields such as `operation=subagents`, `action=inspect`, `delegation_runtime=floret`, normalized `agent_type`, and context mode in the block-level `subagent_actions` sidecar keyed by activity item id. The parent thread header exposes a Sub Agents dropdown with active and settled counts; selecting a child opens a parent-scoped detail overlay and keeps the selected Flower thread unchanged. The overlay reads `GET /api/ai/threads/{parent}/subagents/{child}/detail`, which validates the parent thread in Redeven, calls Floret `ReadSubAgentDetail` with `IncludeRaw=false`, maps Floret child not-found or ownership mismatch through `errors.Is` against Floret public runtime not-found sentinels to 404, and renders user/assistant messages, tool calls, tool results, approvals, turn markers, compactions, errors, custom rows, and Floret-provided row activity timelines from the child execution timeline. Child detail does not require a child row or child Flower metadata in Redeven threadstore, and subagents do not appear as child threads in the sidebar. Threadstore does not perform semantic cleanup or list filtering for old child rows because those rows are not a supported display source. The old transcript `type:"subagent"` block chain and legacy `snapshots`, `snapshots_by_id`, or free-form `subagents` collection payloads are not supported display sources.

Control signal projection is separate from Redeven waiting UI persistence. Floret signal projection emits engine signal facts for `ask_user` and `task_complete`; Flower `RequestUserInputPrompt` objects are created only by Redeven's waiting projection and persistence layer.

`terminal.exec` is the local AI runtime shell. Its schema describes local execution, and successful terminal results include `execution_location=local_runtime` plus the resolved local working directory. Thread target context, Welcome environment context, or a target-shaped id does not change where this builtin runs.

When a thread is configured for explicit target routing, the runtime forwards target-scoped builtin calls through `TargetToolExecutor`. The target executor receives a `TargetToolCall` containing `target_id`, `tool_name`, sanitized arguments, and required capabilities. The run layer returns a result payload that preserves or injects `target_id` and `execution_location`, so target-routed tool results cannot lose provenance before they reach the model or activity timeline.

Flower execution uses the published Floret runtime package through `runFloretHostedTurn` and `flruntime.Host`. Redeven passes the current user input, initial tool registry, dynamic tool surface provider, control signals, provider adapter, opaque previous provider state, and loop limits to Floret; it does not pass a preassembled provider-visible history. Floret `Metrics.Steps` is consumed as telemetry for result projection and activity accounting, not as a product configuration or request field. `ask_user` and `task_complete` use Floret core control definitions and provider-safe projection. Ordered assistant/tool display is derived from Floret `ThreadTurnProjection` segments, while the shape of Flower markdown blocks, activity timeline blocks, file actions, and tool presentation remains Redeven product projection.

# Boundaries

Tool names are not aliases for deleted knowledge-era tools. Current repository knowledge access uses `okf.index`, `okf.search`, and `okf.open`: index for broad directory discovery, search for short candidate lists, and open for detailed concept content. OKF is an embedded project corpus and does not access the internet; external, current, recent, news, third-party, market, pricing, and general web facts must use direct authoritative URLs or web search discovery instead.

Target provenance is part of the tool contract, not a UI hint. Flower must not infer remote execution from thread context alone; it can only claim remote or target execution when a tool result or Redeven product command returns explicit execution provenance.

Redeven must consume published Floret releases. Repository builds, tests, Desktop runs, and release validation must not depend on a sibling checkout, Go workspace, local replace directive, or package-manager local link. The runtime boundary depends on the published `github.com/floegence/floret` release that exposes Floret's `Host` turn and compaction facade, `ThreadTurnProjection`, transcript-free `EnsureThread`, dynamic tool surface provider hook, explicit execution `RunID` turn contracts, committed thread-entry observations, structured context observations, cancellation-aware manual compaction lifecycle events, manual compaction admission no-op observations for short or low-savings context, opaque provider state, provider request budget validation, multi-tool batch boundary preservation during compaction, continuation transcript delta preservation, Floret-owned activity timeline detail and subagent activity timeline projections, structured tool-result status, parent-scoped subagent detail API, product-neutral pending approval snapshots, `CloseSubAgents`, durable `ForkMode`, and cascading engine thread deletion. Redeven must not import Floret `internal/` packages or depend on provider-visible transcript, summary, range-selection, checkpoint payload, engine policy data, or raw tool arguments for Flower approval presentation.
