Redeven's AI runtime treats tools as typed runtime capabilities. Builtins declare mutation and approval requirements, presentation metadata, grouping behavior, and renderer hints before execution dispatch reaches the runtime implementation. Floret owns the provider loop, turn execution, loop progress guards, provider-visible context lifecycle, tool dispatch lifecycle, permission/resource/approval lifecycle, streaming observation, core control-signal projection, opaque provider state lifecycle, activity observation, and engine metrics. Redeven owns product threads, persisted messages, concrete tool implementations, product policy decisions, approval UI, provider credentials, provider/profile selection, opaque provider-state persistence, and Flower timeline projection.

# Mechanism

The builtin registry includes readonly file/search/fetch helpers, standard file mutation, patch, terminal, web search, OKF index/search/open, todo, interaction, skill, and subagent tools. Flower's product permission is `permission_type`, with `readonly`, `approval_required`, and `full_access` as the only current permission values. `read_file`, `read_files`, `rgrep`, `find`, and `web_fetch` are readonly-exclusive tools for the `readonly` surface; standard shell and mutation tools are not visible there. `approval_required` and `full_access` expose the standard host tool surface instead of those readonly-exclusive helpers. `okf.index`, `okf.search`, and `okf.open` are read-only Redeven repository knowledge tools for directory discovery, short concept search, and detailed concept opening; web search is non-mutating product behavior but an open-world network effect in Floret's engine contract. `ToolPresentationSpec` is the single product display source for renderer, grouping, operation, label fields, fallback labels, compact call payload fields, result payload fields, and activity chip fields. Redeven projects builtins into Floret tool definitions with effects, read-only/destructive/open-world flags, resource extractors, static permission defaults, dynamic `PermissionFor`, and activity projection. Tool execution normalizes success summaries, errors, truncation, provenance, and builtin dispatch results so the chat activity timeline can render compact tool activity instead of raw low-level payloads; generic success summaries and event-code details remain model/audit material and are not copied into Flower expanded detail bodies by default. Flower activity detail rendering is Redeven product presentation over the Floret timeline: failures expand to a semantic `error` block that shows only the public failure reason, terminal rows expand to a read-only terminal output panel whose command detail includes the public `process_id` when present, file and patch rows render contents or diffs, web/search rows render result cards, question rows render prompts and choices, completion rows render outcome sections, todo rows render todo lists, subagent rows render delegated task summaries with message-entry actions, and generic structured rows show only allowlisted product detail fields. Flower does not fall back to raw `status`/`kind`/`tool`/`item` field tables when a renderer has no useful detail payload, and it filters non-actionable text such as `tool execution completed`, event codes, `success`, `completed`, or `ok`.

Flower configuration has three independent states. The default `permission_type` is valid without any model configuration and initializes only future threads. An environment `AIModelProfile` exists only when both the provider registry and `current_model_id` are present and the selected model belongs to that registry; either field without the other is invalid. A Desktop model source is a separate runtime catalog and may coexist with that environment profile in one Flower settings snapshot. Env App exposes it only when the required Desktop session route is exactly `remote_desktop`; `local_host` and ordinary browser sessions do not expose Desktop diagnostics. Its status is a strict discriminated union: `ready` owns a non-empty validated catalog, `missing_keys` owns provider ids, `empty` represents a successfully loaded empty catalog, `connecting`, `unbound`, `expired`, and `unsupported` are explicit binding states, and `error` may carry one diagnostic. A missing Runtime declaration maps to `unsupported`; catalog transport or validation failures map to `error` and are never swallowed. Catalog entries enter `ready` only when `source` is exactly `desktop_model_source` and the id matches the opaque `desktop:model_<64 lowercase hex>` contract. The environment profile remains the persisted default and wins initial new-chat selection whenever it exists. Selecting a Desktop model affects only the current mounted Flower surface's new-chat draft or the selected thread and never calls the environment default-model persistence path, so remounting Flower restores the environment default. `AIConfig.HasModelProfile()` is the canonical environment-profile predicate, and the existing `ai_runtime.remote_configured` wire field reports that predicate rather than the presence of an `AIConfig` object. The focused default-permission update merges under the AI service state lock, while provider-bundle updates replace only the model profile under the same lock, so concurrent writes preserve permission and recovery settings. Generic settings updates do not accept `ai`.

Ordinary tool scheduling follows the model's response boundaries. When one model response contains multiple ordinary tool calls, published Floret v0.8.0 starts the whole batch concurrently; each call independently validates arguments, extracts resources, checks permission, waits for approval when required, dispatches, projects output, and records audit facts. Result arrays and provider transcript entries retain the model's original call order, while dispatch and result observations retain actual start and completion order. A failed, rejected, or timed-out call does not stop an independent sibling. A dependent call must be emitted by the model in a later response after the prerequisite result is visible. Neither Floret nor Redeven infers dependencies or conflicts from tool names, arguments, effects, resources, filesystem paths, shell commands, permissions, or approval requirements, and the tool registry contains no scheduling-safety metadata. Main and subagent prompts state this response-boundary contract. `MaxToolCalls=200`, context cancellation, per-call timeouts, permission and approval gates, output limits, terminal active-process capacity, and other resource limits remain protections; they are not ordering policies. Ordinary calls in a mixed response close before Floret reevaluates control signals such as `ask_user` or `task_complete`.

Floret owns the permission lifecycle. Redeven decides product policy from the current thread `permission_type` and tool metadata, then returns allow/ask/deny through the Floret `Approver` before tool dispatch. Each hosted turn installs a dynamic Floret tool surface provider: before each provider step it rebuilds the visible tools, hosted control signals, system prompt, host context, capability contract, and `PermissionSnapshot`; before local dispatch it refreshes the snapshot again so a stale model tool call cannot bypass the current thread policy. The snapshot supplies the prompt capability list, visible tool names, Floret registry names, per-tool approval decision, registry/schema/presentation hashes, and an epoch used in `tool_surface.updated` run events. In `approval_required`, shell and mutating host actions ask before running; in `full_access`, per-tool approval is skipped while schema validation, resource extraction, target routing for eligible tools, output limits, product operation audit, and Floret activity projection still apply. Terminal process creation, input, output access, and termination additionally require effective write and execute permission, so an execute+read runtime policy cannot become a general-purpose shell through Flower. Command classification remains useful for audit and presentation, but it is not the source of permission decisions. Already-sent provider requests cannot be rewritten retroactively, but saved thread permission changes affect the next Floret provider request and the next local tool dispatch. `subagents` is a delegation-control tool visible in all three permission types, and its own `spawn`, `send_input`, `wait`, `list`, `inspect`, `close`, and `close_all` invocations do not require approval. That allow decision covers only the delegation control action itself; child runs receive a derived dynamic tool surface from the latest parent thread permission, and child tool invocations still flow through Floret permission/resource/approval lifecycle. A child approval-required action can delegate the ask back through the parent approver path; direct child user input remains disabled. Tool handlers execute already-approved Redeven domain actions, while Floret remains the single persistent source of truth for tool identity, arguments, lifecycle state, results, errors, completion output, and Activity projection. Redeven product audit records only user operations, policy, routing, and permission facts and never becomes a queryable tool-state copy or Flower display source.

Current ordinary tool approvals come from Floret's product-neutral pending approval read model. Redeven calls `ListPendingApprovals` on the active Floret host, maps each generic pending approval to a Flower `FlowerApprovalAction` using `ToolPresentationSpec`, generic effects/resources/labels/host context, permission policy, delegated approval metadata, and Floret's zero-based `batch_index` plus `batch_size`, and then publishes that product projection through Flower live state. A thread-owned queue combines main tools, delegated child tools, and control confirmations. New actions receive immutable monotonic queue order; new main actions observed in one pending snapshot are first ordered by run, step, batch index, and action id, while later asynchronous or delegated actions append in registration order. Only the queue head is `primary_action` and user-decidable; every sibling remains a read-only locator. Approving the head releases that one waiter immediately and promotes the next action while the approved handler may still be running. Rejecting or timing out one action resolves only that action. The decision timeout begins on promotion, not while queued. Stop or cancellation closes every unresolved waiter, records canceled state, and clears the queue. The thread summary status is an effective product projection: `waiting_user` has first priority, an unresolved approval queue projects `waiting_approval`, and only then do active run or persisted terminal states apply. Lightweight thread list summaries do not carry approval actions or queue details, and their omission is never an authoritative clear of selected-thread live approval state. Pending approval summaries derive their user-visible labels from the same Redeven tool presentation projection used for Floret invocation activity; Floret carries that sanitized activity presentation through approval lifecycle observations, so a requested approval is rendered as the same tool activity item with `requires_approval`, `approval_state=requested`, and `status=waiting`, not as a second approval row or a UI-layer pending approval backfill. `requires_approval` records that the invocation participated in the approval lifecycle; once Floret resolves the approval, approved pending/running/success tool rows keep that history without becoming a locked current approval row in Flower. Activity timeline rows remain audit and locator material and expose no competing Allow or Deny buttons. Submitting an ordinary approval validates queue generation/revision, head identity, expected event sequence, action revision/version, surface epoch, user/thread/run ownership, and the current Floret pending snapshot before calling Redeven's approver channel. Every successful ordinary, control-confirmation, or delegated decision returns the cursor of its persisted `approval.resolved` live event through the existing `current_cursor` response field. The shared Flower surface enters loading synchronously before transport, then uses that receipt cursor plus the current stream generation to wait for an authoritative promoted action, explicit empty queue, or terminal thread. A newer generation is treated as resync and judged from its snapshot rather than compared to the old cursor. If live projection does not reconcile within 1500 ms, the surface performs one canonical fallback reload; missing approval fields preserve the frozen card, while explicit empty approval actions, a null/zero queue, or terminal state settle it. Any stale, duplicate, already-resolved, timed-out, unavailable, or non-head decision is returned as HTTP 409 with the stable `AI_APPROVAL_CONFLICT` error code. The shared Flower surface reloads canonical state on that conflict and retries at most once only when the same action id is still current and actionable with refreshed CAS fields; if another action has been promoted or the old action has resolved, the stale decision is discarded without user-facing send-failure feedback. The task-complete confirmation surface is a Redeven `control_confirm` product confirmation for a Floret control signal, but it shares the same product queue arbitration.

Readonly-exclusive helpers are not general convenience aliases. `read_file`, `read_files`, `rgrep`, `find`, and `web_fetch` are visible only in `readonly`; `approval_required` and `full_access` use the standard host surface instead. `web_fetch` is a host-provided Redeven tool with SSRF protections, manual redirect validation, resolver and dial-time public IP checks, proxy suppression, body and MIME limits, and public-network-only semantics. DNS-derived VPN fake-IP answers in `198.18.0.0/15` are allowed for named hosts so common transparent-proxy/VPN setups can fetch public pages, but literal `198.18.0.0/15` URL hosts and other blocked local/private/reserved targets remain denied. It is not a Floret builtin and must not appear as a Floret core tool or a capability available to non-readonly permission types.

Terminal shell execution is a Redeven-owned process resource rather than a Floret-owned shell lifecycle. `terminal.exec` starts a PTY-backed process in the local AI runtime, immediately publishes a Floret `Invocation.UpdateActivity` presentation update with the public `process_id` read handle, waits only for the `yield_ms` initial wait, and returns either a completed result or a Floret pending tool result whose provider-visible handle is the same bare `process_id` used by `terminal.read`, `terminal.write`, and `terminal.terminate`. `yield_ms` is the only initial-wait argument and is not a hard timeout or automatic process termination request. Long-running commands continue outside the provider step. Redeven stores output as monotonic chunks of at most 4,000 bytes and retains at most 1,000,000 bytes by evicting complete oldest chunks. `Snapshot` returns the retained cumulative view for `terminal.exec`, final settlement, write/terminate responses, and cumulative UI replacement. `ReadAfter` returns only chunks after the caller's last consumed sequence; `terminal.read` uses a fixed five-second wait and 8,000-byte page, while the Flower process-read API uses a fixed one-second wait and 256,000-byte page. A valid Flower process read returns the current snapshot even when the process has already reached `success`, `error`, or `canceled`, so the terminal status can stop UI polling without becoming a canonical Activity lifecycle fact. A future cursor is invalid, an expired cursor returns the currently retained delta with `truncated=true`, and an empty delta preserves `last_seq=after_seq` with `first_seq=0`. Incremental results contain the single text field `output` plus `first_seq`, `last_seq`, `latest_seq`, `has_more`, process status, command, byte count, timing, and completion facts. They do not expose `stdout`, `stderr`, `latest_output`, caller-selected wait/size arguments, or field aliases.

Every `terminal.read` call requires exactly `process_id`, `description`, and `after_seq`. The first read uses zero; every later read copies the preceding result's `last_seq` unchanged, and `has_more=true` means another page is already available. The tool contract explicitly tells the model that `output` is new output since `after_seq`, never a replayed full snapshot, and that an empty value means no new output arrived during the wait. The bounded model-authored `description` uses the current user language, names the command or task being checked, and naturally distinguishes later checks. Redeven projects that description as the call activity label, emits no result label, and lets Floret preserve the call label while the result contributes the real command and that call's delta payload. The description is not duplicated into activity description or payload. `terminal.read` declares Floret's generic polling repeat policy, excludes only `description` from repeat identity through `AnnotationRepeatIdentityIgnoredArguments`, and returns a progress metadata token derived from `process_id`, `last_seq`, `status`, and `ended_at_ms`; changing presentation text cannot bypass no-progress detection. Redeven passes each incremental result once through `tools.Result.Text` and its independent display activity through `tools.Result.Activity`; it does not query Floret storage or persist a second mapped terminal state.

Every `terminal.terminate` call requires exactly `process_id` and a 1-120 rune `description` in the current user language that clearly names the command or task being stopped. Redeven uses `description` as the call activity label, keeps `process_id` in the compact call payload and terminal detail identity, and emits no result label, so Floret preserves the descriptive title instead of replacing it with the tool name. Termination has one ordered execution path: stop the Redeven PTY process, wait for process-tree reap and output drain, settle the original pending `terminal.exec` once through the process-bound Floret Host, apply the returned canonical Floret projection, and return so AppServer can write its product operation audit. Published Floret v0.9.0 requires the complete `PendingToolSettlementTarget` and returns the same target with the settlement result; the creating Host can append settlement through its already-owned active thread without reacquiring the turn lease or starting another provider request, while maintenance hosts still respect another Host's active lease. Redeven does not retry termination, issue `pkill` or `kill` compensation, swallow settlement errors, query Floret storage, or synthesize Flower state from audit rows.

Terminal snapshots and incremental reads carry `execution_location=local_runtime`, while the public Flower activity payload excludes host-local path/input fields such as `cwd`, `workdir`, or `stdin` as well as Floret pending lifecycle fields. Redeven caps active processes, retained output, stdin/input size, initial waits, and maximum runtime. Before a pending terminal process starts, Redeven requires the exact Floret `PendingToolSettlementTarget`, the creating top-level or subagent Host, and a one-shot finalizer. Missing ownership rejects process creation; Redeven never guesses the Host from a run registry and never opens a maintenance Host for terminal settlement. The finalizer becomes eligible only after the pending handle has been exposed to Floret and the PTY has completed reap, then invokes settlement once. `terminal.read` and `terminal.write` are pure PTY operations and cannot trigger, wait for, or retry settlement. `terminal.terminate` waits for reap, output drain, and that one settlement before returning. `Service.Close` terminates and finalizes all pending processes while their bound Hosts and the Floret store are still valid, then releases run and subagent resources. A ready settlement projection advances the shared durable `ThroughOrdinal` cursor and publishes canonical Flower state. An unavailable projection remains committed Floret state, emits diagnostic logging, and is not read back, retried, or locally synthesized. A settlement mutation failure is reported once by the finalizer and is not retried from read, write, cleanup, or later process access. Completion does not synthesize a duplicate provider-visible tool result, query Floret SQLite, persist a tool lifecycle copy, or automatically launch a follow-up model turn.

Provider adapters are model gateways, not Flower renderers. `floretProviderAdapter` maps provider stream bytes into Floret `ModelEvent` values. Assistant text, reasoning deltas, and model-generated tool-call stream observations are emitted as provider-neutral Floret model events before Redeven projects them into Flower live state. `ToolCallStream` identifies a tool call while the model is still generating it; final executable tool calls remain the separate `ToolCalls` batch. Redeven applies assistant text, reasoning deltas, tool-call stream observations, retry/done/abort markers, committed thread-entry observations, and source observations only from Floret runtime events in the event sink. Provider continuation persistence reads the complete opaque provider state from the Floret turn result, stores it as a state JSON envelope including `Attributes`, and only matches provider id, model, base URL, and state kind before passing it back to Floret.

Provider tool-call generation has a private `omit` or `enable` wire mode. `enable` serializes `parallel_tool_calls:true`; it allows a provider model to return more than one call but does not require the model to do so and is never read by the executor. Redeven enables it only for known HTTPS official OpenAI Responses/Chat, Qwen DashScope, OpenRouter, xAI, and Groq endpoints. Anthropic, DeepSeek, Moonshot, ChatGLM, Ollama, generic `openai_compatible`, Desktop opaque sources, proxies, and custom base URLs omit the field. No request path sends a false value. Hosted-turn diagnostics record the selected wire mode, and Floret lifecycle audit events carry batch index, batch size, event type, and observation time so operators can distinguish model serialization, provider single-call output, approval wait, dispatch, and completion without feeding diagnostics back into scheduling.

Thread model and reasoning defaults are Redeven product state. `ai_threads` persists `model_id` and `reasoning_selection_json`; there is no separate model-lock state. Existing thread model changes validate the requested model against the allowed runtime or Desktop model source list, require an idle mutable thread, normalize the thread reasoning selection for the new model, and clear provider continuation because the opaque continuation is keyed by provider/model/base URL/state kind. The configured `current_model_id` is only the persisted environment default for future threads; it must not be synced back into existing thread defaults. When a user changes the composer model for an existing thread, Redeven always updates that thread's `model_id`; only an environment-profile model is then persisted through `persistDefaultModel`, while a Desktop model remains thread-scoped. A pending new-chat Desktop selection is a mounted-surface draft rather than AI configuration. Turn launch resolves the effective model in this order: explicit request model, thread `model_id`, Desktop model-source override/default, then configured `current_model_id`. An explicit request model affects only that turn. User turns and waiting-input responses resolve and store their effective model and reasoning before they start or enter the queued follow-up lane, so queued work is not affected by later thread default changes. Desktop model-source descriptors carry one sanitized model capability used by model listing, thread reasoning defaults, run resolution, compaction, titles, subagents, and Floret provider controls; legacy v1 descriptors without that field are normalized from their existing context, output, modality, and reasoning fields. Floret receives the resolved model and reasoning as hosted runtime input and owns execution, provider loop, and opaque state transport only; it does not own thread preferences, provider profiles, UI policy, or product model switching.

Context pressure and compaction presentation come from structured Floret runtime observations. Redeven maps Floret context status into `context.usage.updated` and maps Floret compaction lifecycle events into `context.compaction.updated`; those product events are persisted as run events and projected into Flower live materialized state. Floret compaction debug observations are persisted separately as `floret.context.compact.debug` context run events with safe correlation, phase, token-pressure, validation, duration, provider-state-kind, error fields, and `next_action`. These debug rows are diagnostic records and never create timeline decorations or chat rows. Redeven owns only product control entrypoints, thread serialization, sanitized live projection, read-state activity, and opaque provider-state persistence for context lifecycle. Manual `/compact` requests enter Redeven only as Flower UI/RPC control signals and are forwarded into Floret `Host` compaction lifecycle APIs. Active manual compaction keeps the current Floret turn running, and `CompactionEvent(complete)` never implies a Redeven run or thread terminal state. Idle manual compaction is a Flower operation rather than an assistant run: the actor publishes the running divider and returns, a background worker calls Floret `Host.CompactThread`, and user turns submitted before the operation clears are held as queued follow-ups without transcript rows. Redeven persists only the opaque provider-state envelope returned by Floret after validating that the product thread boundary has not changed. Gate events for registered, cancel requested, commit attempted, commit rejected, and commit persisted phases make late commits, thread cancellation, and queued-turn drain diagnosable without treating diagnostics as Flower timeline rows.

Run token limits preserve distinct field semantics across the Redeven/Floret boundary. Redeven passes `RunOptions.MaxInputTokens` directly to Floret `TurnLimits.MaxInputTokens`, where it limits cumulative provider input usage across the entire hosted run. `MaxOutputTokens` remains a per-provider-request output ceiling used by Floret context policy and `ModelRequest.MaxOutputTokens`. Redeven does not add input and output limits into a derived `MaxTotalTokens`, and an unset input limit does not create an implicit cumulative token budget.

Redeven accepts a user turn only at one durable boundary. The service prepares the run object and user message payload before storage, then threadstore atomically writes the user transcript message, `ai_runs` row, `conversation_turns` anchor, thread run state, structured user inputs, secret answers, upload references, and optional source follow-up consumption. If storage fails, the reserved active run is released and no orphan transcript message is left behind. A failed terminal thread with no active run, no stop-finalizing guard, no waiting-user prompt, no idle compaction, and no queued lane is eligible for the same direct start path as an idle thread; only those active, waiting, finalizing, compacting, or queued conditions move the submitted user turn into the queue or conflict path. Conversation turns and structured user inputs remain product timeline and UI/audit records; they are not a Redeven-owned provider history source. Floret owns provider-visible history and context lifecycle through its thread store and opaque provider-state envelope. Thread fork coordination uses `ai_thread_fork_operations`: Redeven first records a `pending` operation and fixed snapshot schema v1 in the same transaction that reads the source thread, then calls Floret `ForkThread` with the durable `ForkOperationID`. The snapshot contains only the product records the fork is defined to copy: thread state, transcript messages, conversation turns, structured inputs, todos, memory, upload references, and Flower metadata; it excludes secrets, runs, run events, and every Floret-owned tool lifecycle or projection value. After Floret returns its fixed turn/run mappings, Redeven requires one exact mapping for every captured conversation turn and rewrites destination turn, run, thread, trace, and assistant-message references from that mapping before committing the destination and operation in one transaction. Missing, duplicate, or inconsistent mappings fail explicitly; Floret-only journal turns do not create Redeven rows, and no local fallback turn or run identity is generated. Startup and periodic maintenance replay `pending` operations with the same operation ID, source snapshot, and destination identity. Explicit operation, destination, missing-source, missing-target, or mapping conflicts become `failed`; transient errors remain retryable. Redeven never compensates by deleting the Floret destination and never treats destination existence as proof of success. Forked assistant/activity history stays readable through Floret `ReadTurnProjection`; Redeven does not clone Floret projection content into transcript rows.

Thread deletion is a data-lifecycle transaction, while forced cancellation remains an immediate execution-lifecycle response. Redeven can detach and cancel a stuck in-memory run without waiting for its goroutine to exit, then atomically delete the Redeven product thread rows while persisting the immutable cleanup operation that will remove files, the Floret thread tree, and Flower read state. Threadstore write paths that can insert late product state verify their parent thread inside the same transaction. If deletion commits first, later run, event, thread-state, continuation, todo, memory, permission, approval, upload-reference, and transcript writes fail instead of recreating rows. If a write commits first, the thread deletion transaction removes it with the rest of the product-owned thread state. Floret tool lifecycle state is deleted only through Floret's public thread deletion contract. `run.isDetached()` remains a presentation/admission signal and is not the durable deletion authority.

Flower assistant execution text and tool order come from Floret `ThreadTurnProjection`. During a live run, Floret streaming observations can update temporary markdown, while each committed thread entry can carry a validated `runtime.Event.Projection` whose status is `running` until a terminal turn marker arrives. Redeven applies that projection immediately and maps only assistant text to markdown, activity timelines to Flower activity blocks with Redeven file actions, and control signals to no independent body block. At turn completion, Redeven maps a validated ready `TurnResult.Projection`; after reload it uses the provider-free Floret `ThreadMaintenanceHost.ReadTurnProjection` API with explicit `ThreadID`, `TurnID`, and `RunID`. Floret owns projection validity through `ThreadTurnProjection.Validate()`, including identity completeness, positive `ThroughOrdinal`, finite status and segment kinds, activity timeline validity, and control-signal structure. Redeven adds only the product association check against the current run/thread/turn and returns an explicit error if a valid published segment cannot be mapped. It keeps the last applied `ThroughOrdinal` per full execution identity, ignores equal or older projections, and advances the cursor only after mapping and replacing the assistant blocks succeeds. Runtime events are accepted only after Floret `runtime.Event.Validate()` validates their nested stream, activity, and projection data; Redeven adds only projection association with the current Flower run. Public runtime events contain only `observation.EventType` values, while private harness lifecycle events remain on `HarnessSink` and never enter the Redeven event sink. Aggregate `runtime.Event.ActivityTimeline` remains lifecycle observation data and is not a main-thread block placement source. A completed, failed, or canceled hosted turn must not use `TurnResult.Output`, task-complete signal text, provider deltas, preview detail rows, aggregate activity timelines, product run errors, or audit rows as a fallback execution transcript; explicit history reconstruction represents a missing, rejected, non-terminal, or non-renderable `ReadTurnProjection` result as `turn_projection_unavailable` instead of synthesizing assistant content. Redeven does not store Floret-backed assistant projection JSON as `transcript_messages` shadow rows, query Floret SQLite, or rebuild display rows from audit data. Thread `last_message_preview` remains Redeven-owned summary state. Flower UI tool activity comes from Floret projection activity segments whose presentation was produced from Redeven `ToolPresentationSpec`; detail rendering consumes only public timeline payload and Redeven-owned file actions. A running terminal detail may use the public `process_id` to read live output, but that read path never creates activity rows, transcript messages, or lifecycle facts. Floret's local `tool_call` activity means queued work, while `tool_dispatch_started` marks running work after permission and approval gates pass.

Published Floret v0.8.0 separates engine execution from projection availability through `ProjectionAvailability`, `Projection`, and `ProjectionError`. Floret `TurnResult.ValidateProjection()` and `PendingToolSettlementResult.ValidateProjection()` own the `ready`/`unavailable` combinations, complete projection validation, terminal-state requirement, result/projection status agreement, and settlement identity agreement. Redeven consumes those results without redefining the contract. It does not immediately call `ReadTurnProjection` for an unavailable live run or settlement, and an engine failure remains primary. Explicit bootstrap and history reconstruction use `ReadTurnProjection` only for terminal projections and convert semantic not-found, invalid-contract, and non-renderable outcomes into the strict `turn_projection_unavailable` timeline decoration; storage and I/O errors fail the request. The decoration is anchored after the matching user message and never creates an empty assistant row or retry action. Invalid public events or projections record `floret.contract.rejected` and do not mutate normal Flower state. Provider finish, step end, run end, and context continuation use Floret's typed normalized finish, raw finish, inference, completion, and continuation fields; event metadata is not a compatibility source for those lifecycle reasons.

Flower subagents use the published Floret `Host` lifecycle rather than a Redeven-owned goroutine task manager. Main Flower runs, subagents, and compaction share the Floret-managed `stateDir/ai/floret_threads.sqlite` store, and Redeven accesses that store only through Floret public runtime APIs. Redeven caches one Floret subagent runtime per parent Flower thread so child controllers and the Floret host can outlive an individual parent turn, while restart recovery uses provider-free `ThreadMaintenanceHost` over the same published store contract without querying SQLite directly. Parent host initialization uses Floret `EnsureThread` instead of `ReadThread`. Parent Stop, CancelThread, and CancelRun are execution-lifecycle operations: after canceling the parent run they call Floret `CloseSubAgents` for the parent thread, which closes unfinished descendants, clears queued child input, and preserves completed history. A top-level parent hosted turn that exits failed, canceled, timed out, or with a Floret run error is also an execution-lifecycle terminal boundary: before Redeven performs product terminal cleanup, it classifies the parent reason (`parent_failed`, `parent_cancelled`, or `parent_timed_out`), calls Floret public `CloseSubAgents` through the active child host, parent host, or provider-free maintenance host, and republishes the resulting canonical child snapshots through `thread.subagents`. DeleteThread is a separate data-lifecycle operation documented in [Flower thread deletion coordination](flower-thread-deletion-coordination.md): Redeven atomically retires product data and persists an immutable cleanup snapshot first, then replays file cleanup, one Floret `ThreadMaintenanceHost.DeleteThread` tree deletion, and user read-state cleanup. Delete never calls `CloseSubAgents` as a precondition and never restores old rows as compensation. `Service.Close` finalizes pending terminal processes before releasing their top-level or child Hosts, then releases remaining local host resources. The `subagents` builtin validates `spawn`, `send_input`, `wait`, `list`, `inspect`, `close`, and `close_all`, then forwards lifecycle operations to Floret `SpawnSubAgent`, `SendSubAgentInput`, `WaitSubAgents`, `ListSubAgents`, `CloseSubAgent`, and `CloseSubAgents`; parent child-list refresh calls Floret `ListSubAgents` through the active host or `ThreadMaintenanceHost.ListSubAgents` and publishes `thread.subagents` patches instead of creating independent main-thread activity blocks. `wait` defaults to five minutes and caps requested waits at twenty minutes, while child runs pass a twenty-minute Floret subagent run timeout. The child `ThreadID` is the canonical subagent identity and is also returned as `subagent_id` for model-facing compatibility. Redeven stores a distinct product `child_run_id` for child permission snapshots and delegated approval refs; Floret execution `RunID` remains a per-engine-execution identity and is not interpreted as the product child run id. Delegated approval refs reject missing child run ids, child-thread aliases, and parent-run aliases. `spawn` persists a provisional child permission snapshot keyed by the real Floret `subagents` tool call id, calls `SpawnSubAgent`, and finalizes the snapshot only after host spawn succeeds; those stored snapshots remain product audit and permission records rather than child transcript ownership. Active child hosts install their own dynamic tool surface provider, derive the child surface from the latest parent thread permission on the next provider step and next tool dispatch, preserve child thread id and product `child_run_id` in host context, and emit `subagent.tool_surface.updated` when the child epoch changes. Task names and paths are labels, not lifecycle targets, and the model cannot supply a child `thread_id` or raw `fork_mode`. The parent model chooses `context_mode` on `spawn`: `mission_only` maps to `SubAgentForkNone` and `full_history` maps to `SubAgentForkFullPath`; after spawn, wait/list/detail payloads and parent activity presentation derive `context_mode` only from Floret `ForkMode` instead of a Redeven memory map or duplicate persistence. The thread-scoped child host registers the parent thread's delegated tool surface for the current permission snapshot, removes parent-only controls, and disables hidden Floret control surfaces. If the parent run's model/tool/session configuration changes, the runtime rebuilds the child host only when no child is active; active children continue on the existing host generation while still refreshing permission-derived tool surfaces. Child tools use the Floret invocation `ThreadID` and `TurnID` to verify execution scope, while Redeven product audit and delegated approval records use the explicit product `child_run_id` from host context rather than deriving it from Floret execution `RunID`. Child threads run with a Flower child prompt that forbids direct user input, states the visible tool list and context mode, and requires a complete but focused final handoff report.

Delegated approval is thread-owned product state, not subagent detail state. Redeven stores delegated approval requests with parent thread, parent user, parent run/turn, child thread/run/turn, child tool call, approval id, ref hash, request fingerprint, delivery state, version, and surface epoch. User decisions submit through the parent thread with a delegated ref; the store compares owner, ref hash, action id, version, surface epoch, and idempotency key before resolving. Idempotent replay is accepted only when the key matches the same action, same ref hash, and same decision. Resolved delivery uses compare-and-swap; if delivery acknowledgement cannot be confirmed, Redeven marks the record as acknowledgement-unknown instead of emitting a false delivered state. The parent Flower waiting projection is the only approval surface; subagent detail is a read-only mirror or locator.

Subagent data is intentionally split into model-facing summaries, parent timeline presentation, thread-level child state, and human detail. `subagents(action="spawn")` accepts `task_name`, the legacy `title` alias, a user-facing one-sentence `task_description`, an `agent_type`, and the full delegated `message`. Main-run prompt and tool descriptions ask the model for a 1-5 word English human-facing name such as `Safety Review`, `API Contract Review`, or `AI Research`. Before calling Floret, Redeven tokenizes snake case, kebab case, punctuation, and camel case, preserves common technical initialisms, applies Title Case, caps the result at five words and 48 ASCII characters, and falls back to a role-specific English task name when no usable token exists. The same canonical value is passed to Floret, persisted in lifecycle state, and returned through tool results and Flower summaries; historical stored names are not migrated. The task description remains the stable responsibility text shown in parent Flower UI, while live progress belongs in the child detail overlay. Model-facing `wait` results return bounded status `items` with child identity, task name, task description, agent type, status, timing, timeout facts, and `detail_ref`; they do not include child transcripts, tool calls, raw output, commands, stdout, stderr, `last_message`, `waiting_prompt`, or `result_digest`. Completed waits can include `final_handoff_report`, which carries each child’s final handoff text under a token budget without process detail. Timed-out waits can include `progress_summary`, which carries current status, waiting blockers when relevant, and suggested parent actions instead of execution logs. List and inspect remain bounded monitoring views with detail refs. Parent Flower timeline rows come from Floret `ThreadTurnProjection` activity segments and are projected to a public subagent payload before Flower renders them. That payload keeps only route/display facts such as `thread_id`, `subagent_id`, `parent_thread_id`, `action`, `status`, `task_name`, `title`, `task_description`, `agent_type`, canonical `items`, item status/timing, `timed_out`, `truncated`, counts, and public `error` or summary fields; it filters raw child content, private paths, detail refs, context/debug diagnostics, `last_message`, `waiting_prompt`, queued input counts, `can_*` control flags, and fields outside the current subagent payload allowlist. Flower activity rows render action-specific subagent titles, task/name previews, task descriptions, running elapsed time when available, exceptional status only when the child failed, canceled, timed out, or needs input, and an `Open messages` action derived only from the public `thread_id`/`subagent_id` payload fields.

The parent thread header Sub Agents dropdown is not a timeline reconstruction. Redeven lists child membership through Floret `ListSubAgents` on the active host, or through provider-free `ThreadMaintenanceHost.ListSubAgents` after runtime restart, maps that product-neutral state into `thread.subagents`, includes it in live bootstrap, and publishes the same list in `thread.patched` when subagent lifecycle events arrive. Flower builds the header badge and dropdown only from `thread.subagents`; it does not scan transcript blocks, `ActivityTimeline` rows, audit records, hidden sidecars, or thread list summaries. The shared UI projection derives a compatibility display name for historical snake/kebab/camel task names without mutating persisted state, then sorts children into stable active and ended priorities. The dropdown renders always-visible active and ended groups, exposes localized counts and statuses, focuses the first active row, supports arrow/Home/End navigation and Enter activation, and returns focus to the header trigger on Escape. If a thread list refresh summary omits `subagents`, that omission preserves selected-thread detail state; only a bootstrap or live patch that explicitly owns `subagents` can replace or clear the list. Redeven does not query Floret SQLite tables, write duplicate child membership into Redeven DB tables, or create child Flower metadata as a display source. The overlay reads `GET /api/ai/threads/{parent}/subagents/{child}/detail`, which validates the parent thread in Redeven, uses the active Floret host when present or a provider-free `ThreadMaintenanceHost` otherwise, calls Floret `ReadSubAgentDetail` through a parent-scoped public contract, maps Floret child not-found or ownership mismatch through `errors.Is` against Floret public runtime not-found sentinels to 404, and renders Floret's top-level canonical child activity timeline together with user/assistant messages, tool results, approvals, turn markers, compactions, errors, and custom rows from the child execution timeline. Preview-mode subagent detail is bounded inspection data and must not be used as the complete human transcript for child user or assistant messages; a complete human detail surface requires an explicit Floret display-detail contract while model-facing `wait`, `list`, and `inspect` results remain bounded. Event-row activity snapshots are not a display source, so later tool results or pending-tool settlements replace earlier running updates in the same canonical activity block. Child detail does not require a child row or child Flower metadata in Redeven threadstore, and subagents do not appear as child threads in the sidebar. Subagent model results use canonical `items` only; unsupported detail, transcript, raw-output, or non-allowlisted collection fields are filtered before they reach model-facing results or Flower activity payloads.

Control signal projection is separate from Redeven waiting UI persistence. Floret signal projection emits engine signal facts for `ask_user` and `task_complete`; Flower `RequestUserInputPrompt` objects are created only by Redeven's waiting projection and persistence layer.

`terminal.exec` is the local AI runtime shell. Its schema describes local PTY process execution, with `yield_ms` as the only initial-wait argument. Terminal results include `execution_location=local_runtime` plus the resolved local working directory and process identity. Thread target context, Welcome environment context, or a target-shaped id does not change where this builtin runs. Remote or target diagnostics must use Redeven product commands whose own result payload carries target provenance.

When a thread is configured for explicit target routing, the runtime forwards target-scoped builtin calls through `TargetToolExecutor`. The target executor receives a `TargetToolCall` containing `target_id`, `tool_name`, sanitized arguments, and required capabilities. The run layer returns a result payload that preserves or injects `target_id` and `execution_location`, so target-routed tool results cannot lose provenance before they reach the model or activity timeline.

Flower execution uses the published Floret runtime package through `runFloretHostedTurn` and `flruntime.Host`. Every production host explicitly sets `ThreadTitleModeHostOwned`; Redeven's asynchronous title coordinator is the only product title owner, and Floret does not issue an implicit provider title request. Redeven passes the current user input, current-turn supplemental context, initial tool registry, dynamic tool surface provider, control signals, provider adapter, opaque previous provider state, and loop limits to Floret; it does not pass a preassembled provider-visible history. Supplemental context is the only path for accepted Ask Flower linked context to become model-visible in the current turn: file items are metadata-only, attachments become `attachment_metadata`, monitoring items are process metadata, terminal items carry short selections or length-only truncation metadata, and Git/environment snapshots carry bounded generated summaries. Each accepted context-action item must map to exactly one supplemental item; formatter failure aborts before provider invocation. Floret `Metrics.Steps` is consumed as telemetry for result projection and activity accounting, not as a product configuration or request field. `ask_user` and `task_complete` use Floret core control definitions and provider-safe projection. Ordered assistant/tool display is derived from Floret `ThreadTurnProjection` segments, while the shape of Flower markdown blocks, activity timeline blocks, file actions, and tool presentation remains Redeven product projection.

# Boundaries

Tool names are not aliases for deleted knowledge-era tools. Current repository knowledge access uses `okf.index`, `okf.search`, and `okf.open`: index for broad directory discovery, search for short candidate lists, and open for detailed concept content. OKF is an embedded project corpus and does not access the internet; external, current, recent, news, third-party, market, pricing, and general web facts must use direct authoritative URLs or web search discovery instead.

Target provenance is part of the tool contract, not a UI hint. Flower must not infer remote execution from thread context alone; it can only claim remote or target execution when a tool result or Redeven product command returns explicit execution provenance.

Redeven must consume published Floret releases. Repository builds, tests, Desktop runs, and release validation must not depend on a sibling checkout, Go workspace, local replace directive, or package-manager local link. The runtime boundary depends on published `github.com/floegence/floret` v0.9.0, including explicit `PendingToolSettlementTarget` ownership and result identity validation; active-Host pending-tool settlement; validated polling identity exclusions for presentation-only arguments; host-owned thread titles; typed lifecycle fields; Floret's `Host` and maintenance facades; `RunTurnRequest.SupplementalContext` and `TurnSupplementalContextItem`; `ThreadTurnProjection.Validate()`; `TurnStatusRunning` plus `TurnStatus.Valid()` and `TurnStatus.IsTerminal()`; `runtime.Event.Validate()` with nested observation validation; public-event separation from private harness events; `ProjectionAvailability`; transcript-free `EnsureThread`; dynamic tool surfaces; explicit execution `RunID`; activity updates; structured context observations; opaque provider state; subagent lifecycle; replayable `ForkThread`; durable `ForkMode`; and cascading engine thread deletion. Redeven must not import Floret `internal/` packages or depend on provider-visible transcript, summary, range-selection, checkpoint payload, engine policy data, raw tool arguments, event metadata fallback, or a Redeven-owned tool lifecycle mirror.
