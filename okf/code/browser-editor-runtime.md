# Summary

Browser Editor support is owned by the Code App app server. Codespaces are opened through Local UI `/cs/<id>/` routes or through remote `cs-*` sandbox hosts, while Desktop and Env App can prepare or update managed Browser Editor runtime versions through code workspace engine APIs.

# Contract

## Mechanism

The Code App app server backend owns codespace lifecycle, port resolution, managed runtime status, setup operations, Desktop-transfer chunks, completion, version selection, version removal, and operation cancellation. Local UI routes `/cs/*` to the app server. A setup operation is identified by a caller-generated `operation_id`, records its explicit `desktop_transfer` or `remote_download` method, and is the only authority allowed to mutate the active Browser Editor setup status. Only one setup operation may run at a time, an accepted operation id cannot be reused during the RuntimeManager lifetime, and stale chunk, completion, cancellation, or progress updates cannot terminate or overwrite a different operation.

Setup creation has a method-discriminated response contract. `desktop_transfer` responses contain positive `chunk_size_bytes` and `expected_bytes` values plus the Runtime-confirmed received-byte and next-chunk cursors; Env App rejects a missing, negative, out-of-range, or otherwise invalid upload field instead of synthesizing a value. `remote_download` responses require only the common operation identity, method, and state and do not use the Desktop-only chunk size. A concurrent create returns HTTP 409 with stable AppServer error code `CODE_RUNTIME_OPERATION_CONFLICT`, produced from the typed Runtime conflict error rather than user-facing error text.

Env App treats Runtime status as authoritative as soon as setup creation is observed. A 409 triggers one status read: when that status identifies a Browser Editor setup operation, Env App adopts its actual operation id and install method and observes it to a terminal state; another active Runtime action remains blocked. When create response delivery or parsing is uncertain, Env App adopts an operation only when Runtime reports the exact requested id and method. A definitive non-conflict HTTP rejection fails directly. Env App never resends the create POST, creates a replacement id, switches methods, or infers an active operation from an error message. Settings observes an adopted operation without creating a second local failure, while Codespaces preserves its pending open or start intent and continues it only after the adopted setup succeeds.

`desktop_transfer` is the default when the trusted Desktop package bridge is available. Desktop reads the fixed Redeven Browser Editor catalog, streams the selected package into its verified cache, and exposes a cancellable package job that reads at most one 1 MiB chunk at a time. Catalog and package response headers have a 60-second limit, while package bodies fail only after 90 continuous seconds without bytes; an active download may therefore run for several minutes without a false total-duration timeout. Desktop package preparation requires the Env App `operation_id`, rejects a duplicate running id instead of replacing it, and reports stable catalog-versus-package failure codes. Env App creates the Code App setup operation and sends every chunk through the current AppServer session; Runtime-confirmed received bytes are the upload progress authority. Desktop no longer sends Browser Editor packages through Runtime Control, and Code App no longer exposes Runtime Control binary import routes, so local, Gateway, and SSH environments all use the same AppServer upload path.

`remote_download` makes the environment fetch the package itself. Runtime always reads `https://version.agent.redeven.com/v1/browser-editor/code-server/latest.json`; the client cannot provide a catalog or package URL. Runtime strictly validates the catalog schema, engine, mirror completeness, platform, compression, byte size, and SHA-256, and accepts package URLs only from HTTPS origin `https://agent.package.redeven.com` without cross-origin redirects. Requests use the environment proxy configuration, wait at most 60 seconds for response headers, and fail after 90 continuous seconds without package bytes. The download streams to an operation-scoped temporary file, removes that file on cancellation or failure, and retains only the latest verified cache entry for the platform. An invalid cache entry is deleted and treated as a cache miss; a checksum failure after a new download ends the operation without a second download.

Both methods enter the same Runtime-owned verification, extraction, executable probe, atomic promotion, inventory update, and managed-link repair pipeline. Managed packages validate checksums, compressed archive size, and extracted regular file totals with a 2 GiB safety cap. The code-server runner starts the editor bound to loopback with no editor auth, an absolute proxy base path under `/cs/<code_space_id>`, disabled telemetry/update checks, scoped data directories, and a short stable session socket path.

Remote Code App navigation keeps the Flowersec controller bridge bound to the isolated `app-*` origin. A compatible control plane can place its controller-issued bridge capability nonce and signed `proxy.runtime@1` WebSocket frame limit in that origin's session storage before replacing the bootstrap URL. The injected Redeven bridge restores those values after navigation or refresh and passes them to `registerProxyAppWindow`. Stored frame limits must be positive safe integers and cannot widen Redeven's existing 32 MiB product cap; malformed or oversized values use that cap. If both values are absent, the bridge retains the legacy no-nonce and 32 MiB registration used by older control-plane releases. This rolling fallback does not change target-origin routing, Flowersec wire framing, proxy protocol v1, or Redeven RPC payloads.

Env App chooses an explicit codespace open target before starting the space: `desktop_window` opens a dedicated Desktop codespace window through the shell bridge, while `system_browser` keeps using the existing external browser path. When the Desktop bridge exposes codespace-window support, a running codespace presents Desktop as the primary action with Browser in the adjacent menu; a stopped codespace keeps Start primary and offers both open targets from its menu. Start and Desktop-open actions enter an in-place busy state with the shared shimmer affordance so the card does not appear idle while runtime checks, start, ticket minting, or navigation are still pending.

Browser Editor preparation uses one shared semantic activity model across Codespaces and Settings. Codespaces presents the activity as a full-width, balanced operational band with status and actions beside either the four setup steps or a detected-versus-required platform diagnosis; Settings uses the compact layout of the same component. Before setup, an accessible two-option segmented control shows `Desktop transfer` and `Environment download` plus the selected data path. When the managed runtime is already ready, Settings keeps one update action and opens the same method selector in a confirmation dialog; when setup is missing or retryable, the compact activity owns the setup or retry action and the header does not duplicate it. Each new setup or retry recalculates the default from current Desktop bridge availability and does not persist the previous choice. The selector is read-only while an operation runs. Runtime platform results with `supported=false` and `unsupported_os`, `unsupported_arch`, or `unsupported_libc` are terminal, non-retryable setup results, while method-specific catalog, download, transfer, verification, and installation failures remain retryable. Failure details retain the method actually used, but retry returns to the newly calculated default. Technical paths, error codes, and logs stay collapsed under technical details by default. The Codespaces empty or populated content remains a separate page region below the preparation band.

Setup progress separates stage completion from measurable transfer work. Desktop-local progress exists only while Desktop is looking up, downloading, and validating a package before Runtime operation creation. After Runtime accepts or Env App adopts an operation, every visible stage and byte count comes from Runtime status. Env App therefore does not pre-create a `lookup` step for environment download while its POST is pending; `resolving_catalog` appears only after Runtime reports it. Runtime status exposes environment-confirmed received and expected bytes for Desktop upload or environment download, with a cache marker when the verified environment package is reused. Env App uses exact bytes only for Desktop download, environment-confirmed upload, and environment download; it derives a smoothed current-stage rate and remaining-time estimate only after enough real samples and falls back to stage plus elapsed time when work has no trustworthy total. Cached packages skip the download estimate, and 15 seconds without new confirmed bytes hides ETA while the operation remains running. Desktop and Runtime failures retain stable stage codes through the activity model; Env App does not infer catalog, cache, transfer, verification, or installation ownership from localized error text. Once an operation is observed, local progress and local failure are cleared, every Runtime poll is published to the page resource, and cancellation uses the observed Runtime id and method. Repeating cancellation for the same already-cancelled setup id is idempotent, while stale ids remain rejected. A failure never changes methods, automatically retries a chunk or setup, or silently starts another data path.

Desktop exposes a typed `redeven-desktop:shell-open-codespace-window` IPC request with `loading` and `navigate` modes. The trusted Env App renderer calls `loading` immediately for Desktop-open flows so Desktop can create or focus a native codespace window with a local, scriptless, CSP-bound loading document before the codespace URL exists. After start, local URL construction, or remote entry-ticket minting succeeds, Env App calls `navigate` with `{ url, code_space_id }`; legacy callers that omit `mode` but include `url` are normalized to `navigate`.

The main process validates the sender session for both modes. `loading` only creates or reuses the per-session, per-codespace child window and can later show an error state in that same window. `navigate` continues to check that the URL is both allowed for the session and specifically targets the requested codespace before loading it. Accepted codespace windows use the parent session partition for remote-session continuity, reuse one window per session and codespace id, use native OS chrome with no Desktop preload, and keep `sandbox: true`, `contextIsolation: true`, and `nodeIntegration: false`. Navigation inside that window is limited to the matching local `/cs/<id>/` path or a remote hostname whose first label is `cs-<id>`; other navigations and opened links are sent to the system browser.

# Boundaries

Opening or managing Browser Editor is not just a static file route. Version setup and removal require full read/write/execute runtime permissions, while status reads require read permission. Setup create, chunk, completion, and cancellation requests are audited with their operation id; create also records the chosen method. The remote method's fixed catalog and package-origin validation prevent the Env App request from becoming a general network fetch primitive. The session-scoped Desktop bridge values are transport bootstrap inputs for the current isolated app origin; they do not grant broader runtime permissions or replace controller source, origin, and capability validation.
