Redeven Desktop treats the endpoint runtime as a managed service when it launches `redeven run --mode desktop --desktop-managed`. The handoff is machine-readable: Desktop receives Local UI URLs, runtime-control endpoint data, Runtime Service readiness, and startup failures without parsing human terminal output.

# Mechanism

The CLI requires Desktop-managed Local UI runs to use machine presentation, writes startup reports, records Desktop owner metadata in the runtime lock, and exposes runtime-control only to loopback callers with the Desktop owner header and bearer token. Desktop passes optional Local UI password and bootstrap ticket values through one versioned `--startup-secrets-stdin` JSON envelope, writes it exactly once, and closes stdin. The envelope is capped at 64 KiB, while fixed and legacy secret environment variables are removed from runtime and status-process environments.

Desktop persists network exposure acknowledgement as `{version: 1, bind: canonicalBind}` beside the Local Environment access settings. A saved network bind without an exact acknowledgement remains visible but cannot start; Welcome presents `Review network exposure` and the settings window keeps the current draft. Network selection shows an inline warning before save. Review occurs as a second step in the same settings window, displays the bind, reachable interface scope, plaintext transport, and password state, and requires one explicit checkbox before enablement. Changing host or port invalidates acknowledgement, changing only the password does not, and returning to loopback clears it. Runtime argv receives `--acknowledge-plaintext-network-exposure` only when the canonical bind matches and the private password is available. Runtime catalog writeback preserves that exact startup acknowledgement, so a successful Desktop-managed restart cannot erase the review required by the next start.

Local UI access settings are startup configuration. When the draft changes bind, password state, or acknowledgement and the selected Local Environment has an available Desktop restart operation, the settings footer adds `Save and restart` while retaining a save-only action. Plaintext network review keeps both choices disabled until the explicit checkbox and effective password are present. The restart path saves and closes settings first, then invokes the existing `restart_environment_runtime` lifecycle action; it does not create a settings-specific process launcher, silently restart an offline Runtime, or bypass lifecycle progress and failure diagnostics. The settings Runtime status uses verified managed Runtime presence even when no Env App window is open.

Desktop separates Runtime lifecycle readiness from Env App transport readiness. Health-only probes call `/api/local/runtime/health`, return a typed success or one of `timeout`, `network_error`, and `invalid_response`, and accept an independent request budget and cancellation signal. Runtime health and startup reports carry the canonical `LocalUIExposure` object. Caller cancellation propagates as `AbortError`; it is not converted into a network failure. After an SSH or container placement bridge is established, Desktop performs exactly one startup probe through the bridge loopback URL: Runtime health, Env App HTML, unique entry assets, and Runtime Service open readiness must all validate before the window opens. Failure ends that open attempt immediately; Desktop does not start another readiness period or choose another transport.

SSH Runtime installation, process identity, ownership, Start, Stop, Restart, and Update remain a lifecycle layer represented by `sshRuntimeReadyByKey`; that record does not own Env App transport. Open executes the remote `redeven desktop-bridge` command through Runtime Placement Bridge, creates one local loopback proxy, and stores the active connection only in `runtimePlacementBridgeByTargetID`. Containers use the same placement bridge session and record. The bridge command reads only the required `local_ui_bridge_url` from Runtime attach status. Missing or unreachable trusted bridge state fails immediately. There is no SSH `-L` Local UI path, public Local UI fallback, Host rewriting, cached-failure retry, compatibility field mapping, or alternate connection record.

SSH Runtime bootstrap owns one explicit ControlMaster session context for the master process, control socket, authentication, cancellation, command logs, and multiplexed remote commands. Control readiness checks run with isolated polling logs, so expected pre-readiness `ssh -O check` socket errors are not promoted into user diagnostics; a terminal readiness failure records only the final failed liveness check. After readiness, every multiplexed bootstrap command that exits nonzero is followed by a deterministic liveness classification using both the owned master process state and an isolated `ssh -O check`. If both remain healthy, the current phase reports the real remote command failure; a private upload-directory failure is `ssh_upload_directory_unavailable` and retains the remote `$TMPDIR` or `/tmp` error. If either liveness signal is unhealthy, the transaction terminates as `ssh_connection_interrupted` regardless of the phase that observed it. Desktop does not parse host-specific stderr, reconnect automatically, replay the failed command, or select an alternate installer. Recovery is an explicit user retry after checking network, VPN, and SSH service state. Cleanup remains best effort and cannot replace the first failure. Interruption diagnostics retain the failed command output and one final control-check result without including successful readiness-poll noise.

Desktop Welcome reads concrete SSH aliases from the user's SSH configuration through the main-process loader. The loader follows bounded `Include` directives, excludes wildcard and negated Host patterns from selectable targets, and returns the complete alias set rather than a presentation-sized subset. Entering any SSH-backed Environment or Gateway create/edit flow starts a fresh read; the picker retains the last successful set while refreshing, ranks exact and prefix alias matches before metadata matches, and keeps every result accessible inside a constrained scrolling list. Loading, read failure, empty configuration, and no-match states remain visible without blocking a manually entered `user@host` destination.

Environment windows remain hidden until two readiness gates settle: Env App reports either an interactive access gate or a connected runtime protocol after a painted shell frame, and the Desktop model source has settled. For SSH and container opens, model-source startup begins in parallel with hidden session creation and document loading. Model-source failure settles only its existing gate and does not create or select replacement transport. The renderer reports relative bootstrap, access, protocol, and shell-paint timings through the internal app-ready IPC. Launcher operations retain phase start and duration data, and successful opens emit a desensitized `environment_open_timing` lifecycle event with bridge, model-source, window, document, gate, and renderer timing data.

Desktop treats Runtime Stop, Restart, and Update as terminal boundaries for every Env App session attached to the affected Environment. Once the lifecycle coordinator accepts an SSH or container destructive action, Desktop closes matching windows and the placement bridge before process discovery, stop, package activation, or restart work begins. Provider-link, ordinary window close, bridge-process exit, application quit, and maintenance all resolve the same bridge record; disconnect closes the remote bridge command, loopback proxy, and active sockets. The lifecycle boundary advances an Environment generation and cancels matching Opens so a late response cannot recreate the Env App. When maintenance starts inside Env App, Desktop opens or focuses Welcome before closing the session. The closed session is never rebound or automatically reopened when the Runtime becomes ready again.

Desktop-managed process authority comes from the schema 2 Runtime Process Inventory, not the startup report PID, current state-root lock, in-memory ready record, or a status command exit code. Local, SSH, and container lifecycle paths share the same orthogonal identity, owner, current-layout, stop-authority, digest, executable-inode, user, and namespace rules documented in the Desktop runtime process lifecycle concept. Start and Open are observational. Stop, Restart, and Update proceed automatically only for current-owner processes; a fully verified process with missing or foreign owner evidence requires a digest-bound user confirmation, while any blocked identity prevents every signal.

Desktop model-source RPC v1 publishes opaque `desktop:model_<hash>` identifiers plus an optional sanitized capability descriptor. Desktop resolves the real provider profile locally, then replaces provider and model identity in the descriptor with the opaque Desktop identity while retaining operational limits, modality support, tool-schema behavior, and reasoning controls. The runtime uses that descriptor consistently for UI model listing and execution. A runtime receiving an older v1 descriptor derives a compatibility capability from the existing model fields rather than guessing from the opaque hash. Provider ids, base URLs, API keys, and secret locations remain local to Desktop.

When Desktop Welcome opens Flower and the Local Environment runtime is cold, the Flower attach path uses the same coordinated local lifecycle. It joins an active Start, Restart, or Update, marks that launcher attempt with the explicit `flower_warmup` presentation context, waits for settlement, and attaches again. An active Stop makes Flower explicitly unavailable and never triggers a replacement Start. The Flower surface renders coordinated Start, Restart, and Update progress as warmup instead of treating the surface as stalled.

Desktop Welcome uses the same Flower adapter contract as Env App for thread operations and split settings writes. Sending a turn posts through the runtime Flower IPC proxy, stop posts to the thread cancel route, and `/compact` posts to `/_redeven_proxy/api/ai/threads/{thread}/context/compact` before reloading the canonical live bootstrap. Default permission writes use the exact `PUT /_redeven_proxy/api/ai/default_permission` route, provider profile writes use `PUT /_redeven_proxy/api/ai/provider_bundle`, and composer model selection writes the future new-thread default through `PUT /_redeven_proxy/api/ai/current_model`. The Desktop bridge also exposes the Flower working-directory picker through exact read-only runtime FS paths: `GET /_redeven_proxy/api/fs/path_context` and `POST /_redeven_proxy/api/fs/list`. The bridge allowlist is a single route table that admits only declared paths, methods, and query shapes, so the permission route is PUT-only and path selection data travels in the POST body instead of opening arbitrary query-path proxying. The Desktop bridge treats compaction as a thread action, not as a transcript message or a local UI-only marker, so Desktop and Env App receive the same live timeline decorations and read-state patches from the runtime.

# Boundaries

Runtime-control is not a general network API. It is scoped to the local Desktop/runtime bridge and protected by loopback, Desktop owner id, and bearer token checks even when Local UI is network-exposed. The startup secrets envelope is private process stdin and must not be copied into argv, environment variables, launch reports, logs, or Desktop diagnostics. The acknowledgement contains only the canonical bind and is not a secret. Desktop must not preconnect SSH, keep a hidden bridge solely to improve a future click, forward the public Local UI listener, retry a failed bridge through another URL, or automatically restart a Runtime after bridge readiness failure. Window readiness is not satisfied by `did-finish-load` alone; presentation remains gated by the renderer's painted interactive state and model-source settlement. Flower warmup is a Desktop presentation context over the existing runtime lifecycle operation, not a separate readiness source or a parser for runtime terminal output. Desktop must not implement alternate Flower thread semantics locally; thread stop, send, compact, working-directory path context/list reads, and live reload flow through the runtime proxy contract.

Start is not a destructive lifecycle request and must not close an otherwise valid Env App session. Env App Shell Restart and SSH Update first acquire the same physical lifecycle ownership used by Welcome; session handoff occurs only inside the coordinated replacement callback. A conflicting Stop or other lifecycle transaction therefore leaves the current session untouched and returns structured busy state. Unsupported, unauthorized, or locally invalid Stop, Restart, and Update requests also leave the window open because no lifecycle handoff was accepted. A Desktop release-page handoff is guidance rather than a Runtime update and does not close the session. Once destructive maintenance has been accepted, a later failure or cancellation does not restore the old window; Welcome remains the authoritative progress and recovery surface, and the user explicitly opens a fresh Env App session after Runtime readiness returns.

Runtime process inventory is an internal Desktop machine contract rather than a Runtime Service, OpenAPI, database, or plugin protocol. SSH and container inventory and stop commands always use a temporary helper extracted from the verified current Desktop runtime asset; installed runtimes are not process-contract implementations. The helper does not change the version-stable Restart package selection, does not activate Update staging, and is removed after governance completes.

Runtime-control is also not a plugin grant, plugin management, or plugin capability plane. Its token and routes are reserved for Desktop-managed runtime coordination such as provider-link, code-workspace-engine import, and Desktop model source binding. Plugin workers and sandbox surfaces must not receive runtime-control endpoint data, use runtime-control bearer tokens, or treat runtime-control routes as plugin capabilities; plugin access to Redeven resources must go through released ReDevPlugin brokers and Redeven-registered adapters.
