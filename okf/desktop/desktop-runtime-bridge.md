Redeven Desktop treats the endpoint runtime as a managed service when it launches `redeven run --mode desktop --desktop-managed`. The handoff is machine-readable: Desktop receives Local UI URLs, runtime-control endpoint data, Runtime Service readiness, and startup failures without parsing human terminal output.

# Mechanism

The CLI requires Desktop-managed Local UI runs to use machine presentation, writes startup reports, records Desktop owner metadata in the runtime lock, and exposes runtime-control only to loopback callers with the Desktop owner header and bearer token. Desktop passes optional Local UI password and bootstrap ticket values through one versioned `--startup-secrets-stdin` JSON envelope, writes it exactly once, and closes stdin. The envelope is capped at 64 KiB, while fixed and legacy secret environment variables are removed from runtime and status-process environments. Desktop separates runtime health from Env App shell integrity. Health-only probes call `/api/local/runtime/health`, return a typed success or one of `timeout`, `network_error`, and `invalid_response`, and accept an independent request budget and cancellation signal. Caller cancellation propagates as `AbortError`; it is not converted into a network failure. These probes are used for recurring liveness refreshes and forwarded-runtime polling. Once Runtime Service reports an openable state, Desktop performs one shell validation: it loads the Env App HTML, extracts unique entry assets, and validates those assets with concurrent `HEAD` requests under the same remaining deadline and cancellation signal. Successful shell validation is cached by normalized Local UI URL, runtime start time, and runtime build identity; failed or changed runtime identities are validated again. Desktop then builds the Env App entry URL from the normalized Local UI base URL.

SSH open preserves on-demand connection ownership. A click may reuse a matching ready record only when the associated health snapshot is still fresh within the 30-second TTL. Stale or mismatched state triggers one explicit refresh. Desktop owns one handle for each Local UI and runtime-control SSH forward; either forward failing to spawn or exiting before readiness completes immediately terminates verification. Forward readiness keeps four timing concepts separate: the 200 ms polling cadence applies only after an unsuccessful readiness attempt, each health request receives at most the Local UI default 1500 ms budget, that budget is capped by the remaining overall readiness deadline, and caller cancellation remains independent from both timeout levels. Shell HTML and entry assets are checked once, only after health reports an openable Runtime Service, under the remaining overall deadline. Deadline exhaustion is classified from the latest health failure as `ssh_forward_verification_timed_out`, `ssh_forward_network_failed`, or `ssh_forward_invalid_response`; process failure is `ssh_forward_unavailable`. A reused fresh record may refresh and retry once only for unavailable, network, and invalid-response failures. Verification timeout never starts a second full readiness period. Diagnostics record only the failure type, error code or HTTP status, attempts, latest request budget, and elapsed time; response bodies, tokens, and other secrets are excluded.

Environment windows remain hidden until two readiness gates settle: Env App reports either an interactive access gate or a connected runtime protocol after a painted shell frame, and the Desktop model source has settled. For SSH opens, model-source startup begins in parallel with hidden session creation and document loading. Model-source failure retains the existing unavailable fallback but still settles its gate, so it does not block Env App. The renderer reports relative bootstrap, access, protocol, and shell-paint timings through the internal app-ready IPC. Launcher operations retain phase start and duration data, and successful opens emit a desensitized `environment_open_timing` lifecycle event with cache, probe, SSH, model-source, window, document, gate, and renderer timing data.

Desktop treats Runtime Stop, Restart, and Update as terminal boundaries for every Env App session attached to the affected Environment. Package preparation and process discovery may run while the existing session remains usable. After identity checks accept the destructive transition and immediately before the first signal, Desktop closes matching root windows, child windows, and Codespace windows, then marks the operation non-cancelable. Local, SSH, container, Provider, and Gateway identities resolve through one session-closing path, including every open session for the same Gateway Environment or provider identity. The same boundary advances an Environment lifecycle generation, cancels every matching launcher Open, and performs an idempotent second close pass. Every Local, Provider, SSH, Gateway, and container window creation verifies the generation captured when its Open began, so a late remote response cannot recreate an Env App after maintenance was accepted. When maintenance starts inside Env App, Desktop opens or focuses Welcome before closing the session; Runtime-owned secure Restart and Upgrade RPCs notify Desktop only after the request is accepted or the connection loss proves that maintenance started. The closed session is never rebound or automatically reopened when the Runtime becomes ready again.

Desktop-managed process authority comes from the schema v1 Runtime Process Inventory, not the startup report PID, current state-root lock, in-memory ready record, or a status command exit code. Local, SSH, and container lifecycle paths share the same classification, digest-protected stop, legacy layout, owner, user, executable inode, and namespace rules documented in the Desktop runtime process lifecycle concept. Start and Open are observational; explicit Stop, Restart, and Update may reconcile current and historical processes only after every matching instance is safely classified.

Desktop model-source RPC v1 publishes opaque `desktop:model_<hash>` identifiers plus an optional sanitized capability descriptor. Desktop resolves the real provider profile locally, then replaces provider and model identity in the descriptor with the opaque Desktop identity while retaining operational limits, modality support, tool-schema behavior, and reasoning controls. The runtime uses that descriptor consistently for UI model listing and execution. A runtime receiving an older v1 descriptor derives a compatibility capability from the existing model fields rather than guessing from the opaque hash. Provider ids, base URLs, API keys, and secret locations remain local to Desktop.

When Desktop Welcome opens Flower and the Local Environment runtime is cold, the Flower attach path uses the same coordinated local lifecycle. It joins an active Start, Restart, or Update, marks that launcher attempt with the explicit `flower_warmup` presentation context, waits for settlement, and attaches again. An active Stop makes Flower explicitly unavailable and never triggers a replacement Start. The Flower surface renders coordinated Start, Restart, and Update progress as warmup instead of treating the surface as stalled.

Desktop Welcome uses the same Flower adapter contract as Env App for thread operations. Sending a turn posts through the runtime Flower IPC proxy, stop posts to the thread cancel route, and `/compact` posts to `/_redeven_proxy/api/ai/threads/{thread}/context/compact` before reloading the canonical live bootstrap. Composer model selection writes the future new-thread default through the same proxy using the exact `PUT /_redeven_proxy/api/ai/current_model` route. The Desktop bridge also exposes the Flower working-directory picker through exact read-only runtime FS paths: `GET /_redeven_proxy/api/fs/path_context` and `POST /_redeven_proxy/api/fs/list`. The bridge allowlist is a single route table that admits only declared paths, methods, and query shapes, so path selection data travels in the POST body instead of opening arbitrary query-path proxying. The Desktop bridge treats compaction as a thread action, not as a transcript message or a local UI-only marker, so Desktop and Env App receive the same live timeline decorations and read-state patches from the runtime.

# Boundaries

Runtime-control is not a general network API. It is scoped to the local Desktop/runtime bridge and protected by loopback, Desktop owner id, and bearer token checks. The startup secrets envelope is private process stdin and must not be copied into environment variables, launch reports, logs, or Desktop diagnostics. Health freshness and shell validation caches are reuse decisions, not background connection authority: Desktop must not preconnect SSH, warm tunnels at startup, automatically restart a Runtime after forward verification failure, or keep a hidden SSH session solely to improve a future click. Window readiness is also not satisfied by `did-finish-load` alone; presentation remains gated by the renderer's painted interactive state and model-source settlement. Flower warmup is a Desktop presentation context over the existing runtime lifecycle operation, not a separate readiness source or a parser for runtime terminal output. Desktop must not implement alternate Flower thread semantics locally; thread stop, send, compact, working-directory path context/list reads, and live reload flow through the runtime proxy contract.

Start is not a destructive lifecycle request and must not close an otherwise valid Env App session. Env App Shell Restart and SSH Update first acquire the same physical lifecycle ownership used by Welcome; session handoff occurs only inside the coordinated replacement callback. A conflicting Stop or other lifecycle transaction therefore leaves the current session untouched and returns structured busy state. Unsupported, unauthorized, or locally invalid Stop, Restart, and Update requests also leave the window open because no lifecycle handoff was accepted. A Desktop release-page handoff is guidance rather than a Runtime update and does not close the session. Once destructive maintenance has been accepted, a later failure or cancellation does not restore the old window; Welcome remains the authoritative progress and recovery surface, and the user explicitly opens a fresh Env App session after Runtime readiness returns.

Runtime process inventory is an internal Desktop machine contract rather than a Runtime Service, OpenAPI, database, or plugin protocol. Older installed SSH and container runtimes may be inspected and stopped by a temporary helper extracted from a verified Desktop runtime asset. The helper does not change the version-stable Restart package selection, does not activate Update staging, and is removed after governance completes.

Runtime-control is also not a plugin grant, plugin management, or plugin capability plane. Its token and routes are reserved for Desktop-managed runtime coordination such as provider-link, code-workspace-engine import, and Desktop model source binding. Plugin workers and sandbox surfaces must not receive runtime-control endpoint data, use runtime-control bearer tokens, or treat runtime-control routes as plugin capabilities; plugin access to Redeven resources must go through released ReDevPlugin brokers and Redeven-registered adapters.
