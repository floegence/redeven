# Summary

Desktop-managed Runtime startup uses machine-readable reports, owner metadata, and separate lifecycle and transport readiness. Access settings validate password and plaintext exposure acknowledgement before save, while health probes remain observation rather than lifecycle commands. Environment windows open only after both Env App interactivity and Desktop shell readiness settle, preserving deterministic ownership of startup and failure presentation.

# Contract

## Mechanism

The CLI requires Desktop-managed Local UI runs to use machine presentation, writes startup reports, records Desktop owner metadata in the runtime lock, and exposes runtime-control only to loopback callers with the Desktop owner header and bearer token. Desktop passes optional Local UI password and bootstrap ticket values through one versioned `--startup-secrets-stdin` JSON envelope, writes it exactly once, and closes stdin. The envelope is capped at 64 KiB, while fixed and legacy secret environment variables are removed from runtime and status-process environments.

Desktop persists network exposure acknowledgement as `{version: 1, bind: canonicalBind}` beside the Local Environment access settings. A saved network bind without an exact acknowledgement remains visible but cannot start; Welcome presents `Review network exposure` and the settings window keeps the current draft. Network selection shows an inline warning before save. Review occurs as a second step in the same settings window, displays the bind, reachable interface scope, plaintext transport, and password state, and requires one explicit checkbox before enablement. Changing host or port invalidates acknowledgement, changing only the password does not, and returning to loopback clears it. Runtime argv receives `--acknowledge-plaintext-network-exposure` only when the canonical bind matches and the private password is available. Runtime catalog writeback preserves that exact startup acknowledgement, so a successful Desktop-managed restart cannot erase the review required by the next start.

Local UI access settings are startup configuration. Password validity and bind-scoped plaintext acknowledgement are prerequisites for the single final save action rather than peer footer actions. A new or stale network exposure uses a two-step settings flow: the edit step cannot advance without an effective password, and the review step cannot save without the explicit checkbox. When the draft changes bind, password state, or acknowledgement and the selected Local Environment has an available Desktop restart operation, the final step presents a non-disruptive `Next start` versus `Restart now` timing choice and keeps one primary button whose label reflects that choice. The restart path saves and closes settings first, then invokes the existing `restart_environment_runtime` lifecycle action; it does not create a settings-specific process launcher, silently restart an offline Runtime, or bypass lifecycle progress and failure diagnostics. The settings Runtime status uses verified managed Runtime presence even when no Env App window is open.

Desktop separates Runtime lifecycle readiness from Env App transport readiness. Health-only probes call `/api/local/runtime/health`, return a typed success or one of `timeout`, `network_error`, and `invalid_response`, and accept an independent request budget and cancellation signal. Runtime health and startup reports carry the canonical `LocalUIExposure` object. Caller cancellation propagates as `AbortError`; it is not converted into a network failure. After an SSH or container placement bridge is established, Desktop performs exactly one startup probe through the bridge loopback URL: Runtime health, Env App HTML, unique entry assets, and Runtime Service open readiness must all validate before the window opens. Failure ends that open attempt immediately; Desktop does not start another readiness period or choose another transport.

SSH Runtime installation, process identity, ownership, Start, Stop, Restart, and Update remain a lifecycle layer represented by `sshRuntimeReadyByKey`; that record does not own Env App transport. Open executes the remote `redeven desktop-bridge` command through Runtime Placement Bridge, creates one stable local loopback proxy, and registers the active connection with the internal `RuntimePlacementBridgeRegistry`. Containers use the same placement bridge session and record. The bridge command reads only the required `local_ui_bridge_url` from Runtime attach status. Missing or unreachable trusted bridge state fails the initial Open immediately. After an established SSH bridge loses its remote transport, the loopback address remains stable, active HTTP or WebSocket streams end, and new requests receive the existing `502 Bad Gateway` response until a replacement transport completes its hello and identity checks. Existing streams and request bytes are never replayed or migrated. There is no SSH `-L` Local UI path, public Local UI fallback, Host rewriting, compatibility field mapping, alternate URL, or alternate connection record.

Desktop Welcome treats the main-process launcher operation snapshot as the sole persistent owner of Environment action progress and failure. A failed Open or Runtime lifecycle action occupies the card's primary action position and opens the structured operation detail with recovery, diagnostics, and one dismiss action. Status and lifecycle steps remain first; terminal failure suppresses the generic operation detail above those steps, then presents the localized failure summary and recovery guidance after the timeline and progress meter. Raw error text and structured diagnostics remain inside one default-closed technical disclosure whose viewport is height-constrained and internally scrollable, while recovery and copy/dismiss actions remain outside that local scroll region. The operation detail stays anchored above its card action, is constrained to the available viewport height, and receives initial focus at its status header instead of scrolling directly to a later action. Dismissal removes that launcher operation; a newer retry supersedes the failed presentation, and successful completion releases the primary action without renderer repair state. Disclosure state is transient presentation only. The renderer may still show an auto-dismissing notification or health-derived guidance, but it does not retain a parallel per-Environment failure cache, badge, card border, or acknowledgement path.

Environment windows remain hidden until two readiness gates settle: Env App reports either an interactive access gate or a connected runtime protocol after a painted shell frame, and the Desktop model source has settled. For SSH and container opens, model-source startup begins in parallel with hidden session creation and document loading. Model-source failure settles only its existing gate and does not create or select replacement transport. The renderer reports relative bootstrap, access, protocol, and shell-paint timings through the internal app-ready IPC. Launcher operations retain phase start and duration data, and successful opens emit a desensitized `environment_open_timing` lifecycle event with bridge, model-source, window, document, gate, and renderer timing data.

# Boundaries

Runtime-control is not a general network API. It is scoped to the local Desktop/runtime bridge and protected by loopback, Desktop owner id, and bearer token checks even when Local UI is network-exposed. The startup secrets envelope is private process stdin and must not be copied into argv, environment variables, launch reports, logs, or Desktop diagnostics. The acknowledgement contains only the canonical bind and is not a secret. Desktop must not preconnect SSH, keep a hidden bridge solely to improve a future click, forward the public Local UI listener, retry through another URL or public listener, or automatically start or restart a Runtime after bridge readiness or recovery failure. Stable bridge recovery replaces only the private stdio transport for the verified existing process generation. Window readiness is not satisfied by `did-finish-load` alone; presentation remains gated by the renderer's painted interactive state and model-source settlement. Flower warmup is a Desktop presentation context over the existing runtime lifecycle operation, not a separate readiness source or a parser for runtime terminal output. Desktop must not implement alternate Flower thread semantics locally; thread stop, send, compact, working-directory path context/list reads, and live reload flow through the runtime proxy contract.

Health observation is not a transport lifecycle command. A transient Local UI probe failure, including a failure observed while the bridge snapshot still says `ready`, must not disconnect the bridge, close its stable loopback proxy, release Provider occupancy, or publish a replacement record. Recovery phase transitions and terminal closure come only from the bridge session; explicit lifecycle owners use registry retirement and await the one session settlement.

Start is not a destructive lifecycle request and must not close an otherwise valid Env App session. Env App Shell Restart and SSH Update first acquire the same physical lifecycle ownership used by Welcome; session handoff occurs only inside the coordinated replacement callback. A conflicting Stop or other lifecycle transaction therefore leaves the current session untouched and returns structured busy state. Unsupported, unauthorized, or locally invalid Stop, Restart, and Update requests also leave the window open because no lifecycle handoff was accepted. A Desktop release-page handoff is guidance rather than a Runtime update and does not close the session. Once destructive maintenance has been accepted, a later failure or cancellation does not restore the old window; Welcome remains the authoritative progress and recovery surface, and the user explicitly opens a fresh Env App session after Runtime readiness returns.
