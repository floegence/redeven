Env App is built as a Redeven-specific shell on top of published floe-webapp UI/runtime primitives, floe-webapp protocol connectivity, Floeterm terminal-web components, and Flowersec controlplane artifact helpers.

# Mechanism

The released browser dependency set includes Floeterm terminal-web v0.5.19 in addition to Floe Webapp v0.37.1 and Flowersec Core v0.20.2.

The UI package pins Floe Webapp v0.37.1 and Flowersec Core v0.20.2 from the public registry. `EnvAppShell` imports floe-webapp providers, layout primitives, icons, and protocol hooks; terminal surfaces use Floeterm `TerminalCore` and `PagedTerminalOutputCoordinator`; controlplane services call Flowersec controlplane helpers to exchange entry tickets for connect artifacts; and local package-lock files are the audited dependency source for the embedded UI build. Activity and Workbench terminals map Redeven attach/history RPCs and shell-activity projection into the same upstream output coordinator. `TerminalSessionRuntime` calls `beginAttach` before subscribing to live output, then completes that generation with the Runtime's atomic history boundary; output produced during the attach round trip remains retained, while a stale acknowledgement cannot complete a newer generation. The separate Runtime attach generation is owned by the stable protocol transport proxy, so Activity, Workbench, component remounts, RPC facade recreation, and reconnect share a monotonic sequence for one provider lifetime. Initial replay blocks terminal input only until explicit coverage establishes a baseline and `TerminalCore.writeHistory` reports parser completion; background gap recovery and automatic retry after that fence remain non-blocking. The coordinator owns sparse sequence coverage, retained-live bounds, retry state, snapshot-generation validation, truncation rebases, cancellation, writer completion, and coverage commit ordering. `pause()` returns only after the current writer and its coverage commit are quiescent, and the cancelled recovery cannot start another transform/write batch afterward. Redeven owns only RPC conversion, source tagging, ordered parser transformation, terminal visibility, stable localized status presentation, diagnostics entry, and product activity callbacks. Shell, OSC, working-directory, and visible-output side effects are emitted only from the coordinator's accepted ordered transform path; the host does not keep a parallel payload projection cache. Normal catch-up stays silent, while delayed retry and degraded states use a reserved footer: 28 px on desktop and 44 px while the Floe mobile keyboard is visible. Initialization autofocus requires focus ownership to remain empty or inside the same current-generation terminal. Session-selection focus uses a separate generation-scoped intent captured at request time; it retries after the selected Runtime becomes interactive only while the committed target, view ownership, keyboard policy, and original sidebar/body/origin-or-target-terminal focus ownership remain valid. `TerminalSessionRuntime` owns the Core, coordinator, snapshot, and per-session focus lifecycle, while `TerminalSessionNavigator` renders the desktop sidebar and mobile drawer from lightweight session state without instantiating a Core. `TerminalPanel` remains the product orchestration and composition owner instead of introducing a second terminal state model. Redeven's adaptive working-set manager does not impose a Core count: it keeps all opened cores warm below a device-derived byte budget, protects the current and recently switched sessions, and asks the released Floeterm core to capture or restore memory-only snapshots only from idle work. A hibernated core releases renderer resources while its PTY, output coordinator, unread state, and session-list row continue independently.

Runtime reconnect configuration uses Flowersec's discriminated `ArtifactSource` contract directly. Both Local UI and remote controlplane paths use `refreshable` sources because each reconnect must mint a fresh credential; no fixed artifact is reused or given an auto-reconnect escape path. Entry-ticket exchange uses `createControlplaneArtifactSource` and `acquire`, forwarding abort and trace context. Env App constructs one stable reconnect config for each transport mode so hard reconnects preserve Flowersec trace and source-consumption state. Local Direct explicitly allows plaintext only for canonical loopback authorities and enables 15-second Yamux liveness with a 10-second timeout. Remote Tunnel explicitly requires TLS and retains Flowersec's tunnel liveness default. Both paths select 64 KiB outbound records and explicit WebSocket and Yamux queue, frame, stream, and receive-memory limits.

Terminal diagnostics are renderer-local and content-free. Protocol capture projects `terminal.sessionAttach` to dimensions, acknowledgement, history boundary, and a numeric RPC failure code, `terminal.input` to byte count, and `terminal.history` to page, chunk, byte, coverage, generation, reset, and truncation metadata; Runtime `attach_generation`, command, output, session ID, connection ID, path, raw failure message, and raw failure cause are not retained. Raw fetch/RPC capture runs only while Debug Console is open. Sanitized recovery lifecycle events always enter a bounded 160-event ring so a later diagnostics action can inspect the failure; export includes the bounded client events and dropped count. Recovery traces keep surface generation, Floeterm coordinator attach generation, and Runtime history generation as separate fields and use renderer-lifetime pseudonymous session references. The terminal diagnostics action opens the trace view using only the pseudonym, surface generation, and stable error code. Renderer `performance.mark` entries for attach start/ack, baseline queued/parser committed, interactive, and first live output share the renderer monotonic clock.

Env App keeps only bootstrap, access, protocol, and shell chrome in the initial module. Activity surfaces, Workbench, Terminal, Monitor, Files, Codespaces, Ports, Flower, Codex, Settings, Plugins, Debug Console, audit, file browser, and file preview hosts are loaded through feature boundaries when activated. Flower and Codex providers are scoped to the surfaces that need them, so a connected shell does not initialize AI settings, model catalogs, or thread lists by default. Feature CSS follows the same boundary: Iosevka loads with Terminal, shared Flower and chat styles load with Flower entry points, and Codex styles load with Codex entry points.

Activity navigation does not replace the Activity Shell or change its outer provider structure. `ActivityAppsMain` is the lifetime owner for Activity surfaces: a surface mounts on first activation and remains mounted but hidden while another Activity surface is active. This preserves file browser path, expanded directories, filtering, view mode, selection, scroll position, and equivalent local state in Terminal, Monitor, Flower, Codex, and other registered Activity surfaces. Activity and Workbench are themselves two lazily visited views in one `KeepAliveStack`; after first activation, switching display mode changes visibility and activation after paint without destroying either view. Activity navigation, display-mode navigation, and Workbench widget activation therefore share the same lifetime rule: lazy before first visit, stable DOM/provider/data-flow identity after first visit.

The released floe-webapp selection controller separates immediate visual selection from committed content. Env App opts the Shell into UI-first Activity selection, uses after-paint activation for `ActivityAppsMain`, and uses another UI-first controller plus after-paint `KeepAliveStack` activation for Activity/Workbench mode changes. User input uses the delayed transaction path; initialization, restore, permission fallback, and other canonical corrections may commit synchronously. Persistence runs from the committed state rather than from a visual preview. Navigation does not suspend hidden surfaces, stop subscriptions, hibernate terminal cores, refetch warm content, or recreate providers; only visibility-dependent focus, resize, scroll restoration, viewport movement, and measurement may run after the target content has painted.

Codex keeps its heavy provider and feature CSS behind the lazy `CodexActivitySurface` boundary. The surface creates one shared Codex provider for the page and sidebar. The page stays inside the kept-alive Activity view, while the sidebar is projected through a Solid Portal into a lightweight Shell-owned host that exists only while Codex is active and permitted. The Portal mount remains stable when that host is removed or recreated, so returning to Codex moves the existing sidebar tree instead of recreating the provider, page, thread state, drafts, or sidebar-local state. If the Codex chunk resolves after navigation has already left Codex, it may finish mounting inside the hidden kept-alive surface without replacing the Shell or another Activity page.

The local bootstrap reuses the access status returned by `getLocalRuntime()` instead of issuing a second access-status request. Production builds enforce compressed initial-resource budgets of 600 KiB JavaScript, 120 KiB critical CSS, and 720 KiB total. The budget check also rejects initial HTML references to Markdown, KaTeX, Mermaid, Monaco, Excel, PDF, Shiki, Flower, or Codex feature assets.

# Boundaries

This concept only holds while Env App continues to consume published upstream packages instead of local sibling checkouts or ad-hoc replacements for protocol, terminal, reconnect, and controlplane artifact handling. Redeven may select generic resource values and map product credential acquisition into `ArtifactSource`, but must not recreate Flowersec reconnect state, WebSocket backpressure, Yamux liveness, E2EE framing, or RPC scheduling. Redeven must not recreate Floeterm output recovery, snapshot formats, resource estimation, or shell lifecycle generation in product code. The packaged Env App and Runtime must satisfy compatibility epoch 6 and the v0.8.1 minimum pair so the coordinator never treats missing coverage metadata or an atomic attach boundary as a successful baseline; the published v0.8.0 epoch 5 window is not compatible with this contract. Lazy boundaries change loading time, not product contracts: restored surface state, permission gates, Workbench activation, error recovery, and provider ownership must remain equivalent after a feature chunk resolves. Activity-specific feature providers must live inside their kept-alive surface boundary; an active-surface condition must not wrap or replace the Activity Shell. Activity/Workbench mode changes must not be treated as an unload boundary after either view has mounted. Sidebar portals may change their visible host, but the provider and kept-alive surface remain the state owners. The initial budget is a required build gate and must not be bypassed by renaming or preloading feature assets through a different entry.
