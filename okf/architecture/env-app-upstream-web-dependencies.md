# Summary

Env App is built as a Redeven-specific shell on top of published floe-webapp UI/runtime primitives, floe-webapp protocol connectivity, Floeterm terminal-web components, and Flowersec controlplane artifact helpers.

Env App composes published floe-webapp, Floeterm, and Flowersec web packages.

# Contract

## Mechanism

The browser dependency set includes the public Floeterm terminal-web v0.7.0 release and its exact `@floegence/beamterm-renderer` v1.0.1 dependency in addition to Floe Webapp v0.39.0 and Flowersec Core v0.27.0. Every published UI dependency resolves from the public npm registry. Local paths, links, workspaces, portals, aliases, runtime patches, private registries, and renderer fallbacks remain forbidden.

The UI package pins Floe Webapp v0.39.0 and Flowersec Core v0.27.0 from the public registry. Floe v0.39.0 supplies the built-in shell preset catalog, per-mode preset selection, generated shell token CSS, and Monaco theme definitions. Env App configures those public contracts and projects the Desktop-owned source and versioned light/dark preset selection through its Floe storage adapter; it does not fork the catalog, synthesize shell token maps, or maintain a second Monaco palette. `EnvAppShell` imports floe-webapp providers, layout primitives, icons, and protocol hooks; terminal surfaces use Floeterm `TerminalCore` and `PagedTerminalOutputCoordinator`; controlplane services call Flowersec controlplane helpers to exchange entry tickets for connect artifacts; and local package-lock files are the audited dependency source for the embedded UI build. Activity and Workbench terminals map Redeven attach/history RPCs and shell-activity projection into the same upstream output coordinator. `TerminalSessionRuntime` calls `beginAttach` before subscribing to live output, then completes that generation with the Runtime's atomic history boundary; output produced during the attach round trip remains retained, while a stale acknowledgement cannot complete a newer generation. The separate Runtime attach generation is owned by the stable protocol transport proxy, so Activity, Workbench, component remounts, RPC facade recreation, and reconnect share a monotonic sequence for one provider lifetime. A superseded current interactive surface retries with a newer generation under view, active-session, and focus ownership fences; a background surface waits until it regains those owners. Session disappearance, connection teardown, and transport disconnect end the obsolete surface without presenting a false terminal-recovery error. Initial replay blocks terminal input only until explicit coverage establishes a baseline and `TerminalCore.writeHistory` reports parser completion; background gap recovery and automatic retry after that fence remain non-blocking. The coordinator owns sparse sequence coverage, retained-live bounds, retry state, snapshot-generation validation, truncation rebases, cancellation, writer completion, and coverage commit ordering. `pause()` returns only after the current writer and its coverage commit are quiescent, and the cancelled recovery cannot start another transform/write batch afterward. Redeven owns only RPC conversion, source tagging, ordered parser transformation, terminal visibility, stable localized status presentation, diagnostics entry, and product activity callbacks. Shell, OSC, working-directory, and visible-output side effects are emitted only from the coordinator's accepted ordered transform path; the host does not keep a parallel payload projection cache. Normal catch-up stays silent, while delayed retry and degraded states use a reserved footer: 28 px on desktop and 44 px while the Floe mobile keyboard is visible. Initialization autofocus requires focus ownership to remain empty or inside the same current-generation terminal. Session-selection focus uses a separate generation-scoped intent captured at request time; it retries after the selected Runtime becomes interactive only while the committed target, view ownership, keyboard policy, and original sidebar/body/origin-or-target-terminal focus ownership remain valid. `TerminalSessionRuntime` owns the Core, coordinator, snapshot, and per-session focus lifecycle, while `TerminalSessionNavigator` renders the desktop sidebar and mobile drawer from lightweight session state without instantiating a Core. `TerminalPanel` remains the product orchestration and composition owner instead of introducing a second terminal state model. Redeven's adaptive working-set manager does not impose a Core count: it keeps all opened cores warm below a device-derived byte budget, protects the current and recently switched sessions, and asks the released Floeterm core to capture or restore memory-only snapshots only from idle work. A hibernated core releases renderer resources while its PTY, output coordinator, unread state, and session-list row continue independently. Flowersec reconnects keep one active promise per transport configuration, proxy redirects are returned to the browser, and Service Worker chunk credits prevent a fast runtime from outrunning a slow consumer while legacy workers continue without the optional capability. Local Direct reconnects select `AllowPlaintextForLoopback` for loopback locations or a `createNetworkPlaintextPolicy` bound to the exact browser IP after the Runtime has activated explicit network exposure; construction failure blocks the connection and never retries with an unrestricted policy.

Floeterm terminal-web v0.7.0 publishes the bounded shell-integration parser, foreground-command sanitizer, command-aware and output-activity session types, revision-fenced coordinator, and strict Agent CLI basename classifier. The lightweight sessions entry exports the sanitizer and classifier without loading terminal renderers. Redeven delegates standard A/B/C/D, working-directory, safe-program parsing, and Agent CLI identity classification to Floeterm. The product UI maps only confirmed running Agent CLI identities to audited local brand assets, while rendering process lifetime, PTY output activity, and unread attention as independent signals. Output `settled` is presented as recent quiet output and never as task completion or process exit.

Runtime reconnect configuration uses Flowersec's discriminated `ArtifactSource` contract directly. Both Local UI and remote controlplane paths use `refreshable` sources because each reconnect must mint a fresh credential; no fixed artifact is reused or given an auto-reconnect escape path. Entry-ticket exchange uses `createControlplaneArtifactSource` and `acquire`, forwarding abort and trace context. Artifact requests retain the upstream HTTPS default and redirect rejection; Env App enables loopback HTTP only when the current page is itself an HTTP canonical loopback location, while network HTTP remains denied. Env App constructs one stable reconnect config for each transport mode so hard reconnects preserve Flowersec trace and source-consumption state. Local Direct explicitly allows plaintext only for canonical loopback authorities and enables 15-second Yamux liveness with a 10-second timeout. Remote Tunnel explicitly requires TLS and retains Flowersec's tunnel liveness default. Both paths select 64 KiB outbound records and explicit WebSocket and Yamux queue, frame, stream, and receive-memory limits.

After an already opened Env App loses connectivity, Redeven projects published Flowersec `onDiagnosticEvent` reconnect events into one immutable product recovery snapshot. The `reconnect` stage, event code, result, and monotonic `attempt_seq` are the source for fast-reconnect attempts; Redeven does not recreate Flowersec's retry loop. Runtime availability probes return structured availability, access state, failure code, and HTTP status, and error classification uses only those fields and error types rather than message fragments. A Desktop-hosted session also subscribes to its sender-scoped transport-recovery snapshot. While that bridge is `waiting`, `connecting`, or terminally `failed`, Env App pauses its own Runtime probe loop; a recovered bridge advances directly to protocol reconnect. Generation and revision checks discard stale Desktop updates, and no layer replays the request that observed the disconnect.

Terminal diagnostics are renderer-local and content-free. RPC capture projects retained-history page, chunk, byte, coverage, generation, reset, and truncation metadata; it no longer recognizes attach or input operations because realtime traffic is not RPC. Floeterm live server failures expose structured numeric error codes to product classification, and session closure is a distinct protocol state. Debug Console does not retain terminal input, output, session id, connection id, path, raw server message, or raw failure cause. Sanitized recovery lifecycle events remain in the bounded renderer ring, use renderer-lifetime pseudonymous session references, and keep surface, history, coordinator, geometry, and output sequence generations distinct. Renderer performance marks for attach start/ack, baseline queued/parser committed, interactive, and first live output share the renderer monotonic clock.

The catalog adapter is a separate static module from the Floeterm live transport. Initial catalog hydration and bounded history warmup therefore cannot pull the live client, codec, renderer, or Terminal feature graph into the Env bootstrap. The production initial-build policy verifies this module graph and fails rather than accepting renamed or preloaded live assets.

Each `TerminalSessionRuntimeStatus` carries the diagnostics query derived from that runtime's own trace. Activity and Workbench surfaces for the same terminal session therefore keep distinct surface generations even when their failures interleave; the footer never resolves diagnostics through a session-global latest trace. The terminal runtime root reports `aria-busy=true` only until the initial baseline parser commit makes the surface interactive. Background retry or degraded recovery remains usable and does not set the surface busy again. The tab activity tracker promotes already-observed background provisional unread to confirmed unread before attach-generation changes, history rebases, or ordinary reattach; only an explicit user clear-history reset may discard that provisional state.

Env App keeps only bootstrap, access, protocol, and shell chrome in the initial module. Activity surfaces, Workbench, Terminal, Monitor, Files, Codespaces, Ports, Flower, Codex, Settings, Plugins, Debug Console, audit, file browser, and file preview hosts are loaded through feature boundaries when activated. Flower and Codex providers are scoped to the surfaces that need them, so a connected shell does not initialize AI settings, model catalogs, or thread lists by default. Feature CSS follows the same boundary: Iosevka loads with Terminal, shared Flower and chat styles load with Flower entry points, and Codex styles load with Codex entry points.

Activity navigation does not replace the Activity Shell or change its outer provider structure. `ActivityAppsMain` is the lifetime owner for Activity surfaces: a surface mounts on first activation and remains mounted but hidden while another Activity surface is active. This preserves file browser path, expanded directories, filtering, view mode, selection, scroll position, and equivalent local state in Terminal, Monitor, Flower, Codex, and other registered Activity surfaces. Activity and Workbench are themselves two lazily visited views in one `KeepAliveStack`; after first activation, switching display mode changes visibility and activation after paint without destroying either view. Activity navigation, display-mode navigation, and Workbench widget activation therefore share the same lifetime rule: lazy before first visit, stable DOM/provider/data-flow identity after first visit.

Connection recovery preserves those lifetime owners. Activity and Workbench each mount the same `ConnectionRecoveryView` once at their content root while their existing page DOM remains mounted. The underlying content container becomes `inert`, pointer-disabled, and `aria-hidden`; Terminal, Files, Monitor, Workbench widgets, and other surfaces do not own separate connection curtains or continue accepting stale input. The recovery presentation derives a fixed applicable-step timeline, exact completed-step percentage, real attempt counts, and only known retry timestamps from the immutable snapshot. Successful recovery remains visible for 1.5 seconds before the snapshot returns to idle and the original surface becomes interactive again. Terminal failure retains the window, focuses an alert heading, keeps technical details collapsed by default, and exposes only actions present in the Desktop snapshot. Raw HTTP status text is diagnostic detail rather than primary recovery copy.

The released floe-webapp selection controller separates immediate visual selection from committed content. Env App opts the Shell into UI-first Activity selection, uses after-paint activation for `ActivityAppsMain`, and uses another UI-first controller plus after-paint `KeepAliveStack` activation for Activity/Workbench mode changes. User input uses the delayed transaction path; initialization, restore, permission fallback, and other canonical corrections may commit synchronously. Persistence runs from the committed state rather than from a visual preview. Navigation does not suspend hidden surfaces, stop subscriptions, hibernate terminal cores, refetch warm content, or recreate providers; only visibility-dependent focus, resize, scroll restoration, viewport movement, and measurement may run after the target content has painted.

Codex keeps its heavy provider and feature CSS behind the lazy `CodexActivitySurface` boundary. The surface creates one shared Codex provider for the page and sidebar. The page stays inside the kept-alive Activity view, while the sidebar is projected through a Solid Portal into a lightweight Shell-owned host that exists only while Codex is active and permitted. The Portal mount remains stable when that host is removed or recreated, so returning to Codex moves the existing sidebar tree instead of recreating the provider, page, thread state, drafts, or sidebar-local state. If the Codex chunk resolves after navigation has already left Codex, it may finish mounting inside the hidden kept-alive surface without replacing the Shell or another Activity page.

The local bootstrap reuses the access status returned by `getLocalRuntime()` instead of issuing a second access-status request. Production builds enforce compressed initial-resource budgets of 600 KiB JavaScript, 120 KiB critical CSS, and 720 KiB total. The budget check also rejects initial HTML references to Markdown, KaTeX, Mermaid, Monaco, Excel, PDF, Shiki, Flower, or Codex feature assets.

Terminal stream lifetime and native autofocus are separate contracts. The Floe mobile keyboard may keep the current Activity or Workbench terminal stream attached while deliberately leaving the system textarea unfocused. Every reload entry and asynchronous attach boundary checks the surface disposal fence, so an unmounted panel cannot reopen a stream after teardown. Concurrent surfaces use independent Floeterm connection identities and do not compete for focus-scoped attachment ownership.

Upstream TerminalCore and coordinator loggers are strict no-ops inside the product. Redeven emits only stable event and error codes with numeric or boolean metadata, never raw `Error` objects, upstream messages, terminal content, or encoded content. Console privacy tests reject both a unique raw secret and its base64 form independently.

Renderer validation keeps distinct evidence boundaries. The built-dist shell smoke serves only the production `dist/env` files and proves a nonblank root, expected title, absence of framework overlays or browser errors, content-hashed JavaScript/CSS, and a compilable content-hashed WASM renderer. The terminal recovery process carrier separately starts a temporary production Redeven Runtime and drives the real Local UI, Direct WebSocket/RPC, terminal-go, PTY, TerminalPanel, coordinator, parser, and canvas path in headless Chromium. It exercises Workbench and Activity panel surfaces across reused and fresh browser contexts, verifies parser-committed recovery for 64 KiB, 448 KiB, and the exact 8 MiB retained-byte boundary, compares the recovered canvas against seeded history evidence, and sends a filesystem-backed input probe through the terminal. Neither test starts Redeven Desktop, and neither is described as a packaged Electron test. Reports contain only fixture classes, requested, retained, and fetched byte counts, pseudonymous references, generation and coverage metadata, timings, stable failure stage codes, runner fingerprint data, and boolean foreground-presentation lifecycle evidence. They never record the observed command or directory title. Standard GitHub-hosted jobs use the timings as raw evidence, not as percentile claims.

# Boundaries

This concept only holds while Env App continues to consume published upstream packages instead of local sibling checkouts or ad-hoc replacements for protocol, terminal, reconnect, controlplane artifact handling, shell presets, or Monaco themes. Redeven may select a published shell preset and map generic resource values and product credential acquisition into upstream contracts, but must not recreate Floe shell token generation, Monaco theme definitions, Flowersec reconnect state, WebSocket backpressure, Yamux liveness, E2EE framing, RPC scheduling, or host-scoped plaintext policy semantics. Redeven must not recreate Floeterm output recovery, snapshot formats, resource estimation, or shell lifecycle generation in product code. The packaged Env App and Runtime must satisfy compatibility epoch 8 and the v0.10.0 minimum pair so the coordinator never treats missing coverage metadata or an atomic attach boundary as a successful baseline and the Env App never omits or misinterprets Local UI exposure state. Lazy boundaries change loading time, not product contracts: restored surface state, permission gates, Workbench activation, error recovery, and provider ownership must remain equivalent after a feature chunk resolves. Activity-specific feature providers must live inside their kept-alive surface boundary; an active-surface condition must not wrap or replace the Activity Shell. Activity/Workbench mode changes must not be treated as an unload boundary after either view has mounted. Sidebar portals may change their visible host, but the provider and kept-alive surface remain the state owners. The initial budget is a required build gate and must not be bypassed by renaming or preloading feature assets through a different entry.

The Env App recovery snapshot is a product presentation projection, not another protocol or Desktop transport mechanism. It must not infer retry timing, parse failure prose, invent attempts, add a backup URL, retry through public Local UI, migrate streams, rebind a stale session, or replay user requests. Explicit Start, Restart, Stop, and Update continue to hand off to Desktop Welcome and remain outside this post-open recovery flow.
